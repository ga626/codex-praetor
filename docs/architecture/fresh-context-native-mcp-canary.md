# Fresh-Context Native MCP Canary

Date: 2026-07-10

This is the final native MCP acceptance gate for Codex Praetor before a public alpha release. It proves that the running Codex Desktop host resolved the staged runtime generation, rather than only proving the Node protocol smoke path or a same-name tool set.

Do not run this as a substitute for local validation. Run it after the public-release doctor, minimal validation, MCP protocol smoke, final URL replacement, and release package build have passed.

## Preconditions

- Final GitHub owner/repo URL is confirmed and applied to plugin metadata.
- `scripts/verify/doctor-codex-praetor.ps1 -RequireHead -PublicRelease` passes with the public template.
- `scripts/verify/test-codex-praetor.ps1` passes.
- MCP source tests and packaged protocol smoke pass.
- Local plugin package has been republished or the release package has been installed into a fresh Codex plugin context.
- Duplicate same-name `codex-praetor` MCP registrations have been removed.
- The current stale development thread is not used as the final proof point.

## Fresh Context

Use one of these fresh contexts:

- A newly opened Codex Desktop thread after the running Desktop host has refreshed its plugin discovery state.
- A fresh Codex CLI or ephemeral execution context that exposes native MCP tools.

Opening a new thread alone is not a host refresh. If `codex_praetor_runtime_info` still reports an old runtime contract or cache root, stop: do not create a proof, do not activate, and do not treat another new thread as a repair.

The proof must use native Codex Praetor MCP tool calls from the Desktop context that will use the plugin. A PowerShell script, SDK client, or separately started `codex app-server` is useful support evidence, but it is not the final native MCP canary because it does not inspect the already-running Desktop host.

## Required Calls

The fresh context must show the native Codex Praetor MCP tools and successfully call at least:

- `codex_praetor_runtime_info` first. Record its full structured response, including `runtime_identity.runtime_contract_sha256`, `contract_path`, `project_root`, `process_id`, and `process_started_at`.

- `codex_praetor_route_intent`

- `codex_praetor_dispatch_dry_run`

- `codex_praetor_list_lanes` or `codex_praetor_detect_conflicts`

Use a harmless readonly prompt such as:

```text
Route a readonly release canary for the current repository. Do not dispatch a real worker. Return the selected provider tier and the command that would be used.
```

The dry-run call must not create a real worker job, mutate repository files, or require provider credentials.

## Pass Criteria

- Tools are visible by their native MCP names in the fresh context.
- `runtime_info` version, task contract schema, and runtime-contract SHA256 equal the staged release generation. The runtime path and process identity are present.
- Route-intent returns a structured route decision without `Transport closed`, cancellation, or schema errors.
- Dispatch dry-run returns a dry-run plan and does not create a worker job.
- Lane or conflict read calls return structured state instead of transport errors.
- The main repository remains clean except for intentional release-documentation edits and ignored runtime/output directories.

## Failure Classification

- Tools absent: plugin discovery, packaging, installation, or host-refresh failure.
- `runtime_info` reports an old version/hash/root: the Desktop host still resolves a stale plugin generation. A new thread cannot repair this; refresh or relaunch the Desktop host through a supported user action, then repeat the canary once.
- `Transport closed`: stale context, duplicate registration, packaged runtime failure, or host reload/cache issue. Re-run protocol smoke, remove duplicate registrations, republish, refresh the Desktop host, then create one new context for the canary.
- Tool returns schema errors: MCP tool contract or argument regression.
- Dry-run creates a job or mutates files: release blocker in the dispatch wrapper or MCP adapter.

## Evidence To Record

Record the exact fresh-context used, the tool names seen, the complete `runtime_info` observation, the required call outcomes, and the final git status in the private release evidence before public publication. The proof file is generated by `scripts/verify/new-codex-praetor-fresh-context-proof.ps1` and must contain the native `runtime_info` response, not only a copied tool-name list. Do not paste provider tokens, account pages, local auth databases, or private path-heavy logs into public docs.
