# Codex 执政官下一大版本统一治理与交付规划

日期：2026-07-18
性质：只读考古、外部调研、架构冻结与实施规划
结论：下一次实现应是一个统一治理大 PR；不单独发布 `0.4.2-alpha` 小修复。

## 1. 先讲结论

`0.4.1-alpha` 已公开、已 stage、已 activation，Desktop runtime identity 和 provider readiness 也已通过。但稳定 health 仍是 `degraded`：`CodexPraetor-GenerationReconcile` 维护任务不存在。

这不是一个可以靠“再跑一次脚本”收口的环境小问题。它证明 release controller、维护任务、health 和 active receipt 还没有成为同一套不可矛盾的产品状态。因此：

1. 保留已推送的 `codex/fix-maintenance-release-closeout`，不创建单独 PR。
2. 不覆盖 `v0.4.1-alpha`；GitHub Release/tag 保持不可变。
3. 从最新 `main` 建立下一大版本分支，把该分支的维护任务 fail-closed 修复作为一个工作流并入。
4. 版本提升采用新的 alpha 大版本代际，具体编号以阶段 0 的 MCP/manifest 合同差异为准；本规划预期会改变 MCP 的状态表达，因此默认按 `0.5.0-alpha` 准备。

## 3. 为什么不能继续打散点补丁

历史报告原本已经提出四条尚未统一的链：任务治理、动态健康、运行态卫生、供应链交付。`0.4.0-alpha` 先落了任务账本的基础，`0.4.1-alpha` 落了 Desktop runtime 证明；维护任务失败则揭示这些组件仍各自判断成功。

如果此时单独合入 `0.4.2-alpha`，会继续形成“一个 PR 修 receipt、一个 PR 修 runtime、一个 PR 修 provider、一个 PR 修任务”的控制面碎片。用户仍无法只看一个状态回答：现在是不是能用、哪里被阻断、下一步谁该做什么。

## 4. 外部调研结论

| 来源 | 可采用的结论 | 本项目取舍 |
|---|---|---|
| Microsoft `schtasks` 文档 | `schtasks` 是 Windows 任务创建、查询和删除的官方命令面 | ScheduledTasks cmdlet 被拒绝时可作为 fallback；两条路径均须注册后查询验证 |
| GitHub Immutable Releases 文档 | 已发布 Release 的资产和 tag 不应被修改 | 修复必须用新版本/tag；不覆盖 `v0.4.1-alpha` |
| A2A task lifecycle | 任务、artifact、输入等待与终态需要分开 | 采用语义，不引入 A2A server |
| Temporal workflow execution | 事件历史、单写者、可恢复状态比覆盖式状态可靠 | 采用 append-only event/projection，不引入 Temporal |
| OpenTelemetry GenAI | 模型调用、工具、耗时、usage 和错误应可关联 | 先写本地 OTel-compatible JSONL，不部署遥测后端 |

外部资料只约束机制，项目优先级仍由本地真实派工复盘、stable receipt、health、worktree 和 CI 事实决定。

## 5. 目标：一个最小统一控制面

不引入第二个总控 agent、数据库、队列、Temporal、A2A HTTP server 或 worker 群聊。Codex 仍负责规划、监督和验收；外部 CLI worker 只执行受限 attempt。

```text
用户目标
  -> logical task + completion definition
  -> selection record (资格、适配、预算、止损)
  -> immutable attempt (worktree + contract + base commit)
  -> event/artifact evidence
  -> Codex verdict
  -> outcome/progress summary
  -> promotion/release controller
  -> active receipt + dynamic health + retirement
```

必须分开的状态：

- 执行：`draft/queued/running/cancel_requested/process_exited`
- 证据：`report_valid/artifact_valid/tests_passed/evidence_missing`
- 治理：`awaiting_supervisor/accepted/rejected/retryable/blocked/needs_decision`
- 发布：`built/published/staged/awaiting_evidence/active/delivery_blocked/delivered`

`process_exited`、`active receipt`、`GitHub Release exists` 都不能单独表示“产品已交付”。

## 6. 大 PR 工作流

### A. 治理内核（P0）

1. 统一 logical task、attempt、event、artifact、outcome schema，兼容读取旧 plan/job。
2. 给任务合同补 completion definition、base commit、write set、预算、stop-loss、期望工件和验收。
3. watcher/cancel 只写自己的事件；终态不可覆盖；Codex verdict 才能接受 logical task。
4. 对 plan 写入增加 revision/CAS、唯一临时文件和任务锁。
5. 回归覆盖 max-turns、tool denied、空报告、无 diff、取消竞态、并发写、write set 重叠。

### B. selection、outcome 与用户决策（P0/P1）

1. readiness 只回答“是否可安全运行”；selection 单独记录任务适配、候选、理由、预算和升级条件。
2. outcome ledger 记录 provider/model/task class/wall time/usage-or-unknown/verdict/失败类/证据/过期时间。
3. 首期仅用透明规则产生建议，禁止自动学习后悄悄改变高风险路由。
4. 增加中文 compact summary 与 `needs_decision`，用户不再需要读 JSONL 或日志。

### C. 产品真值（P0）

1. 将 `codex/fix-maintenance-release-closeout` 的 fail-closed 逻辑移植为 maintenance adapter：同一定义驱动 cmdlet、`schtasks`、查询、卸载和 health。
2. 加真实隔离 profile round-trip：安装、查询、禁用/漂移、卸载失败、fallback 双失败均可判定。
3. dynamic health 改为读取当前 CLI hash、expiry、generation、runtime、任务定义和 receipt；发布时 proof 只作历史证据。
4. runtime inventory 分类 active、retired、clean+merged、dirty/unmerged、审计保留、test scratch；默认只报告。

### D. release controller 与供应链（P1）

1. 收束安装、插件/Skill/cache 发布、stage、activate、maintenance、retirement 到一个 controller，旧命令保留兼容入口但不再各自写事实。
2. build 生成并校验 SBOM、provenance/attestation 输入和远端下载验收。
3. GitHub Actions 最小权限、第三方 action SHA pin、Dependabot 配置、CodeQL/secret scanning 验收。
4. 将管理员动作单列为合并后门禁：main ruleset/required checks、immutable releases、private vulnerability reporting、自动删分支。

### E. MCP、文档与版本（P1）

1. MCP 仅扩展统一状态摘要、task/outcome/timeline 查询；不把 MCP 当调度器。
2. 重写安装、排错、卸载、release gate 和路线图，以用户能理解的状态为主。
3. 因状态输出和 manifest/Skill 内容会变化，按新版本 generation 做一次 fresh-context native canary。

## 7. 实施顺序和通过门

| 阶段 | 产物 | 通过条件 |
|---|---|---|
| 0 架构冻结 | ADR、schema、状态转换表、验收矩阵 | 每个字段能对应真实里程碑 A 失败；范围明确排除项 |
| 1 治理内核 | logical task/attempt/event/outcome | 不再把进程退出写成任务完成 |
| 2 product truth | maintenance/dynamic health/inventory | 无任务、漂移、过期、fallback 失败均 fail closed |
| 3 delivery | controller、包、供应链、文档 | 代码、Release、stage、active、health、远端用户路径可一致解释 |
| 4 PR 前 | 测试、构建、packaging、CI、文档、diff 审计 | 只剩用户的 PR 创建/合并与管理员设置 |
| 5 合并后 | 新 Release 与真实安装验收 | health=ready、active receipt、远端包、维护任务、retirement 状态均通过 |

## 8. 不做事项

- 不引入 Temporal、Redis、数据库、A2A server、通用多 agent 平台或 worker 群聊。
- 不读取 provider 账号数据库、余额、token、cookie 或截图。
- 不强删 dirty/unmerged worktree 或占用中的 generation。
- 不覆盖旧 Release/tag，不把 GitHub 管理员设置伪装成代码已完成。

## 9. 分支和提交策略

1. 保留 `codex/fix-maintenance-release-closeout`，不提 PR、不发布。
2. 从最新 `origin/main` 新建大版本分支。
3. 只选择性移植该分支中的 maintenance adapter、测试和失败语义；不直接合并旧 `codex/operational-truth-0-3-1`。
4. 一个大 PR 内按工作流分原子提交；最终只创建一个中文发布影响 PR。

## 11. 当前决定

- 本报告完成后，立即进入阶段 0 和阶段 1 的实现；不再单独推进 `0.4.2-alpha` 小 PR。
- 任何阶段发现范围超出“Codex 对外 CLI 派工治理与交付真值”时，停止扩张并记录为后续候选。
