# Codex 执政官系统整改与单 PR 实施方案

日期：2026-07-16
状态：仅调查与方案；本报告不修改产品代码、不运行真实 worker、不提交或发布。

## 1. 结论先行

这不是“把外部 Agent 限制成只读/搜索工具”的问题。正确的产品边界应是：

1. Codex 是编排者：负责拆分、外部研究、权限决策、主线集成、测试选择和最终验收。
2. KnowledgeRadar 是外部联网研究的主感知层：凡是需要联网搜索、来源发现、事实核查、跨来源取证的任务，都由 Codex 调用 KnowledgeRadar；不把这类任务交给 Qoder、CodeBuddy 或 MiMo。
3. 外部 Agent 是受控代码 worker：它们可以在明确任务合同下读取本地代码、修改代码、运行约定的验证；代码只能写入 Codex 创建的独立 Git worktree，不能直接写主工作区。
4. Codex 只在读取 worker 的结构化结果、审查 diff、运行验证并记录 accepted 后，才把变更整合入主线。

当前实现的问题不是 worker “能力不足”，而是产品把“路由、权限、隔离、健康、安装、验收、展示”拆成了独立脚本和副本，却没有把它们约束成同一套运行时合同。结果是：版本可以混装，canary 不参与路由，权限名不等于真实权限，进程退出不等于任务成功，MCP 工具名存在也不等于用户正在调用的就是新版本。

本报告建议把整改收敛到一个发布影响 PR 中，但在 PR 内按依赖顺序完成五个可单独验证的提交段：运行时代际、健康/readiness、任务合同与 worktree、job 生命周期、MCP 可观测性与发布门禁。这样满足“一个 PR 一次收口”，同时避免在一个不可验证的大改动里混入多个根因。

## 4. 为什么“越修越差”

历史修复大多是局部补救，例如调整 MiMo task packet、把 MiMo 放进 worktree、给 setup 加 canary、给 MCP 增加 tool。它们没有建立下列跨层不变量：

~~~text
一个用户可见版本
  = 一次构建生成的 Skill + Plugin + MCP + wrapper protocol
  = 一次安装 generation
  = 一次新线程中观察到的原生 MCP 工具集合
  = 一个仍有效的 provider capability tuple
~~~

所以系统复杂度增长后，旧的“脚本能 dry-run”“CLI --version 成功”“MCP protocol smoke 通过”仍然能给出成功信号，却无法覆盖真正用户入口。问题不是修复没有执行，而是验收对象与故障对象不同。

## 5. 目标产品模型

### 5.1 任务类别先于 provider 选择

| 任务类别 | 执行者 | 外网 | 写入位置 | 结果门槛 |
| --- | --- | --- | --- | --- |
| external_research | Codex + KnowledgeRadar | 允许，且必须走 KR 路由 | 报告/证据由 Codex 决定 | 来源、证据登记、交叉核查 |
| local_audit | Codex 或外部 worker | 外部 worker 默认禁止联网 | 主目录只读；MiMo 也进 disposable worktree | 文件证据与结论 |
| code_change | 外部 worker | 默认禁止联网 | Codex 创建的 worktree | diff、限定测试、结构化报告 |
| integration_and_acceptance | Codex | 按需要 | 主工作区仅由 Codex 操作 | review、测试、accepted/rejected |
| release_or_account_action | Codex + 用户 | 不自动委托 | 不允许 worker 执行 | 明确人工边界 |

“搜索”必须拆开理解：

- 本地仓库搜索，如 rg、符号索引、读取文件，是 local_audit/code_change 的正常能力。
- 外部联网搜索、查资料、事实核查，是 external_research，只能由 Codex + KnowledgeRadar 执行。

这既保留外部 worker 的编码能力，也避免让其缺少 KnowledgeRadar 感知覆盖、来源治理和证据登记时承担调研任务。

### 5.2 任务合同替代 readonly/edit 二元标签

每个派工请求在调用 provider 前必须编译为一个不可变 task contract，至少包含：

~~~json
{
  "schema": "codex-praetor-task-contract/v2",
  "job_id": "GUID",
  "task_kind": "code_change",
  "repo": "absolute path",
  "base_commit": "full SHA",
  "worktree": "absolute path",
  "allowed_paths": ["mcp/src/**", "scripts/dispatch/**"],
  "forbidden_paths": [".git/**", ".env*", "auth/**", "node_modules/**"],
  "network_policy": "worker_network_forbidden",
  "provider": "qoder",
  "provider_tuple": "hash",
  "permission_profile": "edit-worktree-v1",
  "acceptance": ["targeted tests", "git diff --check", "no main checkout changes"],
  "max_runtime_seconds": 1200
}
~~~

合同不是 prompt 文案。wrapper、job 元数据、MCP 响应、canary、验收器都必须引用同一个 contract hash。任何 provider 只能获得与合同匹配的命令参数；任何 worker 输出都必须回报同一 job_id 与 contract hash。

### 5.3 worktree 是隔离边界，不是权限宣传

所有外部 worker，包括 local_audit，统一在 Codex 创建的 disposable worktree 中执行。这样 MiMo 的 .mimocode/session 文件不会污染主目录，也能让 job 的文件系统副作用可审计。

对 code_change：

1. Codex 先固定 base SHA、GUID job_id、branch 和 worktree。
2. worker 只能在该 worktree 中写入；禁止 commit、push、修改 Git 配置和访问主工作区。
3. 结束后由 Codex 运行 diff allowlist、git diff --check、目标测试和秘密/生成物检查。
4. 只有 accepted 的 patch 才由 Codex 以明确的合并步骤进入主线。

Git worktree 能隔离分支和工作目录，不能隔离凭证、端口、数据库、同机进程和操作系统权限。因此 provider 权限规则、路径 allowlist、测试沙箱和最终验证仍然必需。

## 6. 健康检测：从 doctor 升级为 readiness gate

新健康端点不应只返回 PASS/WARN/FAIL，而应返回分层事实和可用结论：

| 层 | 必查项 | 成功定义 | 失败后的行为 |
| --- | --- | --- | --- |
| install_generation | Plugin、Skill、MCP、wrapper protocol、file manifest、build id | 同一 generation，且 native 工具集合匹配 | real dispatch fail closed；保留 status/doctor/dry-run |
| mcp_runtime | 当前线程可见工具名、runtime_info、协议版本 | 当前 tool surface 与 generation 契约一致 | 要求 reload/新线程并显示具体缺失项 |
| provider_presence | CLI 路径、来源、SHA256、精确版本、OS | 受支持 tuple 可识别 | disabled 或 blocked |
| provider_capability | 当前 tuple 的真实 canary、过期时间、失败分类 | 对应 task profile 最近通过 | 不进入 auto 候选池 |
| job_runtime | job state、watcher/worker identity、timeout、剩余时间 | 可观察并可取消 | timed_out/watcher_failed/cancelled |
| research_dependency | KR tool surface 与具体 research 路线 | external_research 可被 Codex 执行 | 不把研究任务转交 worker |

状态统一为 unknown、ready、degraded、cooldown、blocked、disabled、manual_required、stale。不要再用一个 ready 掩盖“CLI 存在、认证未知、canary 过期”的混合状态。

provider readiness 的 key 必须覆盖完整 tuple：

~~~text
provider + CLI source + CLI file hash + CLI version + model + agent
+ permission profile + tool profile + OS + wrapper protocol + contract schema
~~~

任何字段改变，旧 canary 自动失效。auto 路由的顺序应固定为：

~~~text
任务类别/禁止边界
-> 安全与权限合同适配
-> install generation 一致性
-> provider readiness 与 canary
-> 最近失败、熔断、冷却
-> 成本、时间、偏好
~~~

而不是当前的“时间/成本先选 tier，再希望它能运行”。

## 7. provider 合同重做

### 7.1 CodeBuddy

CodeBuddy 官方文档说明 permission mode 只决定会话基线；deny/allow/ask 规则具有更高优先级，且支持 --allowedTools、--disallowedTools 与工具级规则。因此整改不能继续以“--tools 列表就是只读保证”为前提。

本 PR 要做：

- 把 CodeBuddy 合同拆为 local-audit-v1 与 edit-worktree-v1，不再用 readonly 这个容易误导的产品名。
- 在受控 worktree 中以官方当前 CLI 版本跑矩阵 canary：可读、可编辑、被禁止 Bash、被禁止 WebFetch、语义失败、JSON/text 输出。
- 优先验证显式 deny 与 allowed/disallowed 规则能让模型获得“permission denied”的可分类失败；若当前 CLI 的 --tools 仍导致 tool-not-found，则该 tuple 直接 blocked，不能进 auto。
- CodeBuddy CLI 来源要记录为 standalone 或 bundled。若仍支持 WorkBuddy 内嵌路径，必须将路径 hash 纳入 tuple，桌面更新后自动令 canary 失效。

### 7.2 Qoder

Qoder 官方文档明确区分 permission mode 与 Plan work state，并说明 headless 下 ask 会自动变成 deny。现有 dont_ask + tool whitelist 是可继续验证的候选，不是永久安全结论。

本 PR 要做：

- 固化 Qoder local-audit 与 edit-worktree 两套 profile。
- canary 必须验证当前版本、当前模型、当前 mode、被拒绝工具的分类、edit diff 和测试。
- 不把 Plan 当作权限模式；Plan 只用于需要方案的 task kind，不能拿来代替文件系统隔离。

### 7.3 MiMo

MiMo 官方 README 明示 build 是默认的全工具开发 agent，plan 是只读分析 agent。当前产品曾发现 plan 可能忽略任务包，于是让 agent 为空并落到默认 build。这条路线可以作为“隔离编码 worker”，不能继续命名为 readonly。

本 PR 要做：

- 删除 mimo-auto-readonly 这一安全性暗示，迁移为 mimo-isolated-audit（仅在真实 canary 证明无写入时才可用）与 mimo-edit-worktree。
- 若 plan agent 不能可靠执行 task contract，则把 MiMo audit 标为 unsupported/blocked，而不是以默认 build 冒充只读。
- 所有 MiMo 都使用 GUID job id 命名 worktree；检查 .mimocode、副作用和主目录不变。

## 8. durable job 生命周期与取消

blocking/background 不再拥有两套执行主线。两者都先创建 job，再由同一 watcher 与状态机驱动：

~~~text
planned -> eligibility_checked -> worktree_ready -> starting -> running
running -> worker_completed -> awaiting_codex_verification -> accepted | rejected
running -> failed | timed_out | cancelled | manual_required | watcher_failed
~~~

每个 job 在启动 worker 前落盘 job.json，至少记录 job_id、contract hash、base SHA、worktree、provider tuple、worker PID、PID start time、command hash、deadline 和 event list。

必须新增：

- timeout_seconds 与 deadline。
- codex_praetor_cancel_job(job_id)。
- Windows Job Object 进程树管理；优先以 Job Object 托管 worker，cancel/timeout 时通过 TerminateJobObject 结束整棵树。若环境不允许，才用 Process.Kill(entireProcessTree: true) 兜底，并等待退出。
- cancel 前比对 PID、启动时间、command hash，避免 PID 重用或用模糊命令行误杀。
- stdout/stderr、completion、结构化事件对 blocking 和 background 一致保存。
- worktree 只在 completion 已写入、进程树已退出、Codex verdict 已终态后允许清理。

## 9. MCP 展示与 UI：能保证什么，不能假装什么

用户要看到的不是“运行了一条命令”，而是“Codex 为什么选择某个 Agent、它正在处理哪份合同、现在处于什么状态、结果是否已经被 Codex 接受”。这可以在本 PR 内做到。

MCP 每个涉及派工的返回都要包含稳定的中文摘要对象：

~~~json
{
  "display": {
    "阶段": "已启动 worker",
    "任务": "修复 MCP 路径解析",
    "执行者": "Qoder",
    "模型": "Qwen3.7-Plus",
    "权限合同": "edit-worktree-v1",
    "工作区": "隔离 worktree",
    "状态": "running",
    "下一步": "等待 completion，再由 Codex 验收"
  },
  "job_id": "...",
  "contract_hash": "...",
  "timeline": ["planned", "eligibility_checked", "worktree_ready", "running"]
}
~~~

需要的 MCP 调整：

- 增加 codex_praetor_runtime_info、codex_praetor_health、codex_praetor_job_timeline、codex_praetor_cancel_job。
- dispatch/result/status 的首段输出固定为 display 摘要；原始命令和日志只作为受限诊断字段，不放在用户第一眼。
- 后台完成通知只发送简短中文最终状态、provider、job_id、验收待办；完整事实仍以 completion.json 为准。
- plan task 的 accepted/rejected 状态必须显示为“Codex 已验收/未验收”，不能把 exit 0 显示成完成。

边界必须如实写进产品文档：

- MCP server 可以改善工具卡的名称、描述与返回结构；它不能控制 Codex Desktop 的思维链渲染，也不能要求宿主显示 worker 的内部推理。
- 当前插件没有已验证的 Codex Desktop 自定义面板或流式 UI 协议。不能在本 PR 中承诺“原生实时 dashboard”。
- 可以展示的是真实 job 生命周期事件和最终摘要，不应尝试暴露外部模型的链式思考。
- 若以后要做独立面板，应先取得 Codex Desktop 对插件 UI/进度通知的官方支持证据；ChatGPT Apps SDK 文档不能自动证明 Codex Desktop 插件也支持相同 UI。

## 10. 单一 PR 的实施范围

建议 PR 标题：

~~~text
建立 Codex 执政官运行时合同、健康门禁与受控 worktree 派工闭环
~~~

建议按以下五段提交，但只提交一个 PR：

本 PR 不应做的事：

- 不自动登录 provider，不读取 token/cookie/账号数据库。
- 不把外部联网研究派给 provider。
- 不再宣称 MiMo 默认 Build 是 readonly。
- 不用命令行字符串搜索/强杀进程作为正常取消机制。
- 不让公共 GitHub runner 持有三家 provider 的私人账号。
- 不承诺 Codex Desktop 会展示不可验证的原生实时 UI。

## 11. 验收矩阵与发布门禁

公共 CI 保留可复现的结构和协议测试；真实 provider 验收放到受控 Windows 本机或 self-hosted release-candidate 环境。两者都必须明确命名，不能再用前者替代后者。

最低验收矩阵：

| 场景 | 必须证明 |
| --- | --- |
| 新 Skill + 旧 cache | health=blocked，真实 dispatch 拒绝，doctor 给出修复指引 |
| 新 release 从 zip 安装 | generation 一致；新 Codex 线程可见声明中的 MCP 工具集合 |
| external_research 意图 | 路由为 Codex + KR，不产生 provider dry-run/dispatch |
| local_audit | worker 在 disposable worktree，主 checkout 无任何状态变化 |
| code_change | worker 在 worktree 产出 diff；Codex acceptance 前主线不变 |
| Qoder/CodeBuddy/MiMo 每个支持 tuple | path、version、hash、profile、marker、被拒工具、contract hash 全部记录 |
| CodeBuddy tool mismatch | 归类为 tool_contract_mismatch，tuple blocked，auto 不再选择 |
| MiMo plan 不可靠或 Build 会写入 | 不可称 readonly；profile blocked 或改为 isolated edit |
| semantic exit 0 | completion 为 failed/inconclusive，不能进入 awaiting_verification |
| 两个同秒 job | GUID worktree 无碰撞 |
| timeout/cancel | 进程树退出、终态正确、锁/worktree 不提前释放 |
| MCP 展示 | tool result 首屏显示执行者、合同、阶段、下一步，而不是裸命令 |

发布影响 PR 的最终收口还需要：

1. 合并最新 main 后创建新版本/tag；不得覆盖 v0.1.3-alpha。
2. 构建 GitHub Release zip 和 sha256。
3. 从远端 Release 重新下载，安装到干净用户目录。
4. 新开 Codex 线程完成 native MCP 工具面探针。
5. 对被声明支持的 provider tuple 运行真实 canary；不支持者必须在 health 中显式 disabled/blocked，绝不进入 auto。

## 13. 最终判断

下一次整改的成功标准不是“又一次真实 worker 跑通”，而是下面四个条件同时成立：

~~~text
用户实际加载的版本可证明
-> 当前任务合同可证明
-> 被选 provider 的当前 tuple 可证明
-> worker 结束、验收、主线整合的每一步可证明
~~~

达到这四点后，外部 Agent 可以承担真正的代码修改任务，而不是只做只读装饰；KnowledgeRadar 也会留在它真正有优势的外部研究位置；Codex 则作为唯一具有全局上下文和最终合并权的编排者，保持对主线质量的控制。
