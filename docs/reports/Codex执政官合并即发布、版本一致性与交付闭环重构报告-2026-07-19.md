# Codex 执政官合并即发布、版本一致性与交付闭环重构报告

日期：2026-07-19
项目：D:\Projects\CodexPraetor
范围：当前 main 考古、历史流程对账、外部发布工程调研、下一阶段大 PR 设计与实现验收。

## 1. 结论

反复出现“代码已合并，但产品没有发布”的根因不是某个发布脚本漏跑，而是流程把用户可见变更拆成了两个没有共同门禁的事务：

PR 合并 -> CI 验证 -> 人工决定版本、构建、tag、Release、远端复验。

CI 只证明源码可以通过测试，发布脚本只在合并后等待人工调用；版本又散落在源码、Plugin、Skill、MCP、README、安装器和脚本默认值中。于是合并可以成功，而 Release 仍指向旧 commit，stable active 仍指向旧 generation。

新流程改成：

发布影响 PR -> 同一 PR 提交版本面、release intent、release notes -> required checks 证明候选包和用户入口一致 -> 合并 main -> Release On Main 自动从精确合并 commit 构建、发布、下载复验 -> 远端 Release 成功后进入下一次开发。

GitHub commit 合并和 GitHub Release API 写入不是同一原子事务。不能保证网络请求永不失败；能做到的是让候选在合并前已验证，让合并后的发布由唯一、幂等、可重试且公开失败状态的 workflow 自动完成，不再依赖人工记得再发一次。

## 3. 根因拆解

### 3.1 合并门禁没有把发布意图前移

PR 模板没有强制声明版本/tag，CI 也没有检查 release intent。发布影响代码可以合法进入 main，却没有保证 main 上存在对应版本、release notes 和不可变 tag。

### 3.2 版本没有 canonical source

runtime contract、Plugin manifest、MCP package、MCP server、setup、README、安装文档、roadmap、发布脚本和测试默认值都能成为“看起来正确”的版本来源。generation hash 不能阻止版本文字在合并前分叉。

### 3.3 CI 与 Release 是两个系统

原 CI 在 PR/push 上跑测试和打包，但没有创建 Release。发布脚本要求人工回到 main 后再次运行，这就是尾巴的直接来源。

### 3.4 不可变 tag 没有转成 PR 阻断

旧 v0.5.0-alpha 指向 a0b4ccc，新合并 commit 是 f292431。正确行为是拒绝覆盖旧 tag；错误行为是 merge 先完成，然后收口时才发现不能发。

### 3.5 本机 host 生命周期与公开发布被混在一起

Release 成功也不能让当前 Codex 对话瞬间重建 MCP 工具快照。MCP tools/list_changed 是通知能力，不是当前对话热替换保证。stable stage、fresh context、provider readiness、activate 和 user-path proof 必须单独表达。

### 3.6 隔离 smoke 不能冒充真实交付

公共 CI 不应依赖开发者 provider 登录和 Desktop 进程；fake readiness、隔离 profile 和 protocol smoke 只能证明工程骨架，不能证明当前用户 host 已刷新或 provider canary 已通过。

## 4. 外部调研结论

| 主题 | 证据结论 | 对本项目的约束 |
|---|---|---|
| GitHub immutable releases | tag 和 assets 发布后不可修改；建议 draft 后上传全部资产再 publish | 旧 tag 永远不能装新代码 |
| Artifact attestations | workflow、repo、environment、commit 和 artifact provenance 可绑定 | generation、hash、workflow、Release 必须可回查 |
| GitHub Environments | 支持 required reviewers、wait timer、部署分支和 secrets | 一次性配置保护 main 和 workflow write 权限 |
| release-please | release PR 汇总版本/changelog，合并后创建 Release | 证明版本意图必须进入 PR；本项目不采用合并后 release PR 尾巴 |
| Changesets | changeset、version、publish 分层，版本变更先合并 | 本项目采用同 PR release intent |
| Trunk-based Development | 高吞吐 CD 从 trunk 发布，release branch 只在必要时创建 | main 作为唯一发布源 |
| Argo CD/GitOps | Git desired state 驱动自动同步，健康和 rollback 独立 | 用 controller + health projection 管理状态 |
| MCP tools specification | listChanged 是通知能力，不承诺已有对话热替换 | host refresh 是必要状态 |
| IEEE CI/CD review | 测试、透明度、基础设施和部署可靠性是关键因素 | 测试、构建、发布、运行态不能割裂 |
| SANER release engineering | 发布工程覆盖集成、CI、构建、部署、发布全链路 | Release 不是单独的 zip 命令 |

## 5. 新流程

发布影响 PR 必须包含 release-intent.json、release-intent.schema.json、同版本 runtime contract/Plugin/Skill/MCP/安装入口/公开文档、release notes、changelog，以及候选包和确定性验证。

test-release-intent.ps1 阻断版本分叉、缺少 release intent、没有真实产品变更和复用远端 tag。

Release On Main 在带 release intent 的 main push 上执行：checkout 精确合并 commit，验证 intent 和不可复用 tag，运行 MCP/产品/公开入口/确定性门禁，创建 tag 和 draft Release，上传 zip 与 sha256，发布 immutable Release，下载远端资产并执行 verify-github-release-asset.ps1。

workflow 失败必须公开为交付 incident 并支持重试；不能让下一次 PR 覆盖失败状态。

本机投影仍是 release_published -> remote_asset_verified -> stable_stage -> awaiting_host_refresh -> fresh_context_passed -> provider_readiness_passed -> active -> user_path_passed -> delivered。这不是人工发布尾巴，而是宿主进程、provider CLI 和用户机器都不在 GitHub runner 内的客观边界。

## 7. 必须显式承认的边界

- GitHub Actions 无法重启用户当前 Codex Desktop；host refresh 不能被远端 workflow 伪造。
- provider 登录和 CLI 版本会漂移，公共 CI 不能替用户登录。
- stable activation 需要用户机器上的 fresh-context、provider readiness 和维护任务权限。
- 旧 generation 被进程占用时只能延迟回收，不能强杀。
- GitHub API/网络失败不可能被代码证明为永不发生；只能依靠幂等、重试、公开 workflow 状态和不可变 tag 降低风险。

## 8. PR 通过标准

- main 保护规则要求 CI 和 release-intent gate。
- 发布影响 PR 没有 release intent 时不能合并。
- 版本、tag、artifact、Release notes 和公共入口来自同一个 intent/version surface。
- 合并后只有 Release On Main 能创建公开 Release。
- 旧 tag、旧 asset 和旧 active generation 不会被覆盖。
- workflow 远端下载复验通过后，才把公开 Release 标记为发布成功。
- 本机 stable 的 host/provider/user-path 状态由 receipt 单独记录，不能伪称已热更新。

## 9. 结论状态

本轮目标不是把 0.5.0-alpha 旧 Release 修成新代码，而是将下一次发布升级为 0.6.0-alpha，并把版本与自动发布契约一起放进 PR。这样未来一次发布影响 PR 合并后，不需要再人为开发布尾巴；唯一剩余阶段是远端无法替用户重启本机 host 的运行态投影。
