# Codex 执政官统一证据链治理审计与修复说明

## 结论

这不是再给 MiMo 打一个补丁，而是修复了两条会把“没有成功”写成“成功”的系统链路。

第一条链路是能力 canary：外层调度器会回显命令，而命令里本来就含成功标记。旧实现只要在外层输出里看到这个标记，就给 provider 写入“可用”。因此 MiMo 明明返回了风控拒绝，系统仍可能把它展示成可小范围使用。

第二条链路是 worker 终态：Windows 下旧启动方式在重定向两个日志流时可能拿不到进程退出码；旧代码把空值强转为 `0`。于是一个无法确认失败的 worker 会进入“等待 Codex 验收”。

本次统一修复的目标是：**只有真实 worker 日志、完成回执和退出码三者一致时，才产生 readiness；任何退出码异常、回执异常或观察不到退出码的任务都进入拒绝/阻断，而不是等待验收。**

## 已验证的事实

| 事实 | 证据 | 结论 |
| --- | --- | --- |
| CodeBuddy、Qoder 的最小只读 worker 曾真实返回标记并退出成功 | 隔离 worktree 的 `stdout.log`、`completion.json` 与无主检出漂移记录 | 这两个 tuple 有过真实样本，不等于所有任务都可派。 |
| MiMo 本次真实请求返回 HTTP 400、代码 441、`risk_control` | worker 原始 stdout；watcher 已归类为 `provider_risk_control` | 是 provider 风控，不能重试刷过去，也不是 worktree 故障。 |
| 旧 canary 会把 wrapper 回显当 worker 输出 | 新增故障注入：wrapper 含标记、真实 stdout 只有风控文本、completion 为失败 | 旧测试会误绿；新测试必须失败且不得新增 readiness。 |
| 旧 watcher 会将空退出码转成 0 | Windows 进程重定向复现；新 lifecycle 注入 worker `exit 7` | 这是本地进程采集缺陷，必须和 provider 失败同样 fail-closed。 |

## 修复设计

1. canary 从 `job.json` 找到真实 `stdout.log` 与 `completion.json`；只检查真实 stdout 的标记，且要求 `process_exited`、退出码 0、无 failure class。
2. readiness 条目升级为 v3，保存 job 标识、真实 stdout 哈希、completion 哈希与终态摘要。旧 v2 条目仍可读取为历史资料，但不能授权新派工。
3. `provider_operations` 同样只统计带完整 canary 回执的 readiness，避免界面把旧假凭据说成可用。
4. watcher 改用 .NET `Process`，异步排空两路输出并读取同一进程的退出码；非零退出归类为 `worker_process_failed`，观察不到退出码归类为 `worker_exit_code_unavailable`。两者都拒绝，不进入人工验收队列。
5. 任务账本只允许 `process_exited + exit_code=0 + 空 failure_class` 进入 `awaiting_verification`；其余终态都记录为失败或阻断。

## MiMo 的产品决定

MiMo 不永久删除，但当前 `mimo/mimo-auto` tuple 是**不可自动路由**：不自动重试、不用旧 readiness、不把“免费”或 CLI 可启动当作可用。只有用户按官方路径处理账户/风控后，在新的隔离 worktree 中通过一次最小 canary，才可重新进入受控验证。若之后仍无法获得连续真实任务证据，应保留为手动 opt-in 实验提供方，而不是默认候选。

本项目不读取、复制、修改 MiMo 的认证文件、数据库或缓存；也不采用社区代理绕过方案。

## 故障注入覆盖

本次回归至少覆盖：

- wrapper 回显成功标记而真实 worker 回执失败；
- completion 失败、stdout 无成功标记；
- 非零退出但 stderr 没有可识别关键词；
- 风控 441、超轮数、超时、取消和部分 worktree 改动；
- 旧/无回执 readiness；
- 多 generation readiness 与 CLI hash 漂移；
- worker 完成但未获 Codex 验收时，依赖任务不得解锁。

仍无法保证所有外部 provider 永不故障；能保证的是，本轮已触及的失败不再被伪装成成功，且出现新 provider 故障时会停止在正确状态而不是制造新的“收口后补丁”。

## 外部调研与采用边界

- OpenAI 的 Codex 非交互文档区分最终输出与 JSON 事件流；本项目借鉴其“不能用单一文本当完整事实”的原则，而不引入另一套执行器。
- LangGraph durable execution 文档区分执行 checkpoint 和长期 store；本项目据此保持 job completion、supervisor verdict、capability profile 三种状态分离。
- GitHub artifact attestation 文档说明构建来源证明能验证 artifact 身份；本项目继续以同一不可变 zip 完成发布验真，但不把它混同为 provider 成功证据。
- MiMo 官方排错文档建议查日志、升级或经 `/connect` 处理 provider 配置；官方 API 文档对频率/额度问题建议退避或降频，但没有将本次 441 说明为可自动重试。社区对 “Illegal Access” 的代理经验不是官方保证，不能进入默认路径。
