# Codex 执政官单插件单入口发布体系治理报告

日期：2026-07-20

## 结论

此前的问题不是单次 Desktop 刷新失败，而是把一个插件拆成了全局 Skill、marketplace 插件、手工 cache、active receipt 和维护任务等多个可独立漂移的“正式入口”。本次将产品边界收敛为一个插件包：`plugin/` 内含 Skill、MCP 和精确 generation manifest；GitHub Release 的同一 zip 是唯一发布事实。

## 已挖出的同类问题与处理

| 问题 | 根因 | 永久控制 |
| --- | --- | --- |
| 新插件已加载但 health 仍 blocked | 旧 active receipt 被当作运行时权威 | health 仅把旧 receipt 当诊断；运行合同、marketplace 与实际缓存版本是运行证据 |
| canary 写出 `--packaged--` generation | 打包后的 Skill 找不到仓库 generation 脚本 | 构建时把同一 generation manifest 写入 `plugin/release-generation.json` |
| 同一 Skill 有多份独立入口 | 全局 Skill、源码 Skill、插件 Skill 都可被运行 | 只有插件内 Skill 是运行入口；根 Skill 只是派生兼容镜像 |
| 发布器手工写 Codex cache | 把宿主内部缓存误当作产品安装面 | 安装器仅更新 marketplace source，Codex 自己管理 cache |
| stage/activate/deliver 形成第二个发布状态机 | 本机状态被错误提升为公开产品状态 | `complete-codex-praetor-release.ps1` 已退役；远端同一 artifact 验证决定公开交付 |
| Dependabot PR 必然失败 | 开发依赖变更被一律认定为发版变更 | 仅开发依赖/lockfile 变更仍跑 CI，但不强制伪造版本/tag；运行时依赖仍严格发版 |
| 合并分支长期堆积 | GitHub 未开启自动删分支 | 已开启 `delete_branch_on_merge`；同时启用漏洞告警与 Dependabot 安全更新 |

## 仍然存在但不再是缺陷的边界

正在运行的 Desktop MCP 进程不能热替换。候选版本通过仓库测试和 zip runtime 验收后，用户只需在安装后重启 Desktop 并在一个新对话调用 `runtime_info`。这是一台机器的安装刷新，不是新的 Release 阻断或另一个修复 PR。

## 验证

- MCP `npm test`、bundled runtime smoke、artifact runtime、确定性 zip、plugin-only isolation 均通过。
- 隔离安装证明：不创建全局 Skill，不写 Codex cache，插件内 generation 与 zip generation 一致。
- GitHub 主线保护保留；自动删分支、漏洞告警和 Dependabot security updates 已启用。
