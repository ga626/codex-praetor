## 一、结论先行

上一阶段把 Codex Praetor 做成了一个能调起外部 CLI worker 的产品，也补了发布代际、provider readiness、worktree、job、completion 和 release receipt。真实派工测试进一步证明：它还没有成为可靠的任务治理系统。

问题不在于“缺一个更聪明的总控模型”，也不在于“再接一个多 agent 框架”。外部调研后的结论是：**目前没有一个成熟框架能不改变产品边界地直接替代 Codex Praetor。**

- OpenAI Agents SDK、LangGraph、AutoGen、Semantic Kernel、Google ADK、CrewAI 等，都是应用开发者自己运行的模型编排运行时。直接引入会增加一个与 Codex Desktop 竞争的总控、额外模型调用和新的权限/状态面。
- Temporal 是成熟的耐久执行平台，但引入 server、worker、部署和运维；当前单用户、本机、少量外部 CLI job 不值得承担这个体积。
- A2A 是跨系统 agent 通信协议，MCP 是 host 与工具/上下文协议。二者都不是本机 worker 调度器；Qoder/CodeBuddy/MiMo 也不是可直接接入的 A2A agent。
- Claude Code Agent Teams 的共享任务、所有权和通信有价值，但不能把 Claude 的 agent runtime 当成外部 CLI worker 的通用替代。

但这不等于继续“纯手搓”。应把成熟机制拆到正确层级：

1. **直接采用标准语义**：A2A 的 task/context/artifact/lifecycle、OpenTelemetry GenAI 的 trace/usage 字段、JSON Schema 的 worker 报告合同。
2. **借鉴而不引入运行时**：Temporal 的不可变事件历史/重试边界，Google ADK 的顺序/并行/循环模式，Claude Code 的任务所有权/隔离，Magentic-One 的任务账本和进度账本，LangGraph 的模式选择与人机中断。
3. **明确延后**：A2A HTTP server、Temporal service、通用群聊/agent team、第二个 LLM supervisor、自动账户成本抓取、跨机多租户调度。

下一阶段应把原“里程碑 A”升级为一个完整但有内部阶段门的 **`0.4.0-alpha 任务治理与产品真值大 PR`**。不能继续以 `0.3.1-alpha` 名义做零散 health 补丁，因为 task/job/plan/MCP 状态会发生对外可见合同变化。它的目标是：

> **用户只描述目标；系统先把目标变成可验收、可预算、可恢复的任务；只派适合的 worker；失败后能解释、学习和止损；用户只在真正需要决策时被打扰；最终产品状态、发布状态和运行状态说同一种真话。**

## 三、外部机制地图：别人真正解决了什么

### 3.1 先选择协作模式，而不是默认“多 agent 越多越好”

[Semantic Kernel 的编排模式](https://learn.microsoft.com/en-us/semantic-kernel/frameworks/agent/agent-orchestration/)把协作分为 concurrent、sequential、handoff、group chat、Magentic；[Google ADK 的 workflow agents](https://google.github.io/adk-docs/agents/workflow-agents/)进一步强调顺序、并行和循环是可预测的确定性控制流。它们共同表达了一个被当前项目忽略的原则：

| 模式 | 何时适合 | 对 Codex Praetor 的结论 |
|---|---|---|
| 顺序 | 后一步依赖前一步产物或验收 | 应成为同一代码区域/同一 logical task 的默认 |
| 并行 | 读任务、独立假设、无重叠写集合 | 只在 write set 明确不重叠时允许 |
| handoff | 已有明确失败类、需要换专长/权限/预算 | 应由失败策略触发，不是“随便再试一个 provider” |
| loop | 有明确收敛条件和最大次数 | 可用于测试修复，但必须有预算和 stop-loss |
| group chat / dynamic manager | 探索性强、路径未知、agent 间确实需要互相质询 | 不是代码改动的默认模式，成本和上下文风险高 |

本次里程碑 A 的维护任务实际应是**顺序**：先证明 task definition/fallback 的等价性，再做 uninstall safety；却被拆成多个没有 `depends_on` 的 task，repair 未完全验收时又开始专项测试。这不是 worker 不够聪明，而是没有先选择协作模式。

### 3.2 成熟系统都把“任务”当作不可变业务单位，而不是一次进程

[A2A 的 Task 生命周期](https://a2a-protocol.org/latest/topics/life-of-a-task/)把普通消息与长任务分开：任务有 `contextId`、`taskId`、artifact，并区分 `input-required`、`auth-required`、`completed`、`canceled`、`rejected`、`failed`。终态不可重启，后续修正是在同一 context 下创建新的 task，保留可追溯关系。

当前项目的 `job` 已经有 durable 文件，但语义仍停在“进程退出记录”。真实测试中，一个维护目标被拆成 `maintenance_adapter`、`maintenance_core_qoder`、`maintenance_adapter_repair`、`maintenance_uninstall_failure` 四个孤立 task；它们没有 attempt/refinement 关系，没有一个业务完成定义。

应直接采用 A2A 的**本地数据模型思想**，而不是部署 A2A 网络：

```text
goal context
  -> logical task（业务承诺、完成定义、预算）
       -> immutable attempt（一次 worker/一次 Codex 执行）
            -> artifacts（diff、测试、报告、日志尾部）
            -> outcome（accepted / rejected / retryable / needs_decision）
```

这会同时修复上次报告中的 completion 假成功、取消竞态、plan 重写，以及本次发现的“用户不知道一项工作是否真的完成”。

### 3.3 动态 manager 有价值，但不能替代确定性任务治理

[Microsoft Agent Framework 的 Magentic 说明](https://learn.microsoft.com/en-us/agent-framework/workflows/orchestrations/magentic)指出，动态 manager 会根据上下文、任务进度与 agent capability 决定下一位参与者，并维护共享上下文和进度。Magentic-One 技术报告也使用 task ledger 与 progress ledger 来判断是否重规划。

但该官方页面同时明确：此模式适合路径未知的开放问题，原始 Magentic-One 外的表现未经充分验证。对当前项目而言，维护任务、health、release、PR 验收不是开放式头脑风暴，而是有明确文件、contract 和测试的工程工作。

**应借鉴的不是“再放一个 manager LLM”，而是两本账：**

- **任务账本**：目标、所有者、依赖、write set、完成定义、风险、预算、当前状态。
- **进度账本**：已经观察到的事实、失败假设、已验证证据、下一步最小行动、停止条件。

Codex 仍是人类当前使用的交互与判断界面；账本先由确定性脚本维护，Codex 只在不确定性较高时作判断。这样不会平白增加一个管理模型的 token 成本。

### 3.4 worker 不必互相聊天；共享所有权和工件才是先决条件

[Claude Code Agent Teams](https://docs.anthropic.com/en/docs/claude-code/agent-teams)明确把“独立 worker 回报主控”和“teammate 需要直接沟通、质疑、协同”区分开，并建议仅在并行探索、独立模块、竞争假设或跨层协作有真实价值时使用 teams。其子 agent 文档也强调 context isolation、工具限制和更便宜模型的任务路由。

对 Codex Praetor 的映射：

- 里程碑 A 的 edit worker 仍应默认“回报主控”，不应启动 worker 群聊。
- 当多个 worker 只读研究同一不确定问题时，允许并行竞争假设，但结果必须进入同一 evidence ledger。
- 当多个 worker 修改同一文件集时，禁止 direct parallel；先有 owner、base commit、patch artifact 和串行 integration。
- 未来如果真的需要 worker 互相通信，也必须先有 mailbox/task ownership/artifact contract，不能先做聊天面板。

### 3.5 durable workflow 的价值在于事件、重试和恢复，不在于名字

[Temporal Workflow Execution](https://docs.temporal.io/workflow-execution)把 workflow 描述为可恢复的耐久执行：状态由 event history 支撑，失败后从最后记录的事件继续。它解决的是跨进程/跨机器长期运行的严格可靠性问题。

本项目应借鉴四条不变量：

1. 状态转换追加事件，不用最后一次 JSON 覆盖历史。
2. 同一 logical task 每次尝试有唯一 identity 和输入/输出证据。
3. retry policy 明确可重试、不可重试、最大次数、退避和升级目标。
4. cancel 是一个请求/事件，不是直接改写已完成的结果。

不应在当前版本直接引入 Temporal：它会把 Windows 单用户插件扩成服务端产品，增加部署、账号、数据持久化和支持边界。只有出现“跨机器、跨天、并发多用户、必须在 Codex 未运行时持续执行”的硬需求时才重新评估。

### 3.6 协议不是调度器：MCP 与 A2A 的边界必须守住

[MCP 架构文档](https://modelcontextprotocol.io/docs/learn/architecture)明确 MCP 只定义 host/client/server 的上下文交换与工具生命周期，并不规定应用如何管理 LLM 或提供的 context。A2A 则面向独立 agent 系统的 capability discovery、task、artifact、异步通知和安全通信。

因此：

- MCP 继续是 **Codex 看见和调用 Codex Praetor 能力** 的 surface。
- A2A 的 task/artifact/lifecycle 语义可直接借用到本地 schema。
- 不要把 MCP tool call 当成 durable workflow，也不要为了本机 Qoder/CodeBuddy CLI 再起 A2A HTTP server。
- 只有将来外部 provider 真正提供 A2A Agent Card/Task endpoint，且项目需要跨进程/跨组织发现和委派时，才实现可选 A2A adapter。

### 3.7 可观测性必须覆盖“价值”，而不仅是命令

[OpenTelemetry GenAI 语义约定](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-spans/)与当前 GenAI semantic conventions 迁移说明，代表了模型、模型调用、token usage、工具调用、错误和耗时统一归因的方向。LangGraph 的 [multi-agent 文档](https://docs.langchain.com/oss/python/langchain/multi-agent)也明确比较了不同模式的 model calls、tokens、上下文隔离与集中控制开销。

当前项目记录 provider/model/tier/timeout/stdout/stderr，MiMo 有部分 usage parser；Qoder/CodeBuddy 没有实际 usage/cost，`price_note` 是静态文字。下一阶段不能从账户数据库抓费用，而要做到：

```text
planned budget -> actual usage/cost or unknown -> wall time -> accepted output -> logical-task value
```

`unknown` 是合格状态；把拿不到的费用写成 0 才是错误。第一阶段输出项目本地、OTel 兼容的 JSONL span，不立即引入全套遥测后端。

## 四、外部方案取舍：用什么、不用什么、为什么

| 方案/机制 | 可借鉴点 | 本项目决定 | 原因 |
|---|---|---|---|
| A2A Task lifecycle | task/context/artifact、input-required、不可变终态、refinement | **直接采用本地语义** | 解决当前 job/plan 语义不足，不需网络 server |
| OpenTelemetry GenAI | span、usage、模型/工具/错误归因 | **直接采用字段/事件格式** | 可避免再造私有成本记录；先本地 JSONL，后续再 exporter |
| JSON Schema | worker final report 与验证输出 | **直接采用** | Qoder/CodeBuddy 文本输出需要有可验证的语义合同 |
| Temporal | event history、retry、cancel、replay | **借鉴模型，暂不引入** | 当前规模不值得服务化 |
| Google ADK / Semantic Kernel | 顺序、并行、loop、handoff 的模式选择 | **借鉴模式选择表** | 不需要其模型 runtime；部分 API 仍实验性 |
| Magentic-One / Agent Framework | task/progress ledger、重规划 | **借鉴双账本** | 不使用动态 manager 作为默认，避免 token 放大 |
| Claude Code teams | ownership、隔离、何时允许直接通信 | **借鉴协作准入条件** | Claude runtime/账户不是当前外部 worker 方案 |
| OpenAI Agents SDK | manager-as-tools、handoff、trace 的应用编程模式 | **暂不接入运行时** | 当前产品是 Codex Desktop plugin，不是 Responses API 应用；官方全文本机 403，后续有 API 产品线时重新评估 |
| AutoGen / CrewAI / MetaGPT | 通用角色协作、SOP、群聊 | **不引入** | 另起总控与 agent runtime，会重复 Codex、增加 token/权限/支持面 |
| A2A HTTP server | 跨系统 agent discovery/委托 | **延后** | 现有 provider 为本机 CLI，尚无 A2A endpoint；先做 adapter-ready local contract |
| Temporal server | 跨机器耐久执行 | **延后** | 没有跨机器/多用户硬需求，复杂度不成比例 |

这张表的关键结论是：**要复用成熟方案的“合同和机制”，不要复用它们的“整套运行时”。** 目前直接装一个多 agent 框架不会减少自造轮子，只会把轮子叠两层。

## 五、回看本项目：之前问题现在应归入哪些方向

以下把 0.3.0 后路线图、代际治理报告、真实派工技术复盘、任务治理复盘和本次外部调研合并。每一组都必须进入同一个大 PR 的内部工作流，而不是继续各修各的。

### 5.1 工作定义与协作模式缺口

| 已观察问题 | 根因 | 外部机制映射 | 彻底修复 |
|---|---|---|---|
| 维护包过宽、需要平台语义却禁止 worker 外网研究 | 没有 task shaping / evidence gate | ADK 的确定性流程、Magentic task ledger | 先形成 task class、未知事实、write set、完成定义，再决定先研究、顺序或并行 |
| repair、uninstall、health 任务无依赖 | task 只是 job 标签 | A2A immutable task/refinement | logical task 下保存 attempts；用依赖和必测场景控制下一步 |
| 共享文件被多次不同 worker 改 | 没有 work ownership/patch integration contract | Claude teams ownership，Git worktree | write set 互斥、base commit 固定、只有 accepted patch 可成为后继基线 |
| worker 没有真正需要互相沟通的理由却产生协调开销 | 默认把“多 agent”当成目标 | Claude teams / SK pattern selection | 默认 sequential；并行必须证明独立性；group chat 默认禁止 |

### 5.2 任务状态、验收与恢复缺口

| 已观察问题 | 根因 | 外部机制映射 | 彻底修复 |
|---|---|---|---|
| `Max turns`、tool denied 被写为 completed | process exit 替代业务完成 | A2A terminal states、Temporal events | `process_state`、`worker_report_state`、`artifact_state`、`supervisor_verdict` 分离 |
| cancel 与 watcher 可互相覆盖 completion | JSON 覆盖写、无单写者/event log | Temporal cancel/event history | cancel request 追加事件；watcher 单写终态；终态不可改写 |
| plan `.tmp` 并发冲突 | 无版本/CAS/锁 | durable workflow single owner | mutex + 唯一 tmp + revision/CAS + append-only events |
| dynamic health worker 产物被 cancelled 语义遮蔽 | attempt 与结果没有独立保存 | A2A artifact/task immutability | 每次 attempt 保存 artifact manifest；取消后仍可被 Codex 验收或拒绝 |

### 5.3 provider 选择、失败学习与成本缺口

| 已观察问题 | 根因 | 外部机制映射 | 彻底修复 |
|---|---|---|---|
| canary 通过却不适合复杂 edit 包 | readiness 只验证资格 | Magentic capability/progress、LangGraph pattern cost | 分成 hard eligibility 与 soft suitability；记录 task class 成功率/失败模式/样本量 |
| CodeBuddy 失败后靠人临场改派 Qoder | 没有 failure policy | Temporal retry/handoff | 明确 retryable/non-retryable；输出缩包、改派、先研究、Codex 接手、请求用户决策 |
| Qoder repair 学到 XML 约束但下一次不会复用 | 无 outcome ledger | Magentic progress ledger | 可过期的 outcome records，附证据路径与适用 task class |
| 看不到实际 token/费用和“是否值得继续” | 只有静态 price_note | OTel GenAI trace | planned/actual-or-unknown budget、attempt wall time、accepted-value、stop-loss |

### 5.4 用户体验、发布与运行态缺口

| 已观察问题 | 根因 | 外部机制映射 | 彻底修复 |
|---|---|---|---|
| 多阶段工作全 blocking、notify 关闭 | 把 worker 等待等同于用户应等待 | A2A input-required、LangGraph interrupt | 背景执行 + 单次语义通知；只在预算/风险/授权/连续失败时 `needs_decision` |
| 用户不知道“已完成什么、卡在哪里、是否继续” | plan 是收据不是决策面 | task/progress ledger | 固定中文摘要：交付、阻断、成本、风险、下一动作、默认建议 |
| maintenance、动态 health、inventory 尚未完成 | 产品真值没有接入任务治理 | release receipt + logical task | 作为大 PR 的 product truth workstreams，在 governance core 通过后依次执行 |
| generation 回收、fresh context、stable activation | 发布状态与开发状态容易混淆 | 既有 generation lifecycle | 保留现有 stage/activate/retirement 边界，纳入最终 logical release task，不重新发明 |
| 供应链、SBOM、attestation、GitHub 设置未完成 | 只做本地 patch，没有完整交付层 | 既有路线图/发布 gate | 作为 release workstream；管理员动作单列，不伪装成代码自动完成 |

## 六、目标架构：最小任务治理内核

下一版本不引入新的总控 agent。Codex 继续承担规划、判断、整合与最终验收；外部 CLI worker 执行边界清楚的 attempt；确定性本地控制面维护状态、工件和策略。

```text
用户目标
  -> Codex 任务画像与模式选择
  -> Logical Task / Task Ledger
  -> Selection Decision（资格、适配、预算、止损）
  -> Attempt（worktree + worker contract + base commit）
  -> Event / Artifact Ledger
  -> 自动证据检查
  -> Codex Verdict
  -> Outcome Ledger 与下一步建议
  -> 必要时 Needs Decision
```

### 6.1 三类状态，不能再混用

```text
执行状态：draft / queued / running / cancel_requested / process_exited
证据状态：report_valid / artifact_valid / tests_passed / evidence_missing
治理状态：awaiting_supervisor / accepted / rejected / retryable / blocked / needs_decision
```

`completed` 只能表示 logical task 的 accepted outcome，不能表示进程 exit 0。attempt 终态不可改写；后续修复引用前一 attempt，成为新 attempt。

### 6.2 两本账和三种合同

| 对象 | 内容 |
|---|---|
| Task Ledger | goal/context、logical task、owner、模式、依赖、write set、完成定义、风险、预算、stop-loss |
| Progress Ledger | 已验证事实、失败假设、decision、artifact、下一最小动作、用户输入需求 |
| Worker Contract | 允许路径、禁止路径、base commit、期望工件、测试、machine-readable final report、超时/最大轮次 |
| Selection Record | 资格结果、候选 provider、选择理由、预计 usage/cost、替代方案、升级条件 |
| Outcome Record | 任务类型、attempt、provider/model、实际 usage 或 unknown、wall time、verdict、失败类、可复用规则、证据路径、过期时间 |

### 6.3 只在必要时打扰用户

默认不要求用户盯 worker，也不创建持续自动化。用户只会收到以下四类结构化请求：

1. 预算超出默认或需要使用更贵模型。
2. 范围将改变、需要不可逆动作或真实发布审批。
3. 外部登录/验证码/账户授权已被事实证明需要人工操作。
4. 达到 retry/时间止损，必须在缩包、改派、Codex 接手、停止之间选择。

其他失败由 Codex Praetor 自动按既定 policy 收敛为下一建议，不向用户倾倒日志。

## 七、下一阶段整体路线图

### 阶段 0：架构冻结与验收设计

**性质：实现前的短设计门，不单独发 PR，不改 stable。**

输出一份 ADR 和机器可读 schema 草案，确认：

- task/attempt/event/artifact/outcome 的字段与状态机；A2A 本地语义映射。
- 顺序/并行/handoff/loop 的准入表，默认策略和禁止条件。
- Qoder、CodeBuddy、MiMo 的 structured report 能力探针；若无原生 JSON，则定义 wrapper-side artifact proof，不伪造 JSON。
- usage/cost 字段来源与 `unknown` 规则；禁止读取 provider 账户数据库。
- MCP 既有工具哪些只扩展输出、哪些需要新工具；据此确认 `0.4.0-alpha` 合同变化。

**通过条件**：不存在“先选框架再找需求”；每个字段都能对应里程碑 A 的真实收据问题。

### 阶段 1：`0.4.0-alpha` 任务治理与产品真值大 PR

这是下一次实现的主 PR，内部按下面顺序推进，但对用户是一个一致的大交付。

#### 工作流 A：治理内核

1. 将 `plan/v1` 演进为 logical task + immutable attempt + event projection；兼容读旧 plan，旧作业不丢失。
2. 用任务级锁、唯一临时文件、revision/CAS 消除 plan 并发覆盖。
3. 重写 watcher/cancel 的状态机：exit code、provider report、scope/diff/test 工件、supervisor verdict 分离；cancel request 不覆盖已有工件。
4. 给任务合同加入 base commit、write set、模式、完成定义、预算、stop-loss 和 structured final report 要求。
5. 新增针对本次真实文本的回归测试：`Max turns`、tool denied、空输出、无 diff、已退出后取消、并发计划写入、重叠 write set。

#### 工作流 B：选择、学习与可观测性

1. 拆分 provider readiness（硬资格）与 selection（软适配）。
2. 写入 selection record：理由、候选、预算、升级条件；不是只保存 tier。
3. 增加 outcome ledger，初期采用透明规则和有过期时间的本地统计，不做黑箱学习。
4. 生成本地 OTel-compatible JSONL span：模型、task class、usage/cost 或 unknown、wall time、工具、错误、verdict。
5. 新增 compact governance summary 与 `needs_decision`，让 MCP/Skill 能用中文显示可行动结论。

#### 工作流 C：用新内核完成原里程碑 A 的产品真值

1. **maintenance adapter**：重做 `163ce53` 的独立验收。一个 canonical definition 驱动 ScheduledTasks/schtasks；真实隔离 profile round-trip；安装/查询/卸载/health 对称；不再只靠 mock。
2. **dynamic health/readiness**：重新实现，不搬运 `45852888`；加载合同后再计算 generation，缺失 CLI/hash/expiry/任务漂移一律 fail closed。
3. **runtime inventory**：区分 active、retired、clean+merged、dirty/unmerged、审计保留和 test scratch；默认只读，自动删除仅接受被明确批准的安全候选。
4. **文档与用户状态**：重写过时 roadmap、安装/排错/卸载/release gate，展示任务治理状态而非仅脚本状态。

#### 工作流 D：供应链与交付

1. workflow action full SHA pin、Dependabot、最小 token permissions。
2. release build、SBOM、attestation、远端下载验收脚本。
3. PR 前完成医生检查、产品验证、MCP、PowerShell 5.1/7、Node、isolated profile、determinism、public scan、CI。
4. 管理员动作（ruleset、branch auto-delete、private vulnerability reporting、immutable releases）在 PR 描述中独立列为合并后门禁。

**阶段 1 完成标准**：一条短自然语言“拆分并推进”能够产生可解释计划；本次所有真实失败能落到正确状态；用户无需读日志即可知道结果；原里程碑 A 的产品真值和供应链项均通过 PR 前门禁。

### 阶段 2：合并后交付

按现有发布规则：更新 main、构建新 tag/Release zip、stage/hash verify、需要时做一次 fresh context、provider readiness、activate、远端普通用户路径复验、退休代际回收状态记录。

此阶段不得把“代码已合并”说成“产品已交付”。若合同变化使 MCP tool 输出改变，fresh context 是每代一次验收，不是每次文件编辑一次。

### 阶段 3：条件满足后才考虑的扩展

| 候选 | 只有满足以下条件才启动 |
|---|---|
| A2A adapter/server | 外部 agent 真实提供 Agent Card/Task endpoint，且需要跨机器或跨组织委派 |
| Temporal | 需要跨机器、跨天、多用户或 Codex 关闭后仍必须继续的高价值流程 |
| 动态 manager/群聊 | 有开放式研究任务，且已证明确有 worker 间互相质询的收益，预算可接受 |
| 独立遥测后端 | 本地 JSONL 已稳定，确实需要跨项目查询、告警或团队可视化 |
| 新 provider | 现有 provider 的 task-class outcome 显示覆盖不足，而非仅为“多一个选择” |

## 八、明确不做清单

下一阶段为防止重新膨胀，明确不做：

- 不把 AutoGen、LangGraph、CrewAI、Semantic Kernel、ADK、Agents SDK 任一个作为第二个主控运行时。
- 不做 worker 群聊、swarm、自动多轮辩论作为默认路径。
- 不部署 Temporal、Redis、队列服务、数据库或 A2A HTTP server。
- 不读取 Qoder/CodeBuddy/MiMo 账号数据库、余额、token、cookie 或截图来算成本。
- 不自动删除 dirty worktree、未交付 artifact 或旧 generation。
- 不为了适配旧对话而原地覆盖 plugin/Skill/cache generation。
- 不把 GitHub 管理员设置伪装成 PR 代码已经完成。

## 九、风险与决策点

| 风险 | 处理方式 |
|---|---|
| 大 PR 范围过宽 | 用阶段 0 schema/ADR 和阶段 1 内部 workstream gate 控制；不再拆成无关散点 PR |
| worker 无法输出 JSON report | wrapper 以 diff/test/artifact truth 为主，文本只作补充；保留 provider adapter 层 |
| 实际 usage 不可得 | 显式 unknown，仍用 attempt 次数和 wall time 做止损；不捏造费用 |
| 治理内核改变 MCP surface | 直接升为 `0.4.0-alpha`，做 fresh-context proof |
| 现有工作树/作业含未交付成果 | inventory 先报告后处置；本轮不删除 |
| 用了外部框架反而复杂 | 阶段 0 先锁定“标准语义 + 本地内核”的取舍，任何完整框架引入必须另开 ADR 并满足阶段 3 条件 |
