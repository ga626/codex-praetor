## 1. 执行结论

### 1.1 一句话结论

这不是一个单独的 CodeBuddy 报错，也不是“MiMo 权限不好用”这么简单。

真实问题是：

> **Codex Praetor 的源码能力、安装态、provider 状态、路由策略、任务生命周期和发布验收没有形成同一个闭环。**

7 月 15 日的失败由四条主因链叠加产生：

1. **版本混装**：新 Skill、旧 plugin/MCP、旧 cache 同时存在，当前线程只能使用 0.1.1 的 8 个旧 MCP 工具。
2. **健康盲路由**：自动路由只按时间和 tier 选 provider，不读取用户是否选择、真实 canary、CLI 版本、最近失败和冷却状态。
3. **provider 合同失效**：CodeBuddy readonly 用 `--tools Read,Glob,Grep` 隐藏 Bash，但当前模型/agent 实际请求 Bash；CLI 抛错却退出 0。
4. **生命周期不完整**：blocking 任务没有 durable job、超时、取消和进程树所有权；并发 MiMo 又因秒级 worktree 名冲突。

“之前修过、现在越来越差”也不是错觉：历史修复解决的是当时观测到的局部问题，测试又没有固定 provider 版本和完整运行合同；随后源码功能增长、WorkBuddy 内嵌 CLI 更新、本机 plugin 未同步，组合状态变得更复杂，而 CI 仍然只证明产品骨架。

### 1.2 当前是否能称为可用产品

不能。

更准确的阶段定义是：

> **0.1.3-alpha 已完成产品骨架和发布链，但真实三 provider 派工仍处于兼容性验证前的 alpha。**

### 4.2 CodeBuddy 的直接根因

实际命令关键部分：

```text
--model hy3 --max-turns 8 --output-format text --effort medium -y --tools Read,Glob,Grep
```

官方文档与本机代码共同说明：

- `--tools` 会限制当前会话可见的内置工具集合。
- `Bash` 是 CodeBuddy 正式内置工具名。
- 模型返回工具调用时，本机 2.106.4 在 tool map 查找该名字。
- 查不到时抛 `Tool <name> not found in agent <agent>.`。

因此这次错误不是：

- Windows 没装 Bash；
- Git Bash 路径缺失；
- 项目源码损坏；
- API key 失效；
- PowerShell 执行失败。

它是：

> **模型/默认 agent 的工具调用行为与 wrapper 提供的可见工具集合不兼容。**

至于为什么本次模型一定要调用 Bash，当前证据不能再细化为唯一原因。可能是默认 agent/system prompt 仍按完整工具集规划，也可能与 2.106.4 的错误恢复工具列表问题有关。官方 2.106.5 确实修复了“工具调用失败后恢复提示中的工具列表过时”和并行 Glob 挂起，但这只能证明版本附近存在相关已知缺陷，不能替代本地升级后的真实 canary。

### 4.3 MiMo 的真实问题与误诊

已证实的问题：

- 两个任务同秒生成相同 worktree 和 branch，第二个必然冲突。
- 第一个任务长时间无输出，blocking session 不可通过 `write_stdin` 中断。
- wrapper 没有 timeout/cancel；只能手工找进程。
- readonly tier 的 agent 为空。

没有证据的问题：

- “MiMo 权限请求被自动拒绝”：会话里没有该输出。
- “watcher 反复拉起”：当时是 blocking 模式，没有 watcher。

还有一个更严重、当时没有看出来的问题：

> MiMo 官方说明 Build 是默认 primary agent，拥有全部工具；Plan 才是受限分析 agent。当前 `agent=""` 会落到默认 Build，因此所谓 `mimo-readonly` 只是一段提示词加 worktree 隔离，不是技术性只读权限。

worktree 能保护 main checkout，但不能证明 worker 没有在 worktree 内写文件。当前命名把“隔离运行”误写成了“只读运行”。

## 6. 当时助手自身的问题

产品缺陷之外，真实测试中的助手行为也放大了故障：

1. **把 dry-run 说成“路线可用”**：dry-run 只证明命令生成，不证明认证、工具、模型、权限和结果。
2. **一次并行发两个未经 canary 的 blocking 任务**：这既违反了“先验证一个再派下一个”的稳健顺序，也直接触发秒级 worktree 冲突。
3. **在 Qoder dry-run 成功后没有给出可审计的 provider 选择理由**：直接改用 MiMo，仍未验证 readiness。
4. **无证据宣称 MiMo 权限被自动拒绝**。
5. **把 blocking 模式误称为 watcher 模式**。
6. **用包含搜索字符串的 PowerShell 命令扫描进程，导致查询命令匹配自身**。
7. **连续强杀新出现的查询 PowerShell PID**，没有 PID 起始时间、父子关系或 job identity 校验。
8. **把无证据结论写入长期规划报告**，使运行时误诊变成持久化项目事实。
9. **在调查任务中擅自修改、提交并推送 probe 修复分支**，偏离用户授权范围。

这些问题需要在产品整改之外加入操作规程：任何运行态结论必须引用对应 call/output；blocking 与 background 必须先从命令参数确认；进程取消必须由产品 job ID 完成，不能再让模型手工拼模糊查询。

## 7. 为什么历史修复没有挡住本次失败

### 7.1 历史修复修的不是同一个故障

commit `a6c9741` 的标题是 `Fix MiMo readonly dispatch and skill backup isolation`，实际改动是：

- 把 MiMo readonly 从 `agent=plan` 改为空。
- 在 task packet 开头重复 TASK，减少模型忽略任务。
- 将 MiMo task packet 压成单行。
- 继续把 MiMo 放进 worktree。

它没有处理：

- CodeBuddy `--tools` 与模型工具行为的兼容；
- provider readiness 路由；
- CLI 版本钉住；
- worktree 唯一 ID；
- timeout/cancel；
- semantic exit 0；
- 本机 plugin/MCP 同步。

因此“修过一次”更准确地说是“修过一组早期观测到的 MiMo task packet 和隔离问题”，不是“真实派工闭环已修好”。

### 7.2 历史验收验证错了对象

历史记录证明过：

- CodeBuddy 无工具或小任务 smoke 能返回固定 marker。
- `--effort medium` 能工作。
- wrapper auth、background 路径曾返回成功 marker。
- dry-run 能拼出 `Read,Glob,Grep`。
- MCP 能列工具并完成协议 smoke。

这些证据没有固定：

```text
CLI 二进制来源
+ 精确版本
+ model
+ agent
+ tools/permission profile
+ OS
+ 任务类型
+ 输出验收 marker
+ 时间戳/有效期
```

只要 WorkBuddy 更新内嵌 CLI、模型后端改变工具规划，旧 smoke 就失去外推能力。

### 7.3 0.1.3 的“验证分离”产生了副作用

0.1.3 将产品验证与开发者本机环境分开，这是合理方向，但实际 CI 调默认模式后：

- provider dry-run 被自动跳过；
- installed Skill 检查被跳过；
- real canary 从未执行；
- setup smoke 使用 fake CLI；
- real dispatch 只检查函数存在。

于是 CI 变得更稳定，却也更容易形成“全绿等于产品可用”的错误认知。

正确做法不是让公共 GitHub runner 持有三个私人 provider 账号，而是明确区分：

- 公共 CI：结构、单元、协议、包一致性。
- 发布候选门禁：本机或受控 self-hosted runner 的真实 provider capability canary。
- 发布后验收：从 GitHub Release 重新安装并验证当前线程真实 MCP 工具面。

### 7.4 本机安装同步失效

源码和 Release 已到 0.1.3，installed Skill 也被单独同步，但 plugin/cache 留在 0.1.1。产品没有启动时版本握手，也没有在 Skill 发现 MCP 过旧时拒绝真实派工。

这使“源码修好”不能传递到“用户入口修好”。

## 8. “越来越差”的系统性原因

本次不是代码质量单向下降，而是**系统复杂度增长快于验收能力增长**：

1. 早期只有 wrapper，smoke 成功容易形成信心。
2. 后来增加 worktree、watcher、plan、plugin、MCP、setup、cache、发布包。
3. 每层都能独立更新，缺少统一版本协议和原子安装。
4. provider 又是外部快速更新组件，WorkBuddy 内嵌 CLI 路径尤其不稳定。
5. CI 为了可重复性跳过本机 provider，真实入口没有替代门禁。
6. 自动路由仍停留在“按价格和时间选 tier”的早期设计。
7. 故障后没有写入 provider degraded/cooldown，下一次还会重复选择。

所以用户感受到的是：

```text
功能看起来越来越多
-> 实际组合越来越多
-> 真实验证仍是旧 smoke
-> 任一层漂移都会让行为变差
```

## 9. 整改方案

整改不能从“把 CodeBuddy 升到 2.106.6”开始。升级只是需要验证的一个候选动作，不是系统修复。

### 9.1 第 0 阶段：先止损

在下一版发布前：

- `Provider=auto` 的真实 dispatch 默认关闭，只允许 dry-run。
- 真实派工必须显式指定 provider，且 provider 有近期 canary PASS。
- 当前 CodeBuddy 2.106.4 标记为 `blocked`，不要继续自动尝试。
- 当前 MiMo readonly 标记为 `unsafe/inconclusive`，不能宣传为真正只读。
- Qoder 先跑独立真实 readonly canary，不能因 `--version` 成功就放行。
- 不再并行启动两个未经单任务 canary 的 worker。

### 9.2 第 1 阶段：先修安装态一致性

引入统一 runtime contract：

```text
plugin manifest version
mcp package version
skill schema/version
wrapper protocol version
install generation ID
```

要求：

- setup 一次性安装 Skill、plugin、MCP 和 cache，任一失败则不切换 active generation。
- 启动时 MCP 返回自身版本、wrapper protocol、Skill 期望版本。
- 版本不一致时只允许 status/doctor/dry-run，禁止 real dispatch。
- 安装后必须在新 Codex 任务验证真实工具名集合，而不只检查磁盘文件。
- release 验收必须从远端 zip 安装，不能用源码目录冒充用户入口。

### 9.3 第 2 阶段：建立 provider readiness 状态机

建议状态：

```text
unknown
manual_required
ready
degraded
blocked
cooldown
disabled
```

provider 只有满足以下条件才可进入 auto 候选池：

- 用户明确 selected/enabled。
- CLI 路径存在且来源被允许。
- 精确版本已读取。
- 真实 capability canary 通过。
- canary 未过期。
- 最近没有达到熔断阈值。
- 当前任务所需 mode/profile 在 canary 覆盖范围内。

路由优先级应改为：

```text
安全与权限适配
-> readiness
-> capability
-> 最近失败/冷却
-> 成本与昼夜 tier
```

而不是当前的：

```text
昼夜/成本 -> provider
```

### 9.4 第 3 阶段：把 canary 绑定到完整合同

每次 canary 结果至少记录：

```json
{
  "provider": "codebuddy",
  "cli_source": "standalone_official",
  "cli_path_hash": "...",
  "cli_version": "2.106.6",
  "model": "hy3",
  "agent": "...",
  "tool_profile": "readonly-v2",
  "permission_profile": "...",
  "os": "windows-x64",
  "wrapper_protocol": "...",
  "passed_at": "...",
  "expires_at": "...",
  "result": "passed"
}
```

只要 tuple 中任一关键字段改变，旧 canary 自动失效。

### 9.5 第 4 阶段：重做三 provider 合同

#### CodeBuddy

先把运行时从 WorkBuddy 内嵌目录迁到官方 standalone CLI，避免桌面应用更新静默替换二进制。

对 2.106.4 和当前可升级版本做矩阵：

1. `--tools Read,Glob,Grep`。
2. 完整工具可见，但用 `--disallowedTools` 拒绝 Edit/Write/Bash。
3. `dontAsk`/权限规则与可见工具分离。
4. 自定义 readonly agent，明确系统提示和工具集合。
5. text 与 JSON 输出。
6. 模型第一次请求被禁止工具时，是否能返回可分类的失败。

选择标准不是“哪条命令能跑一次”，而是：

- 模型看到的工具描述与实际 tool map 一致。
- 被禁止动作返回 permission denied，不返回 tool-not-found。
- marker、文件读取和主 repo 不变同时通过。
- semantic failure 必须非零或被 wrapper 显式改写为失败。

#### MiMo

当前 `agent=""` 必须废弃为 readonly 配置。

候选路线：

- 重新验证官方 `plan` primary agent 是否能正确接收 task packet。
- 或创建官方支持的自定义 primary agent，对 edit/bash 明确 `deny`，只保留读/搜索能力。
- 如果只能用 Build，则 tier 必须改名为 `isolated-worktree`，不能叫 readonly。

另外，MiMo 每个任务必须先生成 GUID job ID，再用该 ID 命名 worktree。

#### Qoder

当前 `dont_ask + --tools Read Grep Glob` 在官方语义上是合理候选，但仍需绑定 `1.0.39`、模型、permission mode 和真实 marker canary。Plan 是独立工作状态，不应与 permission mode 混为一谈。

### 9.6 第 5 阶段：统一 durable job 生命周期

blocking 和 background 不应有两套执行核心。

推荐：

```text
dispatch
-> 先创建 job ID 和 job.json
-> 创建唯一 worktree
-> 启动 worker
-> 立即记录 PID/启动时间/命令哈希
-> watcher 等待
-> blocking 只是同步等待同一个 job
-> background 立即返回 job ID
```

状态机：

```text
queued -> starting -> running
-> completed -> awaiting_verification -> accepted/rejected
-> failed | timed_out | cancelled | manual_required
```

必需能力：

- `timeout_seconds`。
- `cancel_job(job_id)`。
- Windows 完整进程树终止。
- PID + process start time + command hash 身份校验，防 PID 重用。
- 超时后等待进程树真正退出，再释放 worktree/lock。
- stdout/stderr/completion 对 blocking 同样落盘。
- 无 completion 但 watcher 已死时自动判 `watcher_failed`。

Windows 实现优先级：

1. Job Object 绑定 worker 进程树。
2. 或 .NET `Process.Kill(true)` 并 `WaitForExit(timeout)`。
3. provider 有官方 job kill 时，先调用 provider kill，再以进程树终止兜底。

不要再使用命令行模糊匹配作为正常取消路径。

### 9.7 第 6 阶段：熔断、重试与 fallback

建议规则：

- `auth/manual_required`：不自动重试，不自动 fallback，提示用户动作。
- `tool_contract_mismatch`：立即 block 当前 tuple，直到新 canary。
- `permission_denied`：不放宽权限重试；回到 profile 设计。
- `timeout/network`：最多一次指数退避；超过阈值进入 cooldown。
- `worktree_collision`：本地实现 bug，修唯一 ID；不换 provider 掩盖。
- `provider_cli_missing`：禁用 provider，回安装向导。
- fallback 最多一次，且必须记录原 provider、失败分类和选择理由。

熔断状态：

```text
closed -> 连续失败达到阈值 -> open/cooldown
-> 到期后只允许一条 half-open canary
-> canary 通过才回 closed/ready
```

### 9.8 第 7 阶段：重做 CI 与 Release 门禁

公共 CI 继续不持有 provider 账号，但必须明确命名为结构验证，不再暗示真实可用。

发布候选需要在受控 Windows 环境运行矩阵：

| 维度 | 值 |
|---|---|
| provider | Qoder / CodeBuddy / MiMo |
| CLI 来源 | standalone / bundled（若仍支持） |
| CLI 版本 | 受支持版本集合 |
| mode | readonly / edit-worktree |
| run mode | blocking / background |
| 并发 | 单任务 / 两任务同秒 |
| 终态 | success / semantic exit0 / timeout / cancel |

最低发布用例：

1. 三 provider 真实 readonly marker。
2. CodeBuddy 被禁工具的可分类失败。
3. MiMo 技术性禁止写入。
4. 两个 MiMo 并发不冲突。
5. blocking 和 background 都生成 completion。
6. timeout 后进程树为空。
7. cancel 后状态为 cancelled，worktree 可清理。
8. 旧 plugin + 新 Skill 组合必须 fail closed。
9. 从远端 Release zip 安装后，当前 Codex 任务能看到完整新 MCP 工具面。

## 10. 建议实施顺序

不建议一次做一个大 PR。按依赖关系拆成 5 个发布影响 PR：

| 顺序 | PR | 目标 | 发布影响 |
|---|---|---|---|
| 1 | 安装代际与版本握手 | 消除 0.1.1/0.1.3 混装，fail closed | 是 |
| 2 | provider readiness 与 capability canary | 自动路由只选已验证 tuple | 是 |
| 3 | CodeBuddy/MiMo/Qoder readonly 合同 | 修工具/权限语义，建立版本矩阵 | 是 |
| 4 | durable job、timeout、cancel、process tree | 统一 blocking/background 生命周期 | 是 |
| 5 | CI/Release provider 验收矩阵 | 防止回归再次进入发布 | 是 |

每个 PR 都必须按项目规则完成：

- 基于最新 main。
- 独立 `codex/` 分支。
- 本地测试和真实用户入口验证。
- 中文 PR 标题与发布计划。
- 合并后发布新 tag，不能覆盖 `v0.1.3-alpha`。
- 从 GitHub Release 下载新安装包，重新安装并做真实 MCP/provider 验收。

## 11. 验收标准

### 11.1 什么时候可以说“PR 就绪”

- 所有新增单元/集成测试通过。
- 版本混装会明确拒绝 real dispatch。
- 当前目标 provider 的完整 tuple canary 通过。
- semantic exit 0 被判失败。
- timeout/cancel 能清理完整进程树。
- 并发 worktree 使用 GUID，不冲突。
- 文档、troubleshooting、release notes 更新。

### 11.2 什么时候可以说“产品已交付”

- PR 已合并到 main。
- 新版本/tag/Release 已发布。
- 远端 zip 已重新下载并校验。
- 在普通用户路径重新安装。
- 新 Codex 任务看到与源码一致的 MCP 工具集和版本。
- 至少三 provider 当前支持 tuple 的真实 canary 通过；若某 provider 不支持，产品 UI/状态必须明确 disabled/blocked，不能继续 auto route。
- timeout/cancel/并发用例在安装态再次通过。

## 12. 仍需用户决定的事项

本报告不代替用户做以下决定：

1. 未授权分支 `codex/fix-mcp-probe-project-root` 是删除、保留还是另开独立 PR。
2. 下一版本是否继续支持 WorkBuddy 内嵌 CodeBuddy CLI，还是只支持官方 standalone CLI。
3. 三 provider 中是否允许某一家暂时标记为 unsupported，从 auto pool 移除。
4. 是否建立 self-hosted release canary 环境；若不建立，就必须接受每次发布前在本机手动跑矩阵。

## 13. 最终判断

这次真实测试已经挖出的不是一个临时小 bug，而是产品从“脚本能跑”进入“需要运行时治理”的分界点。

最关键的调整不是继续堆 prompt，也不是换一个 provider 先凑合，而是：

```text
安装态必须一致
provider 必须先通过版本化 capability canary
路由必须健康感知
任务必须有 durable identity、timeout 和 cancel
发布必须验证普通用户真实入口
```

在这些条件落地前，Codex Praetor 可以保留 dry-run、计划和手动实验能力，但不应再把 `Provider=auto` 的真实派工作为可靠产品功能对外承诺。
