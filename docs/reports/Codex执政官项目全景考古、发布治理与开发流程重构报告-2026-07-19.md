## 3. 项目演进考古

### 3.1 版本路线不是一条产品路线，而是多条局部修复路线叠加

| 代际 | 主要增加内容 | 没有被同时解决的结构问题 |
|---|---|---|
| `0.1.x` | Skill、wrapper、MCP 基础骨架、Windows setup、基础 provider tier | 真实 provider、安装面一致性、取消/超时、用户交付 |
| `0.2.0-alpha` | runtime contract、generation、receipt、stage/activate 雏形 | host 刷新、实际 provider 兼容、发布前真实 canary |
| `0.3.0-alpha` | 产品化文档、发布包、durable job、计划和验收工具 | 维护任务真值、动态 readiness、供应链、任务价值闭环 |
| `0.4.0-alpha` | task governance、更多 MCP 工具、状态/计划语义 | 运行态与产品真值仍分散，测试仍偏结构和模拟 |
| `0.4.1-alpha` | Desktop runtime identity、fresh-context proof、旧 generation 处理 | 仍需要人工刷新 host，provider readiness 与发布控制面未统一 |
| `0.5.0-alpha` | 统一治理、dynamic health、maintenance adapter、readiness helper、release closeout | 新 helper 未做真实组合测试；公开发布仍可先于完整交付证明 |

### 3.2 历史报告反复出现的模式

历史报告总体判断并非无效，但它们经常把“脚本存在”“隔离 smoke 通过”“结构性 gate 通过”写成“产品能力已完成”。随后下一轮又在别的层补一个局部控制器，导致：

1. 同一事实被多个脚本各自解释。
2. 源码、plugin、Skill、cache 需要手工同步。
3. 测试验证的是模拟数据，不是真实调用路径。
4. Release、stage、activate、health 的状态词互相重叠。
5. 新报告继续引用旧报告的“已完成”表格，形成进度幻觉。

### 3.3 `0.5.0-alpha` 的真实新增风险

`0.5.0-alpha` 在 `scripts/dispatch/invoke-codex-praetor.ps1` 中新增点源 `resolve-codex-praetor-readiness.ps1`。被点源文件仍带有顶层 `param(...)`，PowerShell 会在调用者作用域执行它并覆盖 `$Provider`。因此：

- 静态 PowerShell parse 通过。
- MCP source test 通过。
- packaged protocol smoke 通过。
- 模拟 closeout 通过。
- 真实 provider capability canary 失败。

这不是单纯“漏跑一个命令”，而是组合测试缺失：没有验证“被点源的 helper 不污染调用者变量”，也没有强制真实 canary 进入发布前门禁。

- 源码、`skill/`、`plugin/` 三套副本依然存在漂移风险。

### 4.2 运行态和交付层：红

当前 stable active 仍是 `0.4.1-alpha`。`0.5.0-alpha` 只完成了远端发布和 stage，没有完成：

- 当前 Desktop host 刷新。
- 目标 generation 的 native fresh-context proof。
- 目标 generation 的 provider readiness。
- activate。
- stable health=ready。
- 普通用户下载、安装、打开、调用的完整复验。

因此“Release 页面存在”与“产品已交付”必须彻底分开。

### 4.3 真实派工层：红偏黄

当前 wrapper 有较多门禁，但真实 provider 仍是外部漂移面：

- provider CLI 来源和版本可能随第三方应用更新改变。
- 真实 canary 与路由消费没有形成强制闭环。
- `cli_version`、工具集合、模型、权限 profile、task kind、OS 组合没有被统一固定为 capability matrix。
- 失败分类、冷却、重试预算和半开探测仍不完整。

### 4.4 任务治理层：黄偏红

当前已经有 durable job、plan、watcher、completion、verify-task 等组件，但它们更多记录“进程发生了什么”，仍不能稳定表达：

- 用户价值目标是什么。
- 这个 attempt 产生了什么可验收产物。
- 失败是否可重试，为什么重试。
- 已经花了多少时间和未知成本。
- 下一步是继续、缩小范围、换 provider、人工决定还是停止。

这与里程碑 A 复盘中的结论一致：系统更像“受控外部 CLI 调用器 + 人工监督习惯”，还不是完整的治理闭环。

## 6. 为什么开发流程会反复出问题

### 6.1 根因一：把“版本”当成事实源，而不是把“收据”当成事实源

现在有很多版本字符串，但没有一个唯一的发布状态对象回答：

```text
这份源码是什么？
这份包是什么？
本机安装的是什么？
Desktop host 运行的是什么？
provider 验证的是哪个 generation？
用户今天实际能用的是什么？
```

`active.json`、readiness 文件、MCP runtime_info 和 GitHub Release 分别回答不同问题，系统却经常把其中一个的“通过”传播成整体“ready”。

### 6.2 根因二：发布脚本和交付门禁是两套流程

发布脚本验证远端包和 hash；activate 脚本验证 proof/readiness；health 验证 active receipt；Desktop host 又有自己的缓存生命周期。它们没有由同一个 controller 管理状态转换，所以可以合法地出现“已公开、已 stage、未 active、host 旧、provider 未 ready”。

### 6.3 根因三：测试过度偏向可重复，牺牲了真实路径

公共 CI 不应该依赖个人 provider 账号，这是对的；但当前替代方案只是 fake readiness、fake provider、结构 smoke 和 protocol smoke。结果是：

- CI 绿不等于 provider 可用。
- closeout smoke 绿不等于当前 Desktop 可加载。
- parser 通过不等于 PowerShell 组合语义正确。
- Release zip hash 正确不等于用户能在 Codex 中调用到新工具。

缺少的是分层而不是简单“把所有账号塞进 CI”。

### 6.4 根因四：把不可自动化的 host 刷新边界写成了人工说明，没有写进状态机

MCP `tools/listChanged` 只表示服务器可以通知工具列表变化；它不保证宿主会重建已经存在对话的工具快照。成熟 IDE 也把 Extension Development Host、Reload Window 和兼容性验证分开。我们的文档知道这一点，但 release controller 没有 `awaiting_host_refresh` 状态，也没有在 publish 前硬失败。

### 6.5 根因五：副本和入口太多，缺少生成关系

`skill/`、`plugin/skills/`、安装后的 Skill、personal cache、MCP source、bundled MCP、release zip 都可能成为“看起来像真相”的入口。当前主要靠脚本复制和 hash 检查，尚未建立明确的：

```text
canonical source -> generated surfaces -> artifact manifest -> installed receipt
```

### 6.6 根因六：任务治理只记录执行，不记录价值和决策

job/plan 事件能回答“哪个进程退出了”，不能稳定回答“用户要的结果是否达成”。因此失败只能靠 Codex 临场阅读长日志，经验不能沉淀成下一次路由、冷却和停止规则。

## 7. 外部调研结论与可借鉴机制

### 7.1 Codex Plugin / MCP：接受“新上下文”而不是承诺热替换

OpenAI Codex 官方搜索结果说明：Plugin 是可安装能力包，可以包含 Skill、MCP-backed app；Codex Desktop、CLI 和 IDE 共享 Codex host 的 MCP 配置；Plugin 添加的 MCP 工具面向新 chat/新上下文加载。

MCP 规范 `2025-06-18` 的 `tools` 能力允许服务器声明 `listChanged` 并发送通知，但规范并没有承诺宿主必须把当前已经存在的模型对话重新构建成新工具快照。

**对本项目的结论：**

- 不再承诺“编辑源码后当前对话实时热更新”。
- 开发验证使用独立 dev profile、候选 generation、明确 reload/restart 和新 context。
- 当前对话继续使用旧工具面时必须显式显示 generation mismatch，而不是静默退回旧 wrapper。

### 7.2 VS Code：成熟的做法是开发宿主，而不是改生产宿主

VS Code 官方扩展教程采用 Extension Development Host：F5 打开一个独立开发窗口，修改代码后在开发窗口执行 `Developer: Reload Window` 验证新行为。

**对本项目的借鉴：**

- 设立 Codex Praetor dev channel/profile，所有本地迭代先进入隔离 host。
- dev profile 可自动安装候选包、生成诊断报告，但不覆盖 stable active receipt。
- 只有候选 proof 和用户验收通过后，才提升到 stable。

### 7.3 JetBrains：插件兼容性要做成 CI 里的 verifier

JetBrains Plugin Verifier 检查插件与多个 IDE build 的二进制兼容性，并可直接集成 Gradle/CI。

**对本项目的借鉴：**

- 建立 runtime compatibility matrix：Codex Desktop/CLI host 版本、MCP contract、Skill/Plugin manifest、Node、provider CLI、OS。
- 兼容性结果不是 README 文案，而是 CI artifact 和 release receipt。

### 7.4 GitHub Immutable Releases：发布采用 draft -> 全资产 -> publish

GitHub 官方文档说明 immutable release 会锁定 tag 和资产；发布后不能移动 tag、删除或替换资产；官方建议先创建 draft、上传所有资产，再 publish。Immutable release 还会生成 release attestation。

**对本项目的借鉴：**

- `v0.5.0-alpha` 不覆盖、不修补为新代码。
- 以后先构建候选包、完成证据、上传 draft，再由用户批准 publish。
- 真实修复必须产生新 tag/generation。

### 7.5 GitHub Artifact Attestations、Rulesets：把治理移出“口头约定”

GitHub artifact attestation 要求 workflow 具备 `id-token: write`、`attestations: write` 等权限，并用 `actions/attest` 为二进制或 SBOM 建立 provenance。Rulesets/branch protection 可以要求 PR、required checks、禁止 force push 和限制 tag 操作。

**对本项目的借鉴：**

- `main` 的 PR、CI、review、tag 保护不能只写在 AGENTS.md。
- release workflow 生成 zip、hash、manifest、SBOM、attestation。
- 用户下载的包必须能反查到 commit、workflow 和 generation。

### 7.6 release-please / Changesets：把“代码合并”和“版本发布”连接起来

release-please 的官方 README 采用 Release PR：根据提交历史维护一个持续更新的 release PR，合并后更新 changelog、版本、tag 和 GitHub Release。Changesets 采用 changeset 文件描述用户可见变更，再由 release PR 汇总版本和 changelog。

**对本项目的借鉴：**

- 每个用户可见 PR 必须带 release intent 和 changeset/release metadata。
- PR 合并后由 release PR 或 release workflow 产生新版本，而不是手工记住“哪些文件也要改”。
- 仍保留用户发布审批，不自动替用户发布公开资产。

### 7.7 Temporal：借鉴事件历史和不可重试错误，不引入服务化平台

Temporal 官方文档把 workflow progress 记录为 append-only Event History，以支持崩溃恢复和审计；永久性错误（参数错误、授权错误、缺失记录）应标记 non-retryable，避免无意义重试。

**对本项目的借鉴：**

- 为 logical task 保存 append-only attempts/outcomes，而不是覆盖 task 状态。
- 把 provider drift、schema error、权限缺失、用户决策和网络瞬时错误分开。
- provider contract/参数错误立即 blocked；网络超时才按预算重试。
- 不引入 Temporal server，因为当前是单机 Codex 插件，不需要服务化持久执行平台。

### 7.8 NIST SSDF 与 CI/CD 研究：构建、测试、发布、来源证明必须连成供应链

NIST SP 800-218 SSDF 以安全开发实践和供应链风险为核心；CI/CD 系统综述强调持续集成、自动化测试、可见性、部署安全和反馈环。它们共同指向：发布不是“打一个 zip”，而是可追溯、可重放、可验证的流水线。

## 8. 目标整体方案：从“脚本集合”改成“交付控制面”

### 8.1 统一状态机

所有发布和本机交付状态统一为：

```text
source_ready
  -> candidate_built
  -> candidate_verified
  -> release_draft
  -> release_published
  -> remote_asset_verified
  -> staged
  -> awaiting_host_refresh
  -> fresh_context_passed
  -> provider_readiness_passed
  -> active
  -> user_path_passed
  -> delivered
```

任何状态失败只能进入明确的：

```text
blocked / needs_user_action / degraded / retryable
```

不能用一个“PASS”覆盖多个层。

### 8.2 一个事实源、多个投影

新增 canonical release receipt/controller，所有表面都由它投影：

- source generation manifest
- plugin/Skill/MCP/cache tree hashes
- release tag、commit、zip、hash、SBOM、attestation
- host identity、工具集合、runtime contract
- provider tuple、CLI hash、版本、过期时间
- active generation、rollback generation、retirement 状态
- user-path 验收和未完成原因

Health、README 状态摘要、MCP `runtime_info`、安装器和 release notes 都不再各自推断真相。

### 8.3 开发态实时验证方案

不能实现“当前 Codex 对话无感热替换”，应采用以下成熟边界：

1. `dev` channel 使用隔离 `UserProfileRoot` 和独立 generation。
2. 本地构建后执行 `dev-stage`，安装到 dev profile，不碰 stable active。
3. dev-stage 输出一个可复制的 host refresh action：重启 Codex 或启动新的 fresh context。
4. fresh context 首先调用 `runtime_info`，不匹配就停止，不再继续测试。
5. 完成 native tool、dry-run、provider canary 后生成 dev receipt。
6. 用户在新 context 中验证自然语言入口；满意后才进入 release candidate。
7. stable 仍只来自 GitHub Release remote artifact，不直接从源码激活。

这解决的是“快速试最新改动”和“稳定用户入口”之间的冲突，而不是强行让 host 违反自身生命周期。

## 9. 下一阶段 PR：统一交付控制面与开发验证隔离

下一阶段不新增 provider、不新增大功能、不继续做散点补丁。建议一个发布影响 PR，内部按原子提交组织，但必须作为一套用户可理解的改变验收。

### 9.1 PR 目标

> 让一次合并后的变更能够被明确地构建、验证、发布、安装、刷新 host、激活、回滚和交付；任何一层未完成，都不能把整体写成 ready。

### 9.2 实施顺序

#### 阶段 0：状态和合同冻结

1. 写 ADR：定义 source/candidate/release/staged/active/delivered 状态。
2. 定义 release receipt v3 和 provider readiness v3。
3. 定义 generation identity、host identity、tool surface、user-path proof 字段。
4. 画状态转换表，明确每个状态的唯一写入者、失败分类和恢复动作。
5. 明确排除项：新 provider、新 MCP 业务工具、GUI、后台服务化。

通过门：每个当前真实问题都能映射到一个状态或一个证据字段。

#### 阶段 1：修复当前 0.5.0 暴露的组合缺陷

1. 将 readiness helper 改成真正的无参数模块，或改为显式 module/function import，禁止点源污染调用者参数。
2. 增加“点源后调用者变量保持不变”的 PowerShell 回归测试。
3. 用真实 `test-provider-capability-canary.ps1 -DryRun` 验证 wrapper 参数、tier、provider、readiness tuple。
4. 在有授权的本机验证环境跑一次 apply canary；没有 provider 授权时明确为 `needs_user_action`，不能伪造 passed。
5. 将 helper、wrapper、plugin copy、Skill copy 的行为测试纳入同一矩阵。

#### 阶段 2：建立 canonical controller

1. 让 build、publish、stage、activate、health、maintenance、retirement 都读写同一 receipt schema。
2. 发布前必须有 candidate proof；发布后必须下载远端资产并证明 hash 与 candidate 一致。
3. stage 写 `staged`，不再让 stage 输出“可交付”语义。
4. host 未刷新进入 `awaiting_host_refresh`，而不是普通失败字符串。
5. activate 只接受同一 generation 的 fresh proof 与 readiness。
6. user-path proof 完成后才写 `delivered`。

#### 阶段 3：建立 dev/stable 双通道

1. `dev-stage` 接受本地候选包和隔离 profile。
2. `stable-stage` 只接受远端 Release zip。
3. dev profile 的 receipt、cache、marketplace、Skill 和 maintenance 与 stable 分开。
4. 提供 `dev-verify`：输出 host refresh、fresh context prompt、runtime_info 对比和 dry-run 结果。
5. 明确“当前对话不热更新”是产品边界，并在 UI/文档中输出下一步，而不是静默继续。

#### 阶段 4：补齐真实交付测试

1. 公共 CI：源码、MCP、PowerShell parse、package、determinism、public scan、receipt schema。
2. 隔离 closeout CI：candidate stage、fake host mismatch、activation refuse、retirement。
3. 本机 release-candidate gate：真实 Desktop native proof、provider canary、普通用户路径。
4. wrapper contract matrix：Windows PowerShell 5.1/7、provider/tier/model/permission/task kind。
5. 供应链 gate：SBOM、artifact attestation、immutable release、tag/commit/hash 对账。

#### 阶段 5：文档和规则同步

1. README、安装、排错、roadmap、release notes 从 receipt/版本生成，不允许手工散改。
2. 历史报告增加“已被当前事实覆盖”的状态，不再把旧结论继续当现状。
3. 写用户能看懂的状态：代码已合并、Release 已公开、已暂存、等待刷新、已激活、产品已交付。

### 9.3 PR 通过标准

- 工作树干净，目标分支为最新 `main`。
- 所有新增状态都有 schema、写入者和读取者。
- 目标 wrapper 的真实 canary 不再因 helper 点源污染失败。
- 目标 generation 的 candidate proof、remote asset hash、stage、activate 和 user-path proof 能闭环。
- 真实 host 未刷新时，流程明确停在 `awaiting_host_refresh`，不再产生误报。
- 文档、版本、Release notes、安装入口和包内容来自同一 generation。
- 公共 CI、隔离 closeout、release-candidate gate 三层职责分开且不会互相冒充。

## 10. 长期 PR -> GitHub -> Release -> 本机同步流程

### 10.1 每个 PR 的分类

PR 必须声明：

- 是否改变 MCP 工具名/参数/输出合同。
- 是否改变 Skill/Plugin manifest 或安装入口。
- 是否改变 provider wrapper/permission/readiness。
- 是否属于发布影响变更。
- 版本和 Release intent。
- 用户可见行为和普通路径验收方法。

### 10.2 合并前

1. Codex 建 `codex/` 分支并完成实现、测试、构建和文档。
2. CI 只证明公共可重复门禁，不冒充本机 provider 或 Desktop proof。
3. PR 描述明确版本、发布策略、用户入口和合并后动作。
4. 用户在 GitHub 创建、审核和合并 PR。

### 10.3 合并后

1. 回到最新 `main`，生成唯一 candidate generation。
2. 在 draft release 中上传 zip、`.sha256`、manifest、SBOM、attestation。
3. 用户批准后 publish immutable release。
4. 从远端下载同一资产，验证 hash、manifest、公开入口。
5. stable stage，写 staged receipt。
6. 刷新 Desktop host，生成 native fresh-context proof。
7. 生成同 generation 的 provider readiness。
8. activate，安装并验证 maintenance task。
9. 重新从普通用户路径下载、安装、打开、dry-run、只读 canary。
10. health 必须为 ready，receipt 必须为 delivered，retirement 状态明确。

### 10.4 失败处理

- build/package 失败：修复 PR，不发布。
- native host 旧：`needs_user_action`，等待重启/刷新，不重复开新任务冒充修复。
- provider schema/permission 不兼容：`blocked`，不重试同一 tuple。
- provider 网络超时：按有限 attempt budget 重试。
- 远端 hash 不同：废弃 candidate，使用新 generation，不覆盖旧 tag。
- activation 失败：保留旧 active generation，不能回写部分 receipt。
- retirement 被占用：`blocked_by_process`，保留旧路径并记录下次重试时间。

## 11. 规则修订建议

### 11.1 全局规则

建议增加四条跨项目规则：

1. **状态必须分层**：本地完成、PR 就绪、代码已合并、Release 已公开、已激活、产品已交付不得混用。
2. **证据先于结论**：任何“完成/ready/可用”必须绑定真实文件、命令、进程、远端资产或 receipt；状态文字不能作为证据。
3. **外部 host 边界必须显式化**：不能把新任务、reload 探针或 separate-host 进程冒充已经运行的 Desktop host 刷新。
4. **长期流程优先使用 canonical receipt/ledger**：不要再让多个脚本各自维护同一事实。

### 11.2 Codex Praetor 项目规则

建议具体修订：

- 将 `release published` 和 `product delivered` 作为不同的强制状态。
- 将 `awaiting_host_refresh`、`provider_readiness_pending`、`user_path_pending` 纳入 receipt。
- 明确“发布影响 PR 的 public upload 前，candidate proof 必须通过；发布后仍必须 remote re-verify”。
- 明确 provider readiness 不能来自旧 generation、fake readiness 或仅 dry-run。
- 明确 source/skill/plugin/cache 的 canonical source 和生成检查。
- 明确 dev profile 可验证最新候选，但不覆盖 stable；stable 不允许从源码直接 activate。
- 将 release closeout smoke 与真实 release-candidate gate 的职责分开命名。
- 维护任务安装必须注册后读回验证；卸载必须确认任务不存在后才能报告成功。

### 11.3 报告与路线图规则

- 历史报告只描述当时状态，现状报告必须以当前 receipt、GitHub 和 runtime probe 为准。
- 每个“已完成”必须附 `evidence_path`、时间和 generation。
- 旧报告中的未完成项先对账，不能直接复制到新路线图。
- 每个 PR 计划必须有明确排除项、通过门、用户动作和回滚路径。
