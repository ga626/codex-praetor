# Codex 执政官：真实挖掘实验场与系统治理大 PR 规划

> 日期：2026-07-22
> 性质：以当前 `0.8.3-alpha` 恢复分支为基线的内部考古、外部调研、真实实验与统一治理设计。
> 本文的结论不是“已经验收完成”；它规定接下来如何先把问题尽可能挖全，再只提交一个集中修复 PR。

## 先说结论

这次**不应再把 `d53bf10` 单独提成一个修复 PR**。它只证明了一件事：CodeBuddy 当前 CLI 的编辑权限参数应使用 `-y --tools Read,Glob,Grep,Edit,Write,Bash`，而不是已经失效的 `--permission-mode dontAsk --allowedTools ...`。它没有证明产品在真实工作中全面可用。

正确路线是：

1. 保留当前分支 `codex/recover-codebuddy-edit-permissions` 作为唯一的大 PR 分支；不另建产品修复分支。
2. 建立一个由 Codex Desktop 创建的独立 worktree 任务，作为**可丢弃的真实实验场**。worker 可以真实改代码、跑测试、联网、使用已登录的 provider，并消耗正常额度；任何实验 diff、日志和临时文件都不合入主线。
3. 先完成覆盖面尽量大的挖掘，按依赖关系补齐只能在前置问题修复后才能做的测试；每轮只收集可复核证据，不急着打一行补丁。
4. 把已复现的确定性产品缺陷、重复实现和流程矛盾统一归并，按系统边界修复；然后在当前分支一次完成版本、发布物、文档、回归、构建和用户路径验证，形成一个发布影响 PR。

**不需要操作系统级沙箱。** Git worktree 已足以隔离仓库内的代码修改，正符合这轮目标：保护主检出、稳定安装和将来要发布的代码，而不是把外部 agent 当作恶意程序来研究。它不是系统隔离：端口、全局配置、账号、缓存和运行中的服务仍共享。因此测试合同要如实写明两层边界：

- 允许：在实验 worktree 中真实读改项目、运行项目命令、联网、通过官方 CLI 使用已登录账号与正常额度。
- 仍禁止：修改主检出、稳定 marketplace、插件缓存、第三方 agent 的认证数据库或内部状态；不得读取、打印、提交或外传 token/cookie/密钥。"可使用账号"是通过官方 CLI 使用已有登录态，并不等于允许把认证文件当测试素材。

## 2. 这轮真正要挖什么

以前只做过“一个 provider、一次短任务、一次写文件”的恢复验证；那是事故止血，不是产品验收。本轮把问题分成四类，避免把账号、发布和代码错误混成一团。

| 挖掘面 | 要回答的问题 | 通过的最低证据 |
| --- | --- | --- |
| 真实任务能力 | provider 能否读代码、做合理修改、执行测试并留下可验 diff | job、stdout/stderr、completion、diff、测试输出、Codex 复核相互一致 |
| 失败与恢复 | 权限拒绝、无 diff、测试失败、超时、MaxTurns、watcher/清理失败能否被准确分类，不伪装成功或运行中 | 可重复失败注入 + 合同/状态断言 |
| 隔离与副作用 | 主检出、稳定安装与实验 worktree 是否被正确分开；结束后是否能完整回收 | 主检出前后 hash/status、`git worktree list --porcelain`、清理回执 |
| 架构与流程 | 现有层是否存在互相矛盾的规则、镜像漂移、只写状态不执行的“假控制”、重复的编排轮子 | 源码路径、测试覆盖和真实运行共同证明；不是仅凭文字判断 |

### 已经发现的架构疑点

以下是截至本文写作时的**已确认问题或高优先级待证伪假设**，不能只局部修一个参数：

1. **联网合同自相矛盾（已确认）。** `invoke-codex-praetor.ps1` 接受 `-AllowWorkerNetwork`，合同也写 `worker_network=allowed_by_codex`；但发给任何 worker 的固定提示词仍是“Do not perform external network research”。因此调用者以为允许，worker 却被禁止。现有测试没有断言“合同、提示词与实际命令”三者一致。
2. **路径合同不是强制隔离（已确认）。** `AllowedPath` / `ForbiddenPath` 被写进 job contract，但没有执行层拦截。工作树只隔离 Git 的工作目录和索引；它不隔离 OS 文件、端口、用户配置或 provider 登录态。产品不应把它宣传成 sandbox，也不能把合同字段误当安全控制。
3. **账号边界表述过宽（待治理）。** 现有“不要碰 auth”本意是保护秘密和 provider 内部数据库，但提示词把它与“不能使用已有账号能力”混淆。应改为“可通过官方 CLI 使用现有登录态；不得读写、输出或迁移认证数据”。
4. **真实能力覆盖不足（已确认）。** CodeBuddy 已有最小编辑成功证据；Qoder、MiMo、完整项目修改、fail-to-pass、无产出、超时、清理、并发与 background 均未被这一证据覆盖。
5. **控制层过度复杂的风险（待证伪）。** wrapper、watcher、ledger、MCP 薄层、Skill 三份镜像、release/activation 链路都携带状态。需用“一个任务从发起到清理”的实际事件图检查是否有重复状态源、死状态或不同层给出不同结论；不能仅继续增加 gate。

## 3. 外部调研结论：采用什么，不采用什么

本轮先由 KnowledgeRadar 规划路线，并抽取官方 Git worktree 文档、SWE-bench 评估材料、OpenAI eval 资料、GitHub Artifact Attestations 文档、学术资料及开源实现生态；详细来源和强度见同名 evidence sidecar。

| 外部证据 | 可直接采用的做法 | 本项目不照搬的部分 |
| --- | --- | --- |
| Git 官方 `git worktree` 文档 | 可用 detached/throwaway linked worktree 做实验；用 `git worktree remove` 正常回收，`list --porcelain` 与 `prune` 复核，不能只删目录 | 不需要 Docker/VM 才能保护主仓库；但也不能声称 worktree 隔离全系统 |
| SWE-bench 评估方法 | 不信 agent 的口头“完成”；在真实代码上应用 patch，验证 fail-to-pass 和回归测试 | 不把完整 benchmark/容器集群搬进 alpha 项目；先做小型、可重复、贴近本项目的 fixture |
| 评估方法的共同原则（以 SWE-bench 的真实补丁/测试证据为主） | 用短自然语言任务驱动，外围采集 trace、产物和少量稳定检查，比较可复核结果 | 不把复杂评分框架放进 worker prompt；人仍需验 diff 与边界 |
| GitHub artifact attestation | artifact 身份只能证明构建和来源，继续保留 build/upload/download 的同一 artifact 证据 | attestation 不能替代 provider 可用性或真实任务成功 |
| 开源 harness 生态 | trace、任务定义、环境和结果要分离保存；失败样本和成功样本同样重要 | 不引入新的通用多 agent 平台或编排框架；Codex Praetor 保持“外部 CLI worker 调度器”定位 |

因此系统设计选择是：**少造一层平台，多补一层可验证实验合同。** 运行期仍是现有官方 CLI + Git + Codex；项目只负责明确的任务合同、worktree 生命周期、证据采集、状态归因和发布物一致性。

## 4. 真实实验场设计

### 4.1 环境与权限

实验任务由 Codex Desktop 在 `codex/recover-codebuddy-edit-permissions` 基础上创建独立 worktree。该 worktree 不承载最终 PR；实验中产生的代码改动、fixture、报告、下载物和 job 记录全部保留到任务自身目录或项目的忽略运行根，最终不 cherry-pick。

每个真实 worker 的提示词只保留业务目标，例如：

```text
在这个项目里完成一个小而完整的改进，修改代码并运行相关测试；完成后说明改了什么、测试结果和剩余风险。
```

协议、允许联网、最大时长、worktree、基础提交、收集哪些证据、禁止改稳定安装等要求由调度器的外层合同处理，不塞给模型。这样测到的是它处理真实任务的能力，而不是它复述一大段流程的能力。

### 4.2 测试矩阵与依赖

| 编号 | 类型 | 是否可立即执行 | 目标 | 结束后判定 |
| --- | --- | --- | --- | --- |
| E1 | 控制面静态审计 | 是 | 从 MCP → wrapper → watcher → ledger → cleanup 画出状态与证据流，找重复状态源和未执行合同字段 | 架构问题清单，不改主线 |
| E2 | 三 provider capability canary | 是，逐个顺序 | CodeBuddy/Qoder/MiMo 的当前 CLI、模型、权限 tuple 是否能真实启动 | 成功/登录/模型/网络/协议分类 |
| E3 | 真实项目小改动 | 取决于对应 E2 | worker 在隔离 worktree 内读代码、改一个边界清楚的功能、跑相关测试 | diff、测试、completion、主线无漂移、清理齐全 |
| E4 | fail-to-pass fixture | 是 | 初始测试必失败，worker 修代码使其通过；再跑既有回归 | 真实问题闭环而非文字回答 |
| E5 | 失败注入 | 是 | 人为给出无权限/错误任务、无 diff、测试失败、超过时间或 watcher 失败条件，验证分类和停损 | 任一终态不伪装成 running/success |
| E6 | background/通知与回收 | E3 至少一次成功后 | 不高频轮询，等待一次完成信号；验证 job、锁、worktree 和 plan 的闭环 | 完成后无悬挂锁、无错误 active lane |
| E7 | 交叉 provider 复现 | 依赖 E2/E3 中可用 tuple | 用同一短任务比较行为与 evidence 形状，区分 provider 差异和产品 bug | 只将产品可复现差异放入修复清单 |
| E8 | 已发布 artifact 验收 | 依赖最终大 PR 合并发布 | 以公开 zip 的 installed runtime 做 canary 与真实任务 | 这是收口内交付验真，不是另一个修复 PR |

这不是“每项一个 PR”。E1–E7 都是本次大 PR 的发现阶段；只有确定性、可复现、属于产品的缺陷才进入同一份修复集合。登录失效、外部额度耗尽、供应商临时故障会保留为环境结论，不被伪装成代码问题。

## 5. 统一治理范围与实施顺序

### A. 先挖、后收敛

1. 在实验 worktree 做 E1、E2、E4、E5；不合入任何实验改动。
2. 对通过 canary 的 provider 做 E3，随后做 E6/E7；每个 worker 完成后由 Codex 读取 job、日志、diff、测试和 Git 状态，不进行高频轮询。
3. 整理“已证实产品缺陷 / 本机或 provider 边界 / 未覆盖风险 / 架构建议”四张清单。没有复现证据的猜测不直接改产品。

### B. 再一次性修复

对已证实问题按下面的系统边界统一修改，而不是在多个层里复制补丁：

1. **任务合同真实化：** 让网络、账号使用方式、路径范围和副作用边界在输入、合同、worker prompt、job 元数据、MCP 返回和测试中一致。明确 worktree 是代码隔离，不是假沙箱。
2. **运行状态收敛：** 统一 job 的执行终态、evidence 状态和 supervisor verdict；任何 `process_exited`、`timed_out`、`watcher_failed`、无 diff、测试失败都不能显示为可继续运行或完成。
3. **worktree 生命周期：** 固化创建基准、主检出漂移检查、正常 remove、异常保留证据、`list --porcelain` / `prune` 复核；清理失败是独立状态，不能抹掉失败证据。
4. **三份产品镜像：** 根 `scripts/dispatch`、`skill/`、`plugin/skills/` 仅保留一个权威源和可验证同步，打包前检测源码、MCP dist、plugin MCP dist 与 runtime contract 漂移。
5. **发布与用户路径：** 在同一 PR 更新版本、release intent、changelog、release notes、安装/验收文档；main 合并后只能从合并 SHA 自动发布、远端下载复验、自动本机激活。公开交付成功后若 host 未刷新，只报告 `needs_host_restart`，不新开收口补丁。

### C. 最后才进入 PR 前阶段

1. 重新以最新 `origin/main` 复核分支基线并处理冲突；保留 `d53bf10` 的权限恢复作为大 PR 的一部分。
2. 跑与风险相符的完整验证：所有 PowerShell 解析/编码、核心回归、MCP 测试、最终 stage/zip runtime、确定性构建、release-intent、workflow/action pin、用户安装烟测和本文列出的故障注入回归。
3. 做最终差异审计：无实验产物、秘密、本机数据库、截图、job 日志或无关报告进入 Git；PR 仅包含产品代码、测试、发布面、用户文档与必要报告/evidence。
4. 输出中文 PR 标题和完整正文（摘要、改动、验证、边界、用户影响、版本/tag/发布计划、合并后动作），但**在这之前不创建 PR**。

## 6. 合并后的唯一收口

本次大 PR 是发布影响 PR。合并以后不能出现“代码合并、让用户自己下载新包、下次再补收口”的第三种状态。

收口顺序固定为：

1. `Release On Main` 从精确合并 SHA 构建、上传并生成不可变 tag/Release。
2. 普通用户路径下载公开 zip，验证它和 release manifest/attestation 是同一 artifact。
3. 自动下载同一 zip，调用包内安装器更新 stable marketplace，再运行官方 `codex plugin add codex-praetor@personal`，检查 `codex plugin list` 的版本；用户不需要手工下载、解压或改 cache。
4. 若需要刷新 host，准确报告 `needs_host_restart`。用户刷新后先验 `runtime_info`，再跑当前 generation 的 canary 和一项真实 worker 验收。

若第 1–2 步失败，结论只能是“代码已合并，产品未交付（release incident）”，停止下一次发布影响 PR，优先重跑原 SHA；不能用同版本手工上传或“收口修复 PR”掩盖它。若第 3–4 步仅因本机 host 尚未刷新，则产品仍可已交付，按状态提示完成本机体验，不反向否定 Release。

## 6.1 实验执行结果（已完成，不合入实验改动）

独立 Codex worktree 完成了 E1–E6 的核心路径。实验目录、job、日志、diff 与 fixture 都留在该 worktree 的忽略实验根；没有 commit、PR 或合并，实验主 worktree 最终干净。

| 项目 | 真实结果 | 对大 PR 的处理 |
| --- | --- | --- |
| 旧安装与恢复分支比较 | host 仍加载 `0.8.2-alpha`；旧 wrapper 的 CodeBuddy 仍用失效的 `dontAsk/allowedTools`，编辑被拒绝；`d53` 源码 wrapper + 隔离 profile 真实创建了 edit diff | 证实 `d53` 是已知 release incident 的正确恢复，不把旧安装失败误报为新 bug |
| 真实只读、background、Qoder edit | worker/job/watcher/lock/worktree 链路真实跑通；成功任务都只记录为 `evidence_missing` | watcher 改为记录 `report_valid` 或 `artifact_valid` 的客观观察，仍由 Codex 最终验收 |
| fail-to-pass fixture | Qoder 在独立 Git fixture 中把错误的 `$A - $B` 改为 `$A + $B`，外围复跑 `test.ps1` 为 `PASS` | 保留为今后发布前的真实验收样式，不把 fixture 文件合入产品 |
| CodeBuddy edit canary | 旧 `0.8.2` 的 edit canary 通过但真实编辑被拒绝，说明原 canary 只证明“能启动并回答”，不能证明有写权限 | `code_change` canary 改为必须在 worker worktree 留下 marker 文件和 Git 变化 |
| readiness 多代际状态 | 源码 `0.8.3` canary 改写文件顶层 generation 后，虽然文件中仍有 `0.8.2` entry，`0.8.2` health 却看成无 entries | readiness 改为按 entry 的 generation/contract/tuple 判断；顶层字段只作旧格式摘要，不能遮蔽其他 generation |
| 开发候选验收 | 直接用源码 wrapper 会被 stable cache generation 正确挡住；`-RuntimeChannel dev` 不会改变这一点。隔离 `UserProfileRoot` 可在不触碰 stable 安装的前提下让源码 candidate 做完整真实验收 | 固化“开发 profile 与 stable 安装分离”的流程；不以源码检出冒充已安装产品 |

这轮也确认了一个边界：Git worktree 保护了主项目代码，但不能隔离用户 profile。因此开发候选必须显式传隔离 `UserProfileRoot` 和隔离 readiness；稳定用户路径只在公开 Release 的自动激活后验证。两者都是真实测试，证明的对象不同。

### 6.2 发布前供应链复核（已完成，纳入同一 PR）

在分支推送前的 GitHub Dependabot 复核发现两个 MCP 运行时传递依赖告警：`fast-uri 3.1.3` 的 host-confusion 高危告警，以及 `@hono/node-server 1.19.14` 的 Windows static-service 中危告警。当前产品只启动 stdio MCP，未启用 Hono HTTP/静态服务，也未实现 URL 主机白名单；静态阅读未能建立可利用产品路径。它们仍会被打进用户 zip，因此不能仅以“当前不可达”忽略。

修复没有降级 SDK：npm 推荐的 `1.24.3` 虽能移除 Hono，却会触发 SDK 自身的高危告警。最终保留 `@modelcontextprotocol/sdk 1.29.0`，用 npm `overrides` 固定 `@hono/node-server 2.0.11` 和 `fast-uri 3.1.4`，并把该不变量加入供应链回归。`npm ci`、MCP 构建/自测与 `npm audit --omit=dev` 均通过，审计结果为零项漏洞。

## 7. 完成定义与不确定性

这轮不能诚实地承诺“以后绝无任何外部故障”；provider、网络、账号和 Codex Desktop 本身都可变化。可以承诺并验收的是：
