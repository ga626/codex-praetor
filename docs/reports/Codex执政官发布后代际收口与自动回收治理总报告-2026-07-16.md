# Codex 执政官发布后代际收口与自动回收治理总报告

日期：2026-07-16
调查对象：`D:\Projects\CodexPraetor`、Codex Desktop/app-server、Codex Praetor personal plugin/cache、Windows 文件占用生命周期、Windows PowerShell 5.1/7。
目标：把“什么时候刷新、什么时候开 fresh context、旧 generation 什么时候清理、PowerShell 如何长期可靠判断 provider 成功”变成可执行的项目规则和长期机制。
本轮实现目标：把报告中的项目规则、退休登记、自动回收、用户级重试和原生进程调用契约落到源码、安装入口、验证脚本和用户文档；不在当前开发机改写稳定安装态，不删除用户缓存，不重启 Codex。

## 一、给用户的直接答案

### 1. 你不需要每次修改都新建 Codex task

后续按“发布代际”和“工具合同”判断，不按文件修改次数判断：

| 场景 | Codex 的动作 | 你的动作 |
|---|---|---|
| 普通实现修改，MCP 工具名/参数、Skill frontmatter、Plugin manifest 不变 | 继续当前开发 task；必要时做 server reload 和轻量 probe | 不需要新建 task |
| MCP 工具名、参数、Skill/Plugin manifest、插件来源或 cache generation 改变 | 当前 task 继续写代码；合并后由 Codex 做一次新 generation stage；只为这一代做一次 fresh-context native canary | 打开一个新的 Codex 验收 context；不是每次编辑都开 |
| 普通 PR 合并，但不影响用户入口、安装包或插件合同 | 目标分支/CI 检查后直接收口 | 不需要发布操作 |
| 发布影响 PR 合并 | 同步最新 `main`，构建/下载/校验 Release，stage、fresh proof、provider readiness、activate、自动回收 | 只在需要新工具合同的代际打开一次 fresh context；发布审批仍由你决定 |

因此，**fresh context 是“每个变更代际一次”的验收动作，不是“每次改文件一次”的开发动作**。当前 Codex 对已有对话的 Skill、Plugin cache path 和模型工具快照没有稳定的强制替换接口；这部分不能虚构成已经解决。

### 2. 旧东西什么时候清理

旧 generation 不应该在发布脚本中直接删除，也不应该覆盖原来的 version 目录。正确生命周期是：

1. 新 generation 在独立目录完整落地并 hash 校验。
2. activation 只改变 active receipt/活动事实，不改写旧 generation。
3. 旧 generation 进入 `retired/pending` 登记，不再作为 active surface，也不再被新 dispatch 选用。
4. 自动回收器在发布收口、用户登录、定时维护时反复尝试清理。
5. 如果 Windows 报告文件仍被进程使用，状态记为 `blocked_by_process`，保留目录，稍后自动重试；不强杀 Codex、不强删目录、不让新旧内容互相覆盖。
6. 当删除成功，登记为 `deleted`；临时目录、失败目录和备份也进入同一回收队列。

这意味着“干净”不是在发布那一秒强行把旧目录抹掉，而是：**活动面只有一代；旧代际不会与新代际共用可变路径；旧代际由机器自动追踪并在安全时机回收，用户不需要每次手动提醒 Codex 清理。**

### 3. 能不能保证所有旧目录最终都删掉

能保证的是：

- 新旧 generation 不原地覆盖，不会因为清理失败导致新版本没有安装进去。
- active generation 永远不会被回收器删除。
- 旧 generation 会有机器可读的退休登记、最后尝试时间、阻塞原因和下一次重试时间。
- 被 Windows 进程占用的旧文件，在释放后可自动回收。

不能由 Codex Praetor 单方面保证的是：Codex 平台把旧 cache 的绝对路径写进了长期 session；即使当前没有打开文件句柄，未来恢复一个很老的 session 仍可能需要那个旧路径。Codex 官方 Issue #25285 已记录这一行为，并说明删除旧 hash 目录后旧 session 可能丢失 Skill；这属于平台内部 session 持久化缺陷。若用户接受每个工具合同变更代际使用 fresh context，旧 session 不再是交付路径，旧 generation 才可以按保留窗口自动清理。

因此规则必须把“可用性保留”和“磁盘清洁”分开：同一合同的开发代际可以保留兼容窗口；合同变化的旧 session 以 fresh context 为替代入口，不能为了保留它而永久堆积所有缓存。

### 4. 第二个问题如何一劳永逸解决

第二个问题不是 Qoder 不能工作，而是 PowerShell 包装器的 native process 观测契约错误：把 stderr 合并进 PowerShell success stream，并在 `$ErrorActionPreference = "Stop"` 下于读取 `$LASTEXITCODE` 之前处理它。`git worktree add` 成功时也会向 stderr 输出进度信息，所以“stderr 非空”不能等价于“失败”。

长期解决不能只改一行 `2>&1`。项目需要一个统一的 native invocation contract：

- stdout 与 stderr 始终分开捕获；
- 子进程退出后读取 exit code；
- exit code 是成功/失败的第一判断；
- stderr 是诊断字段，不是默认失败条件；
- 超时、启动失败、非零退出、marker 缺失、状态漂移分别分类；
- 所有 dispatch、release、provider canary、GitHub 检查脚本复用同一 helper 和测试矩阵；
- Windows PowerShell 5.1 与 PowerShell 7 都必须通过。

## 二、项目规则变更

本次实现同步修改根目录 `AGENTS.md`，把报告中的判断落成 Codex 每次任务都能执行的短规则：

1. **阶段规则**：区分 PR 开发中、PR 就绪、PR 合并后收口和产品已交付；普通实现修改不新建 task，工具合同变化每个版本代际只做一次全新上下文验收。
2. **交付判断**：明确 GitHub Release zip 是用户入口；发布影响文件必须走 stage/hash-verify、全新上下文 MCP 证明、provider readiness、activate 和普通用户路径复验。
3. **代际回收**：发布器不得直接删除旧目录；旧路径必须登记到退休清单，回收器幂等重试，活动代际不参与回收，回收失败不能改变活动收据。
4. **原生进程**：统一按退出码判断 native CLI，stdout/stderr 分离，禁止用 `2>&1` 代替错误判定，并要求覆盖 Windows PowerShell 5.1 和 7。
5. **责任边界**：Codex 负责分支、改动、验证和 PR 材料；用户负责 GitHub 权限、合并、发布审批和体验判断。

详细命令和发布步骤继续放在 `docs/release/release-gate-checklist.md`，避免把操作手册重复塞进项目规则并增加 Codex 上下文负担。

## 三、当前项目到底处于什么状态

### 3.1 已有的正确基础

当前项目已经有不少正确的发布治理基础：

- `scripts/release/get-codex-praetor-generation.ps1` 生成 `generation_id = version + commit + content hash`。
- `scripts/release/complete-codex-praetor-release.ps1` 已区分 `stage` 和 `activate`，并以 staged receipt、fresh-context proof、provider readiness 作为 activation 前提。
- `scripts/verify/get-codex-praetor-health.ps1` 已检查 active receipt、Skill/plugin/cache tree digest、marketplace、fresh context 和 provider readiness，并在不匹配时阻断真实派工。
- `scripts/verify/test-release-closeout.ps1` 已覆盖无效 zip 不写 active receipt、stage 不激活、activate、健康检查和 surface drift 拒绝。
- 项目规则已经规定 branch build 不得覆盖 stable profile，且 release 是不可变 generation。

### 3.2 当前的关键缺口

现有机制仍有三个生命周期冲突：

1. `complete-codex-praetor-release.ps1` 的 stable cache 路径是 `cacheRoot\<version>`；它并没有把完整 `generation_id` 放进物理 cache 路径。
2. `publish-codex-praetor-personal-cache.ps1` 在成功发布后，把除当前 version 外的旧目录归类为 stale 并立即调用 `Remove-Item -Recurse -Force`。如果旧目录被 Codex 进程使用，删除失败；如果当时恰好未持有句柄但仍被旧 session 依赖，删除会使未来恢复失败。
3. `publish-codex-praetor-personal-marketplace.ps1` 和 `install-user.ps1` 使用安装目录替换、备份和 cachebuster，但没有统一的“退休登记 + 自动重试回收”协议；各脚本各自清理，无法形成单一事实来源。

所以当前状态不是“新版本没有填进去”，而是：**新版本已经可以 stage/activate，旧目录清理被放在了错误的发布时点，并且没有退休队列。**

### 3.3 本次已知现场状态

前一轮现场记录显示：

- 远端 `v0.2.0-alpha` 已安装。
- app-server reload 后显示 server version `0.2.0-alpha`、17 tools。
- 当前旧 Codex 对话仍调用 `...codex-praetor\0.1.1-alpha\scripts\dispatch\invoke-codex-praetor.ps1`。
- 旧 cache 被当前 Codex 进程占用，无法安全删除。
- fresh `codex exec --ephemeral --sandbox read-only` 可加载 `0.2.0-alpha`；严格 native MCP canary 17 个工具全部可见，`missing_expected=[]`，状态 `passed`。
- Windows PowerShell 为 `5.1.19041.6456`，`$PSNativeCommandUseErrorActionPreference` 不存在。
- Qoder 官方 CLI 在隔离 worktree 中返回 `CODEX_PRAETOR_CAPABILITY_CANARY_OK`；失败发生在 PowerShell wrapper 的 stderr 处理路径。

结论是“新 generation 可用、旧 generation 仍处于待回收/旧 session 兼容边界”，不是“新旧已经同时成为 active”。

## 四、外部研究得到的工程边界

### 3.1 Codex 平台边界

- [openai/codex#8957](https://github.com/openai/codex/issues/8957)：`mcpServer/refresh` 会触发 pending refresh，MCP manager 通常在后续 turn 重新初始化；不是当前模型工具快照的同步替换。
- [openai/codex#20605](https://github.com/openai/codex/issues/20605)：已有 Codex thread 无法可靠热加载本地 MCP 工具变化，fresh thread/session 是现实 workaround。
- [openai/codex#16653](https://github.com/openai/codex/issues/16653)：Skill/Plugin 变更不会自动进入当前会话，用户请求 `/reload-skills`、`/reload-plugins`。
- [openai/codex#21138](https://github.com/openai/codex/issues/21138)：Windows plugin cache 可能只按同一 version 目录判断新鲜度，源内容变更但 version 不变时继续使用 stale 内容。
- [openai/codex#24390](https://github.com/openai/codex/issues/24390)：已有 conversation 可能继续引用旧 cache path。
- [openai/codex#25285](https://github.com/openai/codex/issues/25285)：Codex Desktop Windows 可能把 volatile cache hash 的绝对 Skill 路径持久化进 session；旧 hash 删除后老 session 失效，作者建议稳定语义引用或 `current` compatibility pointer。

这些资料支持“不要覆盖/删除旧 path，fresh context 是合同变化的边界”，但它们是公开 issue，不能当作 Codex 已承诺的修复时间表。

### 3.2 Windows 文件更新边界

- [MoveFileExW](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-movefileexw) 说明 Windows 可以把不能立即处理的文件操作登记到 `PendingFileRenameOperations`，但目录只有为空时才会在重启删除；它不能解决当前用户 session 仍需要旧路径的问题。
- [Restart Manager](https://learn.microsoft.com/en-us/windows/win32/rstmgr/about-restart-manager) 说明安装器可以登记资源、识别使用文件的应用，并在有权限时停止/重启应用。它是诊断和受控安装的工具，不授权 Codex Praetor 擅自关闭当前 Codex。
- [RmRegisterResources](https://learn.microsoft.com/en-us/windows/win32/api/restartmanager/nf-restartmanager-rmregisterresources) 与 [RmGetList](https://learn.microsoft.com/en-us/windows/win32/api/restartmanager/nf-restartmanager-rmgetlist) 可以登记待回收文件并获取当前占用它们的进程；未来回收器可用它做解释性诊断，删除动作仍必须失败即延后。
- [Squirrel.Windows update process](https://github.com/Squirrel/Squirrel.Windows/blob/master/docs/using/update-process.md) 采用版本目录并行安装，下一次启动时只保留当前和紧邻上一版本，其余旧版本清理；[issue #589](https://github.com/Squirrel/Squirrel.Windows/issues/589) 记录更新中的旧版本仍会暂时保留。
- [WinSW deferred file operations](https://github.com/winsw/winsw/blob/v3/docs/deferred-file-operations.md) 明确指出 Windows 不能覆盖正在使用的文件，因此把操作延迟到目标进程启动前，并记录成功/失败。

这些成熟做法共同支持：**版本目录不可变、切换和清理分离、清理可失败且可重试、进程退出后再回收**。没有证据支持“发布脚本遇到旧目录就强制删掉”是正确做法。

### 3.3 PowerShell 边界

- [about_Error_Handling](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.core/about/about_error_handling?view=powershell-7.6) 明确 external native program 主要用非零 exit code 报告失败，PowerShell 把它存入 `$LASTEXITCODE`；默认非零退出码不自动成为 ErrorRecord。
- [about_Redirection](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.core/about/about_redirection?view=powershell-7.6) 说明 `2>&1` 会把 Error stream 合并到 Success stream。
- [Start-Process](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.management/start-process?view=powershell-7.6) 和 [.NET ProcessStartInfo](https://learn.microsoft.com/en-us/dotnet/api/system.diagnostics.processstartinfo?view=net-10.0) 提供独立 stdout/stderr 重定向；本机 Windows PowerShell 5.1 下也应优先使用可兼容的 `ProcessStartInfo` 或 `Start-Process -RedirectStandardOutput/-RedirectStandardError`。

因此应统一为 exit-code-first，而不是依赖 PowerShell 版本差异或静默吞 stderr。

## 五、长期方案：四个不变量

### 不变量 A：任何 generation 不原地改写

- 每个 stable release 使用新的版本/tag，并生成新的 `generation_id`。
- plugin、Skill、cache、receipt、fresh proof、provider readiness 都带同一 generation identity。
- 不能把新内容复制到一个仍可能被旧对话引用的 version 目录后再删除旧内容。
- branch/dev 只能写隔离 `UserProfileRoot`，不能影响 stable。

### 不变量 B：active 与 retired 是两个状态

建议引入统一的 retirement manifest，例如：

```json
{
  "schema": "codex-praetor-generation-retirement/v1",
  "active_generation_id": "0.3.0-alpha--<commit>--<hash>",
  "entries": [
    {
      "generation_id": "0.2.0-alpha--<commit>--<hash>",
      "paths": ["<cache-path>", "<plugin-backup-path>"],
      "status": "pending",
      "first_retired_at": "2026-07-16T...Z",
      "next_attempt_at": "2026-07-16T...Z",
      "attempts": 0,
      "last_error": ""
    }
  ]
}
```

状态至少包括：`active`、`staged`、`retired_pending`、`blocked_by_process`、`deferred_retention`、`deleted`、`failed_manual_review`。

### 不变量 C：自动回收器只做安全动作

回收器的触发点：

1. `activate` 成功后立即运行一次；
2. setup/release closeout 成功后运行一次；
3. 用户登录时运行一次；
4. 用户级 Task Scheduler 每 15 分钟运行一次，至少持续到退休队列清空；
5. 下一次 Codex Praetor health/dispatch 前做轻量 reconcile，但不阻塞正常 dry-run。

回收器的动作：

- 读取 active receipt，不碰 active generation、当前 staged generation 和仍在保留窗口内的 generation。
- 检查退休路径是否位于允许的 cache/backup 根目录下，拒绝链接跳转和越界路径。
- 用 Restart Manager 或等价只读占用探针记录可能的占用进程；不自动关闭 Codex。
- 尝试删除退休目录；成功才记 `deleted`。
- 遇到 sharing violation、权限、路径不存在或临时 IO 错误，分别记状态并计算下次重试时间。
- 连续失败超过阈值时仍不删除、不强杀，升级为 `failed_manual_review`，同时保留完整错误原因。
- 清理结果只写本地运行态，不改变 active receipt，不改变 provider readiness，不把回收失败误报成 release activation 失败。

“每 15 分钟”的周期是实现建议；实际实现可在 Windows Task Scheduler 不可用时退化为下次 setup/closeout/health 触发，但必须在报告中标记 `cleanup_degraded`，不能静默说已自动清理。

### 不变量 D：旧 session 兼容性必须有边界

- 普通代码变更且工具合同不变：可以保留一个有限 compatibility window；旧 path 可以保留或在明确同合同时建立非破坏性映射，但不能把不同合同的 tool tree 静默伪装成相同版本。
- 工具合同、Skill/Plugin manifest 或 MCP tool list 变化：旧 session 不作为验收入口；一次 fresh context 是必须的，旧 cache 只保留到退休策略允许的时间。
- 不能承诺所有历史 Codex session 在任意未来都可恢复；这取决于 Codex 是否修复绝对 cache path 持久化。

## 六、项目开发与发布阶段规则

### 阶段 1：PR 开发中

Codex 判断变更是否属于“合同变化”。分支只修改源码和项目文档；任何本机安装测试必须使用隔离 profile。不能把 stable Skill、plugin、marketplace、cache 或 active receipt 当开发输出覆盖。

用户不需要做任何 cache 清理动作，也不需要因为每个文件修改新建 task。

### 阶段 2：PR 就绪

Codex 以最新目标分支做前置审计：工作树、测试、构建、文档、冲突、远端检查。此阶段不做 stable activation，不把“PR 就绪”说成产品已交付。

### 阶段 3：PR 合并后收口

这是本问题的主阶段。只有合并后才按最新 `main` 构建正式 generation：

1. 下载/校验 Release zip 和 `.sha256`。
2. `stage` 安装并 hash verify Skill、plugin、marketplace、cache；只写 staged receipt。
3. 如果合同变化，要求用户打开一次 fresh Codex context；Codex 记录 native tool names、关键调用和 generation。
4. 运行 provider capability canary，生成 generation-matched readiness。
5. `activate` 写 active receipt。
6. 启动自动 retirement reconcile。
7. 通过 health gate 后，按普通用户路径下载、解压、运行 setup/打开/调用一次。

### 阶段 4：产品已交付

只有 active receipt、health `ready`、远端下载包复验、用户入口行为和自动回收器状态都满足规则，才能说“产品已交付”。若旧目录仍被占用但已记为 `blocked_by_process`，产品可以交付，但必须把状态写成“新 generation 已交付，旧 generation 自动延迟回收中”，不能说“缓存已全部清空”。

## 七、PowerShell 长期实现契约

### 统一 helper 的行为

本轮已建立项目级 `Invoke-CodexPraetorNative` helper，并由 provider canary、setup 安装/维护流程复用：

```text
输入：FilePath、Arguments、WorkingDirectory、TimeoutSeconds、Environment
输出：exit_code、stdout、stderr、timed_out、started、duration_ms、command_display
判定：started=false 是启动失败；timed_out=true 是超时；否则 exit_code == 0 才是 native success
```

实现约束：

- `UseShellExecute = false`；stdout/stderr 分别重定向；同时开始读取两条流，避免大输出死锁。
- 不在 native invocation 外层使用 `2>&1`。
- 不使用 `$ErrorActionPreference` 判断 native 成功；它只负责 PowerShell 自身错误。
- 兼容 Windows PowerShell 5.1；PowerShell 7 的 `$PSNativeCommandUseErrorActionPreference` 只能作为环境信息，不作为正确性依赖。
- 统一对输出做 UTF-8/本机编码兼容处理，日志保留 stderr 尾部和完整退出分类。
- 由上层 wrapper 另行检查 marker、worktree 状态、completion、readiness，不把 marker 当作 exit code 替代。

### 必须固定的回归矩阵

| 场景 | 预期 |
|---|---|
| exit 0 + stderr 普通进度 | 通过，保留 stderr |
| exit 0 + stderr warning | 通过但标记 warning |
| exit 1 + stderr | 失败，保留 exit code/stderr |
| exit 1 + stdout marker | 失败，marker 不能掩盖 exit code |
| `git worktree add` 成功并写 stderr | 通过 |
| `git worktree add` 分支冲突 | 失败且分类明确 |
| provider marker 返回且主 repo clean | canary 进入后续 readiness 判断 |
| provider marker 缺失但 exit 0 | 失败，不能把进程成功当能力成功 |
| timeout | 失败，清理 worker/process tree/临时目录并留下证据 |
| Windows PowerShell 5.1 与 PowerShell 7 | 结果一致 |

## 八、当前状态的处理决定

当前不应手动强删 `0.1.1-alpha`，也不应把它移动到新版本路径；那会破坏当前对话的旧引用。当前正确状态记录为：

```text
新 generation：可由 fresh context 验收并继续使用
旧 generation：retired_pending / blocked_by_process
active surface：以 active receipt 和 health gate 为准
人工动作：只在工具合同变化的发布代际打开一次 fresh context
清理动作：由后续自动 retirement reconcile 负责，不由用户反复提醒
```

本轮已把方案落到实现：cache/marketplace publisher 不再直接清理旧代际；退休清单、隔离根目录校验、自动回收器、stable 用户级 Task Scheduler 安装、健康状态摘要、统一 native helper 和 canary 回归测试均已加入。尚未执行的是当前稳定用户环境的实际安装、Codex 重启和真实发布；这些属于合并后的用户授权与发布收口，不在 PR 开发阶段改写。

## 九、最终判断

1. **什么时候做？** 主要在 PR 合并后的发布影响收口阶段做 generation stage、fresh proof、provider readiness、activate 和 retirement reconcile；开发中只做隔离验证。
2. **你什么时候做？** 只在一个发布 generation 的工具合同真正变化时打开一次 fresh context；普通编辑不需要新 task，不需要手动清 cache。
3. **旧东西是否必须清？** 必须进入自动回收生命周期，但不能立即强删。active 只保留一代，旧代际自动登记、自动重试、最终在 Windows 释放占用后删除；平台旧 session 的永久恢复能力是明确边界。
4. **会不会新旧打架？** 通过不可变 generation、active receipt、retirement manifest、health gate 和“回收失败不改 active”避免。新代际先完整落地，旧代际后退休，永不原地覆盖。
5. **第二个问题是否可一劳永逸解决？** 可以。不是加一个 `try/catch`，而是建立全项目 native invocation contract，统一分离 stdout/stderr、exit-code-first、超时/marker/状态分层和跨 PowerShell 版本回归矩阵。

本 PR 的代码验收必须以真实脚本、隔离 profile、退出码、退休清单和测试输出为证据；不能把“脚本曾经尝试删除旧目录”或“stderr 没有内容”当作收口证据。
