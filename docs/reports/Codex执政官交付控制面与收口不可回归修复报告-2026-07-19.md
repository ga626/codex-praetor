# Codex 执政官交付控制面与收口不可回归修复报告

日期：2026-07-19
范围：`0.6.1-alpha` 事故复盘后的开发、构建、发布、安装和收口链路。
目标：让“PR 合并后又发现同一发布版本缺陷、再开小修复 PR”在系统上失去通路。

## 结论

这次问题的根因是发布包复制了仓库里的 `plugin/mcp/dist/server.js`，但候选测试只验证了 `mcp/src` 和 `mcp/dist`。因此源码与 manifest 是 `0.6.1-alpha`，真正被用户启动的 bundled MCP 仍是 `0.6.0-alpha`。这不是收口阶段偶然漏看，而是“候选证明”和“最终用户 artifact”不是同一个对象。

本 PR 把发布链路改成 artifact-first：

```text
源码/版本声明 -> npm test 重建 plugin runtime -> stage -> 最终 zip
-> 解压最终 zip 启动 MCP -> 版本/合同/工具列表验收
-> GitHub Release -> 下载复验 -> 本机 stage/activate/deliver
```

只要 bundled runtime 与源码不一致，候选 CI 就会在合并前失败；只要远端 zip 与候选包不同，远端下载复验就会失败。合并后不再允许为同一版本开“收口修复 PR”。

## 本地考古证据

1. `mcp/package.json` 的原 `test` 只运行 TypeScript 编译和 `mcp/dist/self-test.js`，没有运行 `build:plugin`。
2. `scripts/release/build-codex-praetor-release.ps1` 原先直接复制 `plugin/`，没有先生成 `plugin/mcp/dist`。
3. `scripts/verify/test-public-entry-consistency.ps1` 原先只检查 `mcp/src/server.ts` 和两个 package manifest，不检查最终 bundled runtime。
4. `plugin/mcp/dist/server.js` 在事故时仍报告 `0.6.0-alpha`，而源码和 manifest 已经是 `0.6.1-alpha`。
5. Windows 工作区可能受换行配置影响产生不同原始字节；`.gitattributes` 已声明文本文件使用 LF，发布生成现在以最终 stage 内容计算 generation，并保留精确 commit SHA。

## 已落地的结构性修复

### 1. 单一构建入口

`npm test` 现在执行 `npm run build:plugin`。发布器也强制执行同一入口后才复制 stage，禁止把 `plugin/mcp/dist` 当作人工维护副本。

### 2. 最终 artifact 真实启动验收

新增 `scripts/verify/test-release-artifact-runtime.ps1`。它解压最终 zip，并调用 `mcp/scripts/smoke-plugin-mcp.js` 启动 zip 内的 MCP，检查：

- MCP handshake 的 `serverInfo.version`；
- `runtime_contract.version`；
- 必需工具和只读/写入注解；
- `codex-praetor-release-generation.json` 是否随包存在。

共享 candidate/release workflow 在发布前执行该门禁；远端 Release 下载复验也对下载包执行同一真实启动检查。

### 3. 版本一致性前移

public-entry gate 现在同时检查 `plugin/mcp/dist/server.js`。任何源码、manifest、bundled runtime 的版本分叉都会在 PR 阶段失败。

### 4. generation 与最终 stage 绑定

generation 脚本支持从指定 `ContentRoot` 计算 tree manifest，发布器对最终 stage 计算 generation，而不是先对可能尚未构建的工作树计算。commit SHA 仍来自精确 Git HEAD，便于追溯来源。

## 为什么这次可以避免同类收口尾巴

以前的失败路径是：源码检查通过 -> 发布器复制旧 dist -> GitHub Release 看似成功 -> 本机启动才发现旧版本。现在这条路径在“发布前最终 zip 真实启动”处被截断，不能到达 tag/Release。

以后如果出现 GitHub API 瞬时失败、权限不足、网络中断或本机 Desktop host 没刷新，它们不会被代码伪装成成功：

- tag/draft/Release 尚未创建：修复自动路径后重新走同一 PR 门禁；
- tag/draft/Release 已创建：只重跑原 SHA 的 Actions run；
- Release 已发布但本机 host 未刷新：状态是 `代码已合并，产品未交付/needs_user_action`，等待重启或刷新，不开同版本补丁 PR；
- 本机文件被占用：保留旧代际并进入回收重试队列，不撤销已验证 activation。

这些是 GitHub、Windows 进程和用户会话的外部边界，无法由仓库代码保证“永远不失败”；能保证的是失败不会越过状态机，也不会再制造隐形收口尾巴。

## 仍不能由代码消除的边界

1. GitHub API、Actions runner、网络和权限的瞬时故障。处理：保留 incident、重跑原 SHA，不启动下一次发布影响 PR。
2. Codex Desktop 已运行进程的发现缓存和 MCP host 生命周期。处理：fresh-context proof；没有 proof 只能 `needs_user_action`。
3. Windows 文件占用、杀毒软件锁定和用户 provider 登录。处理：可重试状态、旧代际保留、明确下一动作。

## PR 前验收记录

- `npm test --prefix .\\mcp`：通过，并重建 bundled MCP。
- `test-public-entry-consistency.ps1 -SkipRemoteRelease`：通过。
- 发布器构建 `0.6.1-alpha`：通过，最终 stage MCP smoke 通过。
- `test-release-artifact-runtime.ps1`：通过，解压 zip 后真实启动并暴露 `0.6.1-alpha`。

## 规则结论

项目规则新增两条硬约束：发布影响 PR 必须通过最终 artifact 真实启动验收；合并后的失败必须进入 release incident 或 `needs_user_action`，禁止为同一版本开收口修复 PR。全局规则候选只写入规则管家报告，不直接改全局 `AGENTS.md`。
