# Codex 执政官里程碑 A 真实派工测试：任务治理可用性复盘与整改报告

日期：2026-07-17
审计对象：任务“拆分并推进里程碑 A”、计划 `milestone-a-0-3-1-20260717`、其 job/worktree/收据、当前分支 `codex/operational-truth-0-3-1`。
本轮边界：只重写本报告。未修改产品代码、stable、Release、GitHub 设置、分支、作业收据或 worktree；不删除任何残留物。

## 一、先讲结论

这次测试证明 Codex Praetor **已经是一个能派发外部 worker 的工具**，但还不是一个让用户可以放心交付大任务的**任务治理系统**。

它做对了三件事：

1. 用户只给了一句自然语言目标，Codex 能读路线图、建立本地计划、创建隔离 worktree，并真实调用 CodeBuddy 和 Qoder。
2. Codex 没有把所有失败都当成功。它人工识别出 CodeBuddy 超轮数、Qoder 首版缺少重复触发、工具调用被拒绝，并拒绝其中几个结果。
3. 安全边界基本守住：stable 没被覆盖，worker 改动留在隔离 worktree，未通过的产物没有被盲目合入。

但用户真正关心的五件事，目前都没有形成系统能力：

| 用户问题 | 本次事实 | 结论 |
|---|---|---|
| 分得准不准 | 初始维护包同时要求 canonical definition、双 backend、安装回读、对称卸载、负向测试；第一次 CodeBuddy 8 turns 耗尽，第一次 Qoder 自己承认不满足 XML 语义 | 有人工拆包，没有基于复杂度/依赖/证据需求的任务设计器 |
| worker 选得对不对 | 两个 provider 都先通过 canary；实际选择只留下 tier/model/价格备注，没有留下“为何这个任务选它、替代方案和预算” | readiness 证明“能跑”，不证明“适合且划算” |
| 监督有没有效 | 人工监督最终发现了关键问题；系统收据仍把 `Max turns`、tool denied 写为 `completed` | 人工有效，产品监督无效或不完整 |
| 失败有没有学到 | Qoder 首版失败后，下一包确实针对 `PT15M/P3650D` 收窄；但计划把它记成新 task，没有 attempt 关系、失败标签、provider 画像或复用策略 | 临场学习存在，持久学习不存在 |
| 用户要不要一直盯 | 本次全部采用 blocking，5 个开发 job 合计约 21.2 分钟；`notify_enabled=false`，无预算阈值、无“需要你决定”的状态 | 用户不用逐秒盯日志，但必须依赖 Codex 临场盯守，失败后也没有清晰的决策交接 |

**整体判断：当前产品价值更接近“受控的外部 CLI 调用器 + 人工监督习惯”，而不是“能把目标可靠转化为少量高价值 worker 工作、自动收敛失败并只在必要时打扰用户的治理系统”。**

下一阶段的第一优先级不应再是新增 provider，而是补上“任务定义 -> 选择 -> 执行 -> 验收 -> 学习 -> 用户决策”的闭环。上次报告中的终态/取消/并发问题仍需修，但本报告把它们放回用户体验的根因：**系统不知道一项工作是否真的创造了价值，也不知道下一次该怎样更聪明地做。**

## 二、审计方法与证据边界

本报告重新走完整链路，以磁盘事实为每项发现的依据：

1. 原任务提示词、截图、`plan.json` 的 8 个 task 和 22 个事件。
2. 7 个真实作业的 `job.json`、`task-contract.json`、`completion.json`、stdout/stderr、worker 参数和 worktree 差异。
3. 当前 skill、编排策略、tier 配置、readiness 记录、dispatch/plan 脚本与路线图。
4. 独立的 Git/远端核对：分支仍只比 `origin/main` 多 `163ce53`，未推送、无 PR、无 CI 证据。
5. KnowledgeRadar 感知层：已完成 `health_check(summary)`、`get_capabilities(summary)`、`kr_research(deep_route)`，并抽取任务可观测性、重试和人机中断相关资料。研究收口时 KR 没有 active/stale 后台任务。

成本结论特别说明：本机没有读取 provider 账户、余额或 token 页面；CodeBuddy/Qoder 本次 text 输出没有返回实际 input/output token 或账单。因此本报告只统计可验证的 wall time、作业次数、输出大小和“是否被接受”，绝不把 `0.1x` 或“0 credits 观察”误写成实际成本。

## 三、完整链路重放：每一环到底发生了什么

用户原始输入只有一句：

```text
看一下 D:\Projects\CodexPraetor\docs\reports\Codex执政官0.3.0交付后现状审计与下一阶段路线图-2026-07-17.md，把里面的里程碑 A 拆分给其他 agent 完成，并按照全局规则和项目规则推进到 PR 前。
```

这是一条合格的真实用户输入：它没有替系统预拆任务，也没有指定 provider 或微观步骤。治理系统应该负责把“里程碑 A”转换为可验证、可预算、可恢复的工作单元。

| 环节 | 实际行为与磁盘证据 | 做得好的地方 | 更深的治理问题 |
|---|---|---|---|
| 1. 目标理解 | `plan.json` 建出 maintenance、dynamic health、runtime inventory、supply chain/docs 等 8 个 task | 宏观范围没有遗漏到只剩一个脚本 | plan 没有记录目标价值、风险、用户影响、Codex 保留项、完成定义或关键路径；用户无法知道“为什么先做这个” |
| 2. 拆包 | `maintenance_adapter` acceptance 同时含双 backend、回读、卸载、负向测试，允许 `scripts/install`、`scripts/maintenance`、`scripts/verify` | 有允许/禁止路径和验收文字 | 一个包跨四个行为面，且涉及 Windows XML 语义；违背本项目 skill 的“一项可观察验证目标”原则。它不是不可派，但应先被标成高复杂度/高证据依赖 |
| 3. provider 准入 | CodeBuddy code-change canary 在 17:33 通过；Qoder code-change canary 在 18:01 通过；readiness 有 CLI hash、model、permission、expiry | 没跳过 readiness，避免了未知 provider 直接改代码 | readiness 只是二元资格。`codex-praetor-readiness.json` 没有成功率、任务类型适配度、平均用时、失败模式、费用或置信度；不能回答“为什么让这个 provider 做这个包” |

| 5. 执行 | 7 个 job 都是 `run_mode=blocking`；真实 worker 在隔离 worktree 中执行 | 没有实时轮询、没有让用户逐步授权 worker | 多步骤计划没有使用 background + 单次完成通知；job 元数据的 `notify_enabled=false`。用户看不到“已处理多久、消耗多少、下一步是否需要决定”的简洁状态 |

| 7. 修复/重试 | Qoder 首版后新包明确要求 `PT15M/P3650D`；随后又出现卸载专项和 dynamic health 包 | 有针对根因的 prompt refinement，说明 Codex 的临场判断有效 | 同一逻辑工作被拆成多个平行 task，没有 attempt 链、预算累计、最大重试数或“应换 provider/应由 Codex 接手”的规则。repair 尚未验收时又启动卸载任务，依赖门禁失效 |
| 8. 集成 | `163ce53` 合入维护候选；动态 health worktree 仍有未提交改动 | 只把一个人工接受的结果放入主分支，未盲目搬运其余 worktree | 接受的不是完整业务成果：维护测试是 mock，后续卸载专项失败，dynamic health 未完成。系统没有“逻辑任务整体验收”而只有“某次 patch 进入分支” |
| 9. 结束/交接 | 当前有 8 个 plan task，其中 maintenance repair 仅一个 completed，dynamic health/runtime inventory/supply chain docs 未完成或失败；分支未推送、无 PR | 线程最终承认未到 PR 前 | 没有面向用户的治理摘要：花了多少、得到了什么、哪些失败是可重试、哪些需要用户决定、何时应停止投入 |

## 四、量化事实：本次投入换来了什么

以下只使用 job 收据时间，不推测模型 token 或账户账单：

| 类型 | 数量 | 可验证耗时/结果 |
|---|---:|---|
| provider canary | 2 | CodeBuddy 约 20 秒、Qoder 约 17 秒，证明指定 tuple 当时可启动并完成小探针 |
| 开发 job | 5 | 合计约 1,274 秒，即约 21.2 分钟 wall time |
| 产生有内容 stdout 的开发 job | 3 | Qoder 首版 1,892 bytes、repair 1,811 bytes、dynamic health 864 bytes |
| 空/拒绝/超轮数开发 job | 2 | CodeBuddy 空 stdout + `Max turns`；Qoder `Repeated tool call was denied` |
| 进入主分支的候选结果 | 1 | `163ce53`，2 个文件、379 行变动；未等于完整/独立验收通过 |
| 完整里程碑 A 结果 | 0 | dynamic health、inventory、供应链、文档、PR 验收均未完成 |
| 实际 token/金额记录 | 0 | Qoder/CodeBuddy completion 没有 usage/cost 字段；只有 MiMo 解析器具备该类字段 |

这不能说明“21 分钟浪费了”。它产生了真实失败证据和一个候选修复，研究价值很高；但对于一个用户期待“推进到 PR 前”的任务，系统没有给出投入上限或收益信号，所以无法回答“是否还值得继续给 Qoder 再派一包”。

### 4.1 现有成本治理为什么不够

当前 tier 配置已经有正确的**事前护栏**：固定模型、时段、reasoning effort、禁止 auto、需要明确同意才可用强模型。它避免了明显的失控。

缺的是**事中/事后经济账**：

- 每个 task contract 没有 planned token/credits/金额上限，也没有 max turns、context window、最大输出等预算字段的稳定记录。
- completion 对 CodeBuddy/Qoder 没有实际 token、provider cost、重试成本、总 wall time、接受产出率。
- `price_note` 是静态说明文字，不是该 job 的真实价格。
- 失败不会消耗一个可见的 logical-task budget；因此连续“再试一次”在系统里没有止损点。

[OpenTelemetry 的 GenAI 语义约定](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-spans/)把模型、输入/输出 token 等作为标准可观测属性；[OpenAI Agents tracing](https://openai.github.io/openai-agents-python/tracing/)也把 agent/LLM 执行记录为可追踪的 spans。它们不要求本项目引入任何框架，但证明“模型选择、耗时、usage、结果”应属于同一条任务记录，而不是散落在日志和人的记忆里。

## 五、从“好不好用”看发现的问题

优先级按对用户省心、任务成功率和成本可控性的影响排序，而非按代码修改难度排序。

### G0：系统没有“逻辑任务的完成定义”，只能管理进程和 patch

**证据**：计划有 `maintenance_adapter`、`maintenance_core_qoder`、`maintenance_adapter_repair`、`maintenance_uninstall_failure` 四个相近 task；都没有 `depends_on`。`maintenance_adapter_repair` 最终 completed，但同一目标的卸载专项 failed，主分支仍已有候选提交。

**为什么不好用**：用户问“维护任务做完了吗”，系统只能回答某个 job 退出、某个 patch 提交，不能回答业务验收是否闭环。失败的专项不会阻止一个不完整的“已完成”感。

**改进**：plan 首先创建一个 logical task，例如“维护任务可验证安装/卸载”，内部才允许 attempts。它必须有：完成定义、必须场景、依赖、写集合、预算、最终 verdict。任一必测场景未通过，logical task 不能为 accepted。

### G0：任务包没有复杂度和证据需求判定，导致“便宜 worker 做了需要前置研究的设计题”

**证据**：首个 CodeBuddy 包同时跨安装、XML fallback、回读验证、卸载、负向测试；worker 合同禁止外部网络研究。Qoder 首版明确指出 XML 没有 repetition，说明它未拿到足够的 Windows Scheduler 约束。项目自身编排策略要求单个 packet 有一个可观察验证目标，但该包实际有多个。

**为什么不好用**：用户给短目标是正确体验；不应该由用户预先知道“这个包需要先做 Microsoft XML 研究”。如果系统不判定证据缺口，就会把架构性不确定性伪装成普通编码任务，反复消耗 worker。

**改进**：派发前加入 task-shaping gate：

```text
目标 -> 影响面 -> 写集合 -> 已知事实/未知事实 -> 是否需 Codex 先研究
     -> 复杂度/可逆性 -> 一个验收目标 -> 推荐 packet
```

当“平台语义未知、接口合同变化、跨模块设计”出现时，Codex 先做只读研究并把结论写入 packet；worker 只实现明确方案。首包应拆成“生成并回读等价 task definition”与“卸载失败不删脚本”，而不是一次承包所有行为。

### G0：provider 选择只有资格门禁，没有任务适配和经济决策

**证据**：readiness 记录 provider、CLI hash、model、permission、task kind、passed/expires；job 则记录 tier/model/静态 `price_note`。两者都没有 task 类型成功率、历史失败、预计耗时、实际 usage、选它的原因或替代方案。CodeBuddy canary 成功后紧接着在更复杂维护包上超轮数，正说明 canary 不等于适配。

**为什么不好用**：用户无法判断工具是在“聪明地节省 token”，还是在“先试免费/便宜的，失败再换”。后一种策略可以接受，但必须有止损和透明理由。

**改进**：把 provider selection 拆成两层：

1. **资格**：现有 readiness，回答能否安全执行。
2. **适配**：按 `task_class + risk + evidence_need + expected_scope` 输出候选排序、理由、置信度和预算。初期不需要机器学习，只需本地统计和规则：同类任务近期出现 `max_turns`，默认不再给该 tier 发送同等或更大包；先缩包、提高 budget，或改由 Codex 接手。

每次选择必须写入 job：`selection_reason`、`alternatives_considered`、`expected_cost_band`、`stop_loss`。未知成本必须标 `unknown`，不能沿用静态备注。

### G1：监督是“人工读日志”，不是“只在必要时把结论推给人”

**证据**：skill 已明确要求 exit 0 后检查 stdout/stderr/diff/tests；本次 Codex 的确做了。但 watcher 只识别少量错误模式，plan 对 `Max turns` 和 `Repeated tool call was denied` 仍先登记为 completed/awaiting verification。全部 job 是 blocking，且 `notify_enabled=false`。

**为什么不好用**：用户不应该盯每个 job；Codex 也不该反复从长日志重新推理。当前系统把真正的治理工作留给单次会话的记忆，任务一中断，监督质量就下降。

**改进**：把检查变成三种低噪声 checkpoint，而非实时监控：

| checkpoint | 自动完成 | 只有何时打扰用户 |
|---|---|---|
| 派发前 | 展示拆分、provider、预算、风险；Codex 自检范围 | 高风险/超过用户预算/需要改变产品合同 |
| worker 后 | 解析报告、diff、测试、失败类；给出 accept/retry/reroute/stop 建议 | 需要在成本、范围或不可逆动作间选择 |
| 逻辑任务后 | 汇总价值、累计成本、剩余风险、是否继续 | 已达到止损或目标需要改写 |

用户收到的应是一条“需要决定什么、各方案代价、默认建议”，不是 stdout。LangGraph 的 [interrupt 文档](https://docs.langchain.com/oss/python/langgraph/interrupts)展示了可持久化 checkpoint 和显式、结构化的人机暂停；本项目应借鉴这个交互边界，不引入其运行时。

### G1：失败虽被临场修正，但没有变成下一次的资产

**证据**：首版 Qoder 后，repair acceptance 精确加入 `PT15M/P3650D`，说明 Codex 产生了正确新知识；但 plan 中没有 attempt 关系、failure taxonomy、root cause、建议动作，tier/readiness 文件也没有更新。下次同类任务仍可能再发出同样宽的包。

**为什么不好用**：用户会感觉工具“每次都从头犯错”。失败的价值没有被沉淀，低价 worker 的试错成本也就无法递减。

**改进**：新增项目本地、非敏感的 `governance/outcomes` 记录，每次 attempt 只写：

```text
task_class, provider/tier/model, packet_shape, outcome,
failure_class, root_cause, observed_cost/time, action_taken,
reusable_rule, evidence_paths, expires_at
```

规则要有边界和过期时间，不能把一次 CodeBuddy 超轮数永久归因于 provider。失败分类至少区分：任务过大、前置事实缺失、工具拒绝、环境/权限、产物不满足、监督/集成缺失。后续 router 只能读取聚合结论，不读取用户代码、token 或账户资料。

[Temporal 的重试策略说明](https://docs.temporal.io/encyclopedia/retry-policies)强调可配置的重试边界和非可重试失败。这里同样不需要引入 Temporal；需要的是“什么失败可以缩包重试，什么失败必须换策略或停止”的显式政策。

### G1：计划是收据，不是用户可用的决策面

**证据**：`plan.json` 有 8 个 task、22 个事件，但每个 task 的 summary 主要是 `worker_status`/`exit_code`；没有 owner、业务优先级、预计/实际成本、下一决策、风险、attempt history。task contract 只有 job id 和范围，没有 task id/plan id 自描述字段。现有 `next_ready` 只能按依赖展示，不解释价值。

**为什么不好用**：计划在崩溃恢复时有用，但对用户回答不了“现在进度如何、卡在哪里、需要我做什么、继续值不值”。

**改进**：不做花哨仪表盘，先让 `codex_praetor_job_timeline` 和 plan summary 返回固定中文结构：

```text
目标：...
已交付：...
当前阻断：...
已花费：已知/未知成本、用时、attempt 数
下一动作：自动执行 / 需要 Codex 验收 / 需要用户决策
默认建议与停止条件：...
```

### G2：blocking 模式让“异步 worker”在体验上仍像用户等待同一段对话

**证据**：7 个 job 都写 `run_mode=blocking`。skill 明确提供 background、completion 收据和可选回原线程通知，但本次未使用；job 元数据也记录 `notify_enabled=false`。

**为什么不好用**：短任务 blocking 很合理。本次是多阶段 PR，连续 blocking 让 Codex 必须维持长时间控制循环，用户在结束后才知道整体失败；这不等于“实时监督”，但也没有兑现“用户不必一直盯”。

**改进**：按逻辑任务而非单个 shell 调用选择模式：小于一个明确验收周期的 packet 用 blocking；可独立的中等 packet 用 background + 一次语义 completion；跨多小时/预算较高任务在派发前建立 checkpoint 和预算阈值。通知只在 `needs_decision`、`budget_exceeded`、`accepted`、`blocked` 时发，避免噪声。

### G2：缺少可解释的停止条件，导致“再派一包”只能靠感觉

**证据**：五个开发 job 中，CodeBuddy 首包失败，Qoder 首包失败，repair 被接受但未完成整体验收，卸载专项失败，dynamic health 被取消。plan 没有记录累计 attempt/时间上限或 escalation 规则；task timeout 均是 1,200 秒，但那是单进程上限，不是业务止损。

**为什么不好用**：失败后的继续投入可能是正确的，但用户和 Codex都无法量化何时从“正常迭代”变成“应该停止、重新设计或由 Codex 接手”。

**改进**：每个 logical task 需要 `attempt_budget`，例如初期按风险设 2 次 cheap attempt 或 20 分钟 wall time。达到阈值时 router 不再自动重试，只输出三种选择：缩小范围重试、升级 provider/预算、Codex 接手并先研究。用户只在跨越默认预算或改变风险时被问一次。

## 六、目标形态：不做通用平台，做可用的“治理闭环”

目标不是引入多个 agent 互相讨论，也不是让用户管理复杂看板。目标是让用户仍然说一句自然语言，系统在后台完成大部分判断，并在真正需要授权/取舍时给出短而准确的问题。

```text
用户目标
  -> Codex 形成任务画像和工作拆分
  -> 资格 + 适配 + 预算 选择 worker
  -> worker 执行一次完整 packet
  -> 自动检查产物与语义结果
  -> Codex 验收 logical task
  -> 写入 outcome，调整下一次选择
  -> 只在预算/风险/范围需要取舍时询问用户
```

### 6.1 最小数据合同

无需新服务，项目本地 JSON 即可先落地；但字段必须完整且可重建。

| 对象 | 最少新增内容 |
|---|---|
| logical task | 用户价值、owner、write set、依赖、完成定义、证据需求、风险、预算、停止条件 |
| attempt | base commit、packet 版本、选择理由、候选 tier、预计成本、实际 usage/未知、wall time、产物、失败类、结论 |
| provider profile | 按 task class 聚合的近期成功率、acceptance rate、典型失败、耗时/成本区间、样本量和过期时间 |
| 用户决策 | 决策问题、可选项、默认建议、影响、恢复方式；没有决策时保持安静 |
| 最终摘要 | 已交付、未交付、总 attempt、已知/未知成本、剩余风险、下一动作 |

实际 usage 不能取得时，字段必须是 `unknown` 并写明 provider 没有返回，而不是推算或展示为零。MiMo 已有 JSON event parser 的 token/cost 字段，可作为同一 completion schema 的实现参考；Qoder/CodeBuddy 需要先探测官方 CLI 是否能提供结构化 usage，不能从账号数据库取数。

### 6.2 推荐策略，先规则后统计

第一版不应假装有足够数据训练智能路由。采用透明规则：

1. 用现有 readiness 排除不能安全运行的 tuple。
2. 根据 task 画像排除不合适的选择：需要外部事实但 packet 未提供事实、共享写集合、合同变化、高风险不可逆操作。
3. 对剩余选择按用户策略排序：`最省`、`均衡`、`最快`；默认由项目/用户配置决定。
4. 样本少时明确标“证据不足”，优先做廉价 canary 或小 packet，不把它叫最佳选择。
5. 每个 attempt 更新本地 outcome 聚合；只在样本量和时间窗口足够时调整默认次序。

这样“学习”是可解释、可撤销、可审计的，而不是不透明地记住某个 provider “不行”。

### 6.3 本次任务按新机制应如何运行

| 阶段 | 应有动作 | 用户是否需要盯 |
|---|---|---|
| 规划 | Codex 先声明：维护任务、动态 health、inventory、供应链/docs；维护任务需要 Task Scheduler 外部事实，先研究 | 否，除非范围/预算超过默认 |
| 维护任务 | 拆成 definition round-trip 和 uninstall safety 两个 logical subtask；明确它们串行 | 否 |
| 选择 | 记录 CodeBuddy/Qoder 的候选与理由；复杂度高于 CodeBuddy 8-turn 基线时先缩包或提高受控预算 | 否 |
| 首次失败 | `max_turns` 归类为 packet-too-large 或 budget-exhausted，不写 completed；自动建议缩包/改派 | 否，除非要提高预算 |
| repair | Qoder repair 作为同一个 logical task 的 attempt 2，必须等待 definition 验收后才开始卸载任务 | 否 |
| 结束 | 输出：维护任务是否通过所有场景、花费/未知成本、dynamic health 是否可派、剩余任务和预计下一步 | 只在预算或风险选择时需要 |

## 七、按“好不好用”排定整改优先级

### P0：先让用户得到可信的“做成了没有、还值不值得做”

1. 引入 logical task/attempt/outcome 模型，禁止用 process `completed` 表示业务完成。
2. 为每个 task 补完成定义、写集合、依赖、预算、停止条件；同一工作重试必须挂在同一 task。
3. completion 自动汇总 worker report、diff、测试、scope check 和 Codex verdict，生成中文简报。
4. 把 `Max turns`、tool denied、空输出、无 diff、验收不通过变成结构化失败；明确下一步是重试、缩包、改派、研究还是停止。

**验收标准**：用户问“这一项完成了吗、为什么继续、已经花了什么”，不需要打开日志即可得到真实答案；任一必测场景失败不会出现“完成”的假象。

### P1：再让 worker 选择和失败处理越来越划算

1. job 记录 `selection_reason`、候选、预期成本区间、实际 usage/unknown、wall time 和 stop-loss。
2. 对 CodeBuddy/Qoder 补结构化 usage 探测；拿不到时显式 `unknown`。
3. 建立 task-class outcome 聚合和有过期时间的规则；开始只做建议，不自动改变高风险路由。
4. task-shaping gate 识别“先研究再派发”的设计类任务。

**验收标准**：第二次遇到同类 Scheduler 任务，系统能引用本地 outcome，自动避免已证实不合适的包大小/路线，并解释理由。

### P1：降低用户和 Codex 的注意力负担

1. 多阶段计划默认采用 background + 语义 completion；不是实时流式监督。
2. 引入 `needs_decision`：只在成本、范围、不可逆动作、外部授权或连续失败跨阈值时停下。
3. 固定输出一页治理摘要，而不是要求用户读 plan JSON 或长日志。

**验收标准**：用户提交一个大目标后，可以离开；回来只需看到结果或一条明确决策，而不是“请继续盯着外部 agent”。

### P2：随后改善运营质量

1. 在 MCP 中提供 compact plan/outcome/timeline 查询和按 task class 的 provider 画像。
2. 对 stale worktree/job、测试临时目录和 retention 给出 runtime inventory，不自动删除 dirty 证据。
3. 再把供应链、文档和 release 验收接入同一 logical task 摘要，避免“代码 patch 已有”被误读成“产品已交付”。
