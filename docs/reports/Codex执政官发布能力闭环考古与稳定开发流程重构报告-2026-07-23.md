# Codex 执政官发布能力闭环考古与稳定开发流程重构报告

> 日期：2026-07-23
> 范围：只读考古、发布链路检查与多源外部调研。本文不修改产品代码、运行态、账号、认证、GitHub 设置或现有规则。
> 要回答的问题：过去的修复是否只是局部补丁；如何一次性清掉遗留问题；以及如何把以后每次开发变成稳定、自动、不会不断产生收口尾巴的流程。

## 先说结论

过去的修复不应简单删除或推倒重来。它们大多修复了真实故障：数据确实曾漏进安装包、任务类别确实曾在派工链路丢失、包内脚本确实曾找不到依赖。问题在于，这些修复都是按“这次发现的故障”分别加的，尚未成为一个能覆盖**所有公开能力**的统一机制。

因此，过去的补丁有两种状态：

| 类型 | 判断 | 正确处理 |
| --- | --- | --- |
| 已修复的真实故障 | 保留，不能回退。 | 把它改写为能力矩阵中的长期回归用例，而非散落的特例。 |
| 只覆盖当前症状的检查 | 不能再单独当作“已彻底治理”的证据。 | 纳入统一能力合同，由同一个生成式门禁执行。 |
| 仍未完成闭环的能力 | 当前应视为产品缺口，不应继续靠收口发现。 | 在下一次统一治理 PR 内设计、实现并用最终安装包验收。 |

当前最明确的未闭环能力是“真实评测准备”。源码脚本 `scripts/evaluation/initialize-codex-praetor-evaluation.ps1` 可以把评测清单变成带任务族、权限、检查、预算与验收标准的计划；最终稳定插件却没有该脚本，也没有等价 MCP 工具。已安装产品只能通过 `codex_praetor_evaluation_suite` **查看**清单，不能把清单变成可累计能力画像的真实评测计划。通用 `codex_praetor_plan` 会产生 `unclassified` 任务，不能替代它。

这解释了为什么前面的“大矩阵”仍会漏：它实际验证的是“包能启动、工具能列出、关键数据能读”，不是“用户是否能从公开入口完成每一项承诺”。

解决方案不是要求每次人工重做一轮考古，而是建立一份机器可读的**发布能力清单**，让它驱动日常自动门禁。一次性的深度考古负责把现有能力和历史事故全部入账；以后绝大多数 PR 只需自动跑受影响能力，发布影响 PR 自动跑最终包的完整确定性矩阵。只有新增能力、改安装结构、改 MCP/Skill 入口或改 provider 合同时，才扩充矩阵，而不是重新发明流程。

## 1. 先把三个概念分开

过去容易把下面三件事混为一谈，导致“绿了但用户不能用”。

| 层 | 人话 | 现在已有的强项 | 仍缺的东西 |
| --- | --- | --- | --- |
| 代码正确 | 源码中的函数、脚本、单元测试能工作。 | TypeScript、PowerShell、合同与任务账本已有较多检查。 | 源码能工作不等于安装包有入口。 |
| 交付物正确 | zip、版本、哈希、tag、MCP runtime 都是同一个产物。 | 已有确定性构建、SHA、generation、远端下载复验和不可变 Release 路线。 | 产物身份正确不自动证明每项功能能完成。 |
| 用户能力完成 | 安装后的用户能从文档、Skill 或 MCP 入口，把一件事走到结果。 | 安装边界和 host 刷新状态已被明确区分。 | 尚无覆盖全部能力的统一执行矩阵。 |

可以把它理解为：以前已经很会检查“快递单号、箱子封条和箱内零件”，但没有逐项检查“用户按说明拿到零件后，能否装出每个承诺过的功能”。

## 2. 本次考古的方法与证据边界

### 2.1 本地考古

本次检查了：

- `main` 当前基线和 PR #39 至 #47 的合并记录；
- 四个能力 PR 的实现范围，以及 #43 至 #47 的后续修复；
- `Release On Main`、共享发布流水线、构建器、最终 zip 运行时验收、远端下载复验、安装边界 smoke 与 mutation 测试；
- `plugin/`、`mcp/`、`scripts/`、`skill/` 的职责边界；
- 当前 22 个 MCP 工具、Skill 入口、公开文档与评测准备脚本。

没有修改代码、没有写入 provider 状态、没有碰认证材料，也没有尝试真实派工。

### 2.2 外部调研

先通过 KnowledgeRadar 做能力握手和深度研究路线，再使用官方文档与开源实现交叉验证。结论不把“供应链来源证明”误说成“功能质量保证”，也不把 MCP Inspector 这种调试工具误说成完整发布流程。

外部研究得到四条可直接采用的原则：

1. GitHub 与 SLSA 都强调：artifact 的来源证明必须绑定到具体产物，并且需要实际验证；但它只证明“从哪里构建”，不证明“功能是否完成”。
2. GitHub 的 immutable release 建议在公开前先把所有资产装入 draft；一旦发布，tag 与资产不应被补改。因此，发现产品能力缺口时，应在合并前拦住，而不是依赖发布后的补发。
3. MCP 官方 Inspector 的定位就是对真实 MCP server 做测试与调试。这支持本项目继续用协议级调用验证最终 bundled MCP，而不是只跑源码函数。
4. 合同测试的核心不是堆更多 mock，而是把消费者真正依赖的行为写成可执行合同。对应到本项目，就是把“用户从入口做到结果”的场景写成合同，而不只校验工具名或文件存在。

来源、提取时间、关键原文与本地证据位置见同名 `.evidence.json`。

## 3. 过去到底发生了什么

### 3.1 四个原始发布影响 PR 完成了什么

| PR | 原始目标 | 实际交付的核心 | 当时尚未被统一检查的层 |
| --- | --- | --- | --- |
| #39 | 能力画像与 adapter 合同 | 将 provider tuple、任务族和证据投影纳入产品。 | 新数据与入口是否完整进入最终用户运行时。 |
| #40 | 真实任务评测 | 增加版本化评测清单、准备器和评测合同。 | 准备器是否是已安装插件可调用的能力。 |
| #41 | 可解释路由与恢复 | 加入解释性路由和失败恢复逻辑。 | 它依赖的所有画像/评测事实能否由已安装产品产生。 |
| #42 | provider 运营 | 增加 adapter 合同和 onboarding 展示。 | 合同/清单是否随 plugin 而不是只随仓库或 zip 根目录安装。 |

四个方向本身没有错，顺序也合理。问题不是“做了不该做的功能”，而是发布时把“源码、zip 根目录、plugin、Stable marketplace、host runtime”当作同一层，实际它们是不同层。

### 3.2 后续修复的真实含义

| PR | 暴露的事实 | 修复是否有效 | 为什么仍不足以终结循环 |
| --- | --- | --- | --- |
| #43 / `0.9.4` | provider 运营数据不在安装插件里。 | 有效：现有 bundled MCP smoke 现在会读取 plugin 中的 provider 数据。 | 检查是为这一份数据补的，不会自动发现下一份能力的入口缺失。 |
| #44 / `0.9.5` | 评测清单不在已安装 runtime 中。 | 有效：`evaluation-suite.json` 现在会随 plugin 安装，并有删除文件的 mutation test。 | 证明“能读清单”，没有证明“能完成评测准备”。 |
| #45 / `0.9.6` | 固定测试任务丢失 `task_kind`，被错当只读任务。 | 有效：计划合同保留任务类别和最小权限。 | 是针对任务合同字段的回归，没有覆盖产品入口与安装路径。 |
| #46 / `0.9.8` | MiMo 因外部风控不适合作为公开可用 provider。 | 属于产品范围收敛，不是掩盖错误：公开产品现在只支持 Qoder、CodeBuddy。 | 它不能替代对两个保留 provider 的完整用户路径验证。 |
| #47 / `0.9.9` | 插件内 canary 脚本第一次执行时找不到哈希 helper。 | 有效：最终 zip 现在实际启动包内 canary，检查受控 repo 边界。 | 它只证明一个启动边界，未枚举全部可公开完成的任务。 |

结论：这些不是“无效补丁”，但它们是五个分散的经验测试。应把它们保留并升级为**同一能力模型生成的历史故障注入**，而不是继续每次手写一条新例外。

### 3.3 当前可证明的结构性缺口

当前发布构建器会把整个 `scripts/` 目录放进 zip 根目录，但稳定安装只安装 `plugin/`。这是合理的分层：发布维护脚本不应自然变成插件运行时能力。

问题在于评测准备器在产品设计中被描述为真实评测闭环的一部分，却只位于 zip 根目录 `scripts/evaluation/`：

```text
源码 / 发布 zip 根目录：initialize-codex-praetor-evaluation.ps1
稳定 plugin/skills/.../scripts：没有该脚本
MCP：只有 evaluation_suite（查看），没有 prepare_evaluation（准备）
通用 plan：可以建计划，但缺少评测任务所需分类与合同字段
```

这不是“少一个方便命令”。它会使“评测 -> 派工 -> Codex 验收 -> 能力画像积累”的设计在安装产品中断在第一步。

## 4. 现有流程已经做对了什么

不能因为仍有缺口，就把已有治理说成无用。当前基础实际上已经比普通小项目严格得多：

- PR CI 与 main 发布复用同一共享 pipeline；main 发布从合并 SHA 构建。
- 构建是确定性的，zip 具备 hash、generation manifest、版本与 runtime contract 对账。
- 最终包会启动 bundled MCP，检查工具集、版本、合同 SHA、评测数据和 provider 运营数据。
- mutation 测试会删除 runtime contract 或关键数据，确保这些已知事故不能静默通过。
- 远端下载会对账 hash、tag commit、release notes 和 bundled MCP；本机激活与 host 刷新有明确的 `needs_install` / `needs_host_restart` / `needs_canary` 状态。
- 已明确禁止同版本手工补发；公开失败应被标为 release incident，并重跑原 SHA。

因此，下一步不是换掉发布系统，也不是引入一套大型 CI 平台。缺的只是把上面已经可靠的产物链，连接到“能力是否真能由用户完成”的语义层。

## 5. 现有流程的系统性问题清单

| 问题 | 为什么会造成循环 | 严重度 | 建议处理 |
| --- | --- | --- | --- |
| 没有统一能力清单 | 公开承诺散在 README、release notes、MCP、Skill、脚本和历史报告里，CI 无从知道该验什么。 | 高 | 建立单一能力注册表。 |
| runtime contract 只列工具名 | “工具存在”不等于它能完成用户动作。 | 高 | 每个能力声明入口、依赖、场景、成功谓词和 mutation。 |
| mutation 是手工列举 | 每个 incident 后新增一条，天然滞后。 | 高 | 从能力注册表生成或驱动 mutation，不再靠记忆。 |
| zip 根目录与 stable plugin 职责未显式建模 | 根目录有脚本会让开发者误以为已交付，实际 marketplace 只安装 plugin。 | 高 | 在注册表中为每项能力声明“交付通道”：release-only、installed-plugin 或 developer-only。 |
| 公共入口检查主要是文本一致性 | 检查版本、URL、关键词，无法证明路径能完成动作。 | 高 | 改成从最终 zip 启动的场景测试。 |
| MCP smoke 检查工具集和少量调用 | 能抓出启动、数据和合同漂移，但没有覆盖每个承诺的终点。 | 高 | 保留它作为协议层 gate，并在其上增加能力场景层。 |
| 已知事故未形成统一回归资产 | 修复一轮后，知识留在特定脚本和 release note 中。 | 中 | 每个 incident 映射一个能力 ID 和失败注入，不再单独维护散点。 |
| 真实 provider 与确定性 CI 混在概念上 | CI 不应假装拥有用户账号；但没有 provider 证据又不能宣称可用。 | 中 | 分层：确定性包矩阵每次跑；真实 provider 只在受影响合同改变时用隔离环境验证，并记录明确状态。 |
| host 无法热替换运行中的 MCP | 这由宿主进程模型决定，不是脚本小修能解决。 | 中，外部边界 | 把一次受支持 host 刷新定义为交付后的唯一人工边界；不能将其伪装成可自动完成。 |
| 用户功能、维护功能和开发功能混在文档语义中 | 可能把“工程师可从源码运行”误当“安装用户可用”。 | 中 | 三类能力必须显式标识，公开文档只承诺安装用户可完成的项。 |

## 6. 新的核心机制：发布能力注册表

建议在下一次统一治理 PR 中新增一个机器可读注册表，例如 `config/public-capabilities.json`。它不是又一份说明文档，而是 CI、文档检查、zip 测试和 mutation 的共同输入。

每个条目至少包含：

```json
{
  "id": "evaluation.prepare",
  "audience": "installed_plugin",
  "promise": "把版本化评测清单准备为可派发的受控计划",
  "entry": {
    "kind": "mcp_tool",
    "name": "codex_praetor_prepare_evaluation"
  },
  "package_requirements": [
    "plugin/data/evaluation-suite.json",
    "由该能力实际调用的全部 plugin 内脚本、数据与配置"
  ],
  "scenario": "evaluation_prepare_from_final_zip",
  "success": "所有任务保留 task_family、task_kind、范围、检查、预算和验收字段",
  "mutation": "remove_required_package_file",
  "risk_tier": "deterministic_artifact"
}
```

这份清单必须覆盖三类东西，避免再次误报：

| 类别 | 例子 | 必须满足 |
| --- | --- | --- |
| `installed_plugin` | 路由、派工、评测准备、能力画像、provider 状态。 | 必须从 plugin/MCP/Skill 的安装路径完成。 |
| `release_bundle` | 一次性安装器、公开 zip 校验、stable marketplace 激活。 | 必须从下载的发布 zip 完成。 |
| `developer_only` | 构建器、版本更新器、release 发布器、内部诊断。 | 不作为普通安装用户承诺，但必须明确标注，防止混淆。 |

任何新功能在编码前必须先写能力条目；任何能力被删、降级或改入口时必须同步改条目和公开文档。没有条目的“用户可用功能”不能进入发布影响 PR。

## 7. 稳定、低摩擦的长期开发流程

这不是每次人工做大考古。大考古只在现在和重大架构调整时做；日常由自动化执行。

### 7.1 开始一个任务时

1. 先判定是调研、普通 PR、发布影响 PR，还是 release incident。
2. 若会改变用户下载、安装、打开、调用或版本展示，则关联已有能力 ID；若是新能力，先新增能力条目和验收场景。
3. 根据变更影响范围自动选择测试，不让人靠记忆勾清单。

### 7.2 开发分支内

| 门禁层 | 何时运行 | 目的 | 成本 |
| --- | --- | --- | --- |
| 源码与合同 | 每次相关改动 | 类型、schema、权限、参数、生成物同步。 | 快。 |
| 能力闭包检查 | 每次发布影响改动 | 检查每个受影响能力声明的包内文件、MCP/Skill 入口与文档声明。 | 快。 |
| 目标场景测试 | 每次发布影响改动 | 从候选 zip 运行受影响能力，不从源码目录偷跑。 | 中等。 |
| 全量确定性能力矩阵 | PR 就绪前和 CI | 用临时 profile/临时 repo 跑所有不需外部账号的公开能力。 | 中等，应自动化。 |
| 真实 provider 验证 | provider、权限、任务合同、派工或风险恢复发生变化时 | 在隔离 worktree + 隔离 UserProfileRoot 中验证真实 CLI 行为。 | 较高，但只在相关变更时跑。 |

“全量”不等于每次都让两个 provider 烧大量额度。它指所有可确定执行的安装包场景。真实 provider 可以是受影响范围的最小 canary/真实工作；其证据必须与版本、tuple 和任务合同绑定，不能拿旧回执冒充。

### 7.3 PR 就绪条件

只有以下条件全部成立，发布影响 PR 才能进入人工合并：

1. 变更关联的能力条目完整，且没有未分类的用户可见功能；
2. 所有受影响场景在最终 candidate zip 中通过；
3. 全量确定性能力矩阵通过；
4. 历史事故 mutation 全部通过，且新增事故已转为能力级 mutation；
5. 真实 provider 只在它的合同发生改变时验证，并给出 `ready`、`blocked` 或 `needs_user_action` 的真实结论；
6. 版本、release intent、changelog、release notes、用户入口和发布计划同步；
7. 工作区干净，目标分支新鲜，远端必需检查通过。

这一步完成后，PR 描述自动汇总“改了哪些能力、从哪个入口验证、运行了哪些场景、哪些外部边界未承诺”。用户不需要再替系统记住十几条收口步骤。

### 7.4 合并后的唯一收口

合并后不再做“发现新产品问题”的探索性验证。收口只验证同一个已批准的能力集合是否被真实发布：

```text
合并 SHA
  -> Release On Main 构建唯一 zip
  -> 对候选 zip 已跑过的确定性能力矩阵
  -> 上传不可变 Release
  -> 下载同一 zip，核对 hash / tag / generation
  -> 对下载 zip 重跑轻量协议和关键能力场景
  -> 自动安装同一 zip 到 stable marketplace
  -> 若 host 未刷新：准确停在 needs_host_restart
  -> 用户刷新一次 host 后：runtime_info、canary、受影响真实 worker
```

若 artifact、发布或远端下载失败，这是 `代码已合并，产品未交付（release incident）`，只能重跑原 SHA 的自动发布链。若外部 provider 在它没有变化的情况下临时不可用，产品可仍然完成 artifact 交付，但状态必须准确显示该 provider `blocked` 或 `needs_user_action`，不得宣称它可用，也不得为此改写发布包。

## 8. 如何处理历史补丁与当前遗留问题

建议下一次治理 PR 按以下顺序完成，而不是再开多个小修复：

1. **建能力清单。** 先列全当前公开能力，至少包含 22 个 MCP 工具、安装/激活流程、Skill 自然语言入口、两个 provider 的 canary/真实任务边界、评测准备与评测派工链。
2. **逐项定级。** 选择 `installed_plugin`、`release_bundle` 或 `developer_only`。不允许继续存在“它在源码里，所以大概算产品功能”的模糊状态。
3. **审计历史修复。** 将 #43 到 #47 的每个事故变成条目和 mutation。原测试可保留，但应由新清单调用或至少被清单引用。
4. **处理评测准备缺口。** 先作产品决策：
   - 若真实评测是正式产品能力，则在插件中提供受控入口，优先是 MCP 工具加明确的写入边界；它必须使用 bundled 数据和包内 helper，不能依赖仓库源码路径。
   - 若暂时不把它交给安装用户，则从公开产品承诺和路线中降级为 `developer_only`，同时不再声称已安装产品能够完成真实评测闭环。
   - 不能保持现在这种“文档承诺闭环、用户实际只能查看”的中间态。
5. **实现统一 runner。** runner 从最终 zip 安装到临时 profile，再按能力条目调用 MCP/Skill/release entry，校验最终结果；不是只检查文件名和工具名。
6. **实现通用 mutation。** 对每类依赖至少注入“文件缺失、入口缺失、入口输出不符合合同”之一；历史事故作为固定样本。
7. **把 runner 纳入共享 pipeline。** PR candidate、main Release On Main、远端下载复验使用同一场景定义，不允许三套不同脚本各验一部分。
8. **最后更新规则。** 规则只写不可妥协的流程，不堆具体文件名；具体能力和命令留在项目配置与 runner 中。

这样做能让过去的补丁真正“升格”为系统治理，而不是留下五块各自为政的修补痕迹。

## 9. 全局规则和项目规则应如何分工

本轮不直接修改规则，以下是下一 PR 的建议文本方向。

### 放入全局规则的原则

适用于所有会发布用户可运行产物的项目：

- 发布影响变更必须声明受影响的用户能力；新能力必须有可执行用户场景。
- 合并前必须从最终构建产物而不是源码目录验证受影响能力；发布影响 PR 还必须运行项目定义的全量确定性能力矩阵。
- 已知发布事故必须成为可重复的能力级回归或故障注入；不得只留 release note。
- 合并后收口只验证已批准能力是否从同一 artifact 交付；发现内部能力缺口说明合并前门禁缺失，应先停止并补齐门禁，不进入连续尾修。

### 放入项目规则的事实

只适用于 Codex Praetor：

- 稳定安装物是 `plugin/`，而不是仓库检出或 zip 根目录；
- 能力入口只能是 plugin MCP、plugin Skill 或明确的 release-bundle installer；
- 22 个 MCP 工具、评测计划、Qoder/CodeBuddy 的具体合同与 provider evidence 层级；
- runner 的隔离 UserProfileRoot、worktree、不能接触认证数据、host refresh 的唯一人工边界；
- capability manifest、场景 runner、mutation runner 和 CI 的实际文件路径。

这能让全局规则保持短且可迁移，项目规则保持准确且可执行，避免又把几十条实现细节塞进每个对话的提示词。

## 10. 能保证什么，不能保证什么

新机制可以保证的，是**内部可验证的交付缺口**不能再靠运气逃过合并：源码有而包里没有、包里有而安装入口没有、入口有而场景走不通、文档承诺和实际能力不一致，都会在合并前被能力矩阵拦住。

它不能保证的，是第三方系统永不变化：Codex host 不会自动热加载、Qoder/CodeBuddy 的服务不会临时故障、网络不会中断、账号不会过期。这些必须被建模成明确状态和用户动作，而不是被伪装为“收口完成”或被新 PR 掩盖。

真正的目标不是承诺“世界上再也不会出错”，而是让所有可由本项目控制的错误都在合并前暴露；即使发生外部错误，也有唯一、诚实且不会制造新尾巴的处理路径。

## 11. 下一步建议

下一步不应立即修一个 `prepare_evaluation` 工具然后再发 PR。应把第 8 节定义的“能力清单 + 历史事故升级 + 最终 zip 场景 runner + 当前评测准备缺口”作为**一个统一治理 PR**的范围。

该 PR 在开始编码前要先完成一张实际能力矩阵，并决定评测准备是正式安装用户能力还是明确降级为开发能力。只有这张矩阵覆盖完成、每个历史事故都有归属后，才进入实现；实现完成后必须重新跑整张矩阵，再创建 PR。这样才能结束“发现一个、修一个、合一个”的循环。
