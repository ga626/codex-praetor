## 一、执行结论

```text
不可变发布代际
-> 统一安装/激活/退休
-> generation-matched provider readiness
-> 统一 durable job 生命周期
-> Codex 验收和真实交付收据
```

建议新 PR 的产品版本为 `0.3.0-alpha`，原因是该 PR 会统一 provider 合同、任务生命周期和 MCP/状态数据结构，已经超出单纯补丁版本。若最终确认公开 MCP 工具名、参数和 Skill/Plugin manifest 完全不变，版本才可以降为 `0.2.1-alpha`；不能在实施中一边改合同一边继续沿用 `0.2.0-alpha`。

## 三、现有实现与真实缺口

### 1. 已有实现，不能重复造轮子

上一轮已经实现并合并的基础包括：

- runtime contract、generation identity、stage/activate receipt。
- Skill、Plugin、cache、marketplace 的 hash/一致性检查。
- `Invoke-CodexPraetorNative` 的 stdout/stderr 分离、退出码优先和 timeout 结果。
- retirement manifest、旧 generation 自动 reconcile、活动代际保护、锁定路径延迟回收。
- stable 维护脚本入口和用户级登录/定时重试设计。
- provider readiness 基础记录：provider、CLI path/hash、model、permission profile、task kind、wrapper protocol、expiry。
- GUID job id、worktree 隔离、repo edit lock、job.json、stdout/stderr、completion、watcher、cancel。
- plan task 的 `awaiting_verification` 边界，依赖任务不能只因 worker 退出而解锁。
- MCP 的 route intent、runtime info、health、dispatch、result、job timeline、cancel、plan 和 verify 工具。
- MiMo 默认 Build 不再被称为技术只读，所有 provider 都在 Codex 创建的 disposable worktree 中运行。
- 外部研究已划给 Codex + KnowledgeRadar，provider worker 只允许在研究合同下做受控补充。

这些是大 PR 的基线，不应重新复制一套并行发布器、并行 readiness 文件或并行 job 状态机。

### 2. 仍然是半成品或存在冲突的地方

#### 2.1 发布层有多套事实来源

`complete-codex-praetor-release.ps1` 已经有 stage/activate，但个人 plugin、Skill、cache 的发布脚本仍是历史上独立的入口。大 PR 要保留兼容命令，但让它们全部调用一个 release controller：

```text
artifact verify -> staged surfaces -> proof/readiness -> activate -> retire -> receipt
```

不能继续让每个脚本各自决定版本、备份、删除和成功条件。

#### 2.2 readiness 有 tuple，但还没有完整 generation 绑定

当前 canary 已检查 CLI path/hash、model、permission profile 和 task kind，但仍需要加入：

- generation id。
- runtime contract hash 和 task contract schema。
- CLI 精确版本和来源类型。
- agent/profile、output protocol、OS/架构。
- canary 用到的 provider wrapper 版本。
- capability 覆盖范围和失效原因。

否则同一个 provider 记录可能被错误复用于另一个发布代际或另一个 CLI 合同。

#### 2.3 路由仍有“时间/成本优先”的历史倾向

配置中仍有昼夜 tier、成本倍率和默认 provider。它们只能做最后的偏好排序，不能先于安全边界和 readiness。正确顺序必须是：

```text
任务类别与禁止边界
-> install generation 一致性
-> provider 合同与权限适配
-> 当前 readiness/canary
-> 最近失败、冷却、熔断
-> 成本、时间和用户偏好
```

#### 2.4 job 已有 durable 外壳，但需要统一终态语义

当前 wrapper/watcher 已支持 timeout、cancel、completion 和若干 failure class，但大 PR 需要把 blocking/background 完全统一为同一 job 状态机，并明确：

- `exit_code=0` 不是语义成功。
- completion 缺失是 watcher failure，不是“仍在运行”或成功。
- provider 工具不存在、权限拒绝、未登录、CLI 缺失、超轮数、无产出、工作树漂移必须是不同分类。
- cancel/timeout 后必须确认进程树、锁和 worktree 状态，不能只写一个 `cancelled` 文件。

#### 2.5 provider 文案和真实合同仍有错位风险

外部官方资料确认：

- CodeBuddy 的权限是分层规则，deny 高于 allow，非交互模式下 unresolved ask 会转为 deny；工具可见性和权限拒绝不能混成一个 `--tools` 白名单语义。
- Qoder 的 Plan 是工作状态，不是权限策略；不能用 Plan 代替文件隔离或权限门。
- MiMo 的 Build 默认全工具可写，Plan 才是受限分析模式；MiMo 只能以 isolated audit 或 edit-worktree 语义进入产品。

大 PR 必须把三家 provider 统一成“provider adapter + capability contract”，而不是继续靠 provider-specific 命令字符串猜测安全性。

#### 2.6 文档路线图落后于实现

`docs/roadmap.md` 仍把真实 dispatch、失败分类、验收工具和公开入口一致性列为近期目标，部分已经完成；同时没有把 generation promotion、readiness contract 和统一 job controller 写成下一阶段主线。大 PR 必须用一个新的状态表替换旧的“完成/近期/候选”混杂列表。

## 四、目标架构：一个控制面，五个受控对象

### 1. 五个对象

#### A. Generation

```json
{
  "product": "codex-praetor",
  "channel": "stable",
  "version": "0.3.0-alpha",
  "tag": "v0.3.0-alpha",
  "commit": "<immutable-main-sha>",
  "artifact_sha256": "<release-zip-sha256>",
  "content_manifest_sha256": "<tree-sha256>",
  "runtime_contract_sha256": "<contract-sha256>",
  "task_contract_schema": "codex-praetor-task-contract/v4",
  "required_mcp_tools": [],
  "created_at": "<utc>"
}
```

版本号不是唯一身份。任何 Skill、Plugin、cache、receipt、fresh proof、provider readiness 和 job 都必须能回指同一个 generation。

#### B. Surface

统一描述 Skill、Plugin、MCP、marketplace、cache、用户配置和维护任务。每个 surface 有 path、tree hash、来源、generation、状态和最后验证时间。

#### C. Provider Readiness

```json
{
  "provider": "codebuddy",
  "source": "standalone_official",
  "cli_path": "<local path>",
  "cli_hash": "<sha256>",
  "cli_version": "<exact version>",
  "model": "hy3",
  "agent": "<agent>",
  "permission_profile": "local-audit-v2",
  "tool_profile": "<tool contract>",
  "task_kind": "local_audit",
  "os": "windows-x64",
  "wrapper_protocol": "2",
  "runtime_generation_id": "<generation id>",
  "runtime_contract_sha256": "<hash>",
  "status": "ready",
  "passed_at": "<utc>",
  "expires_at": "<utc>",
  "failure_class": ""
}
```

状态固定为：`unknown`、`manual_required`、`ready`、`degraded`、`blocked`、`cooldown`、`disabled`、`stale`。

#### D. Durable Job

所有 blocking/background 调用都先落盘：

```text
planned
-> eligibility_checked
-> worktree_ready
-> starting
-> running
-> worker_completed
-> awaiting_verification
-> accepted | rejected | retry | human_required | skipped
```

异常终态：`failed`、`timed_out`、`cancelled`、`watcher_failed`、`orphaned`、`worktree_drift`。

`job.json` 至少记录：job id、base SHA、generation id、task contract hash、provider tuple、worktree、PID 和启动时间、进程拥有方式、deadline、命令 hash、事件列表、completion 路径、cleanup 结果。

#### E. Promotion Receipt

receipt 是唯一的交付事实，不是脚本输出文字。它要关联：

- artifact URL、tag、commit、zip hash。
- 每个 surface 的 tree hash。
- fresh context 工具名和关键调用结果。
- provider readiness 记录。
- active generation、previous generation、retirement 状态。
- activation 时间、维护任务状态、失败/回滚原因。

## 五、统一状态机与不变量

### 1. 发布状态机

```text
main_merged
 -> artifact_built
 -> artifact_hash_verified
 -> remote_release_verified
 -> staged
 -> fresh_context_verified
 -> provider_readiness_verified
 -> activated
 -> retirement_reconciled
 -> delivered
```

任意阶段失败都不能把 generation 写成 `active`。旧 active receipt 保持不变，失败 candidate 进入 `failed_manual_review` 或待清理队列。

### 2. 五条硬不变量

1. **不可变代际**：不覆盖旧 tag、旧 zip、旧 cache generation 或仍可能被 session 引用的目录。
2. **激活与回收分离**：新代际先完整激活，旧代际后进入退休队列；回收失败不回滚新 active。
3. **真实派工 fail closed**：generation、task contract 或 provider tuple 不匹配时，只允许 status、doctor、dry-run，禁止 real dispatch。
4. **worker 完成不等于任务完成**：只有 Codex 读取结果、检查 diff/测试并记录 accepted，计划依赖才解锁。
5. **Codex 是最终控制者**：provider 不负责最终研究结论、主线合并、发布激活或用户账号操作。

### 3. 自动回收边界

回收器只能在批准根目录内处理非 active、超出保留窗口、未被占用的退休路径；拒绝 reparse point 和越界路径。锁定/权限/临时 IO 失败必须保留路径并记录下一次重试时间。不得强杀 Codex 或停止 provider。

Windows 官方 Restart Manager 可以登记资源并识别使用它们的应用，但它只在调用者有权限时关闭应用，不能授权 Codex Praetor 擅自关闭当前 Codex。成熟更新器 Squirrel.Windows 也采用新版本目录并行、启动后保留当前和紧邻上一版本，再清理更老目录；WinSW 的 deferred file operations 同样把更新动作延迟到目标进程启动前。

## 六、大 PR 的明确范围

### 1. 必须纳入

#### 6.1 规则和测试合同

- 修复全局中文规则导致的开发环境测试断言过时问题。
- 让测试验证语义，不验证固定语言、标题或用户个性化文字。
- 将 release closeout、generation、readiness、job 和研究路由写成机器可检查的合同。

#### 6.2 统一 release controller

- 新增一个 canonical controller，兼容现有发布命令但不再重复实现 copy/hash/backup/cleanup。
- 所有安装面使用同一个 generation manifest。
- stage 使用下载的 Release zip；稳定通道不允许源码目录冒充 artifact。
- activation 写 active receipt 后才触发 retirement reconcile。
- stable 安装成功后安装用户级登录触发和定时重试维护任务；隔离测试必须显式跳过。
- 增加 artifact provenance、远端下载、tag/commit/hash 绑定和可恢复回滚。

#### 6.3 generation-aware readiness

- readiness schema 升级到包含完整 provider tuple 和 generation id。
- CLI path、hash、版本、来源、model、agent、tool profile、permission profile、task kind、OS、wrapper protocol、task contract 和 expiry 任一变化都使旧 canary 失效。
- health 输出 `ready/degraded/blocked/manual_required/stale`，不再把“CLI 存在、登录未知、canary 未跑”合并成 ready。
- 路由先做安全/合同/readiness，成本与时间只做最后排序。

#### 6.4 三 provider adapter 与合同矩阵

- Qoder：区分 Plan work state 和 permission mode，固定 local-audit、edit-worktree 两类合同。
- CodeBuddy：不把可见工具列表当成权限拒绝；验证 deny/allow/ask/non-interactive 行为，`tool-not-found` 与 `permission_denied` 分开分类。
- MiMo：保留 `mimo-isolated-audit` 和 `mimo-auto-edit`，禁止把 Build 宣称为 readonly；Plan 行为必须通过真实 canary 证明才可进入 audit。
- 每个 provider profile 必须有 preview、真实 canary、失败注入和主仓库 clean 检查。

#### 6.5 统一 durable job controller

- blocking/background 共用 job 创建、watcher、completion、终态和清理逻辑。
- 新增 deadline、orphan detection、worker start time、命令 hash、process ownership、cleanup verification。
- Windows worker 使用 Job Object 或等价进程组拥有机制，确保 timeout/cancel 能针对 worker 进程树；不能把它用于 Codex 主进程。
- cancellation 只有在进程树确认退出、锁释放和 worktree 状态检查后才写最终 `cancelled`。
- semantic failure 不再依赖 exit code 单字段；`exit 0 + tool mismatch/marker missing/no output` 必须失败或 manual_required。

#### 6.6 研究和派工边界

- `external_research_support` 必须带 `codex_kr_primary`、`supervisor_verified`、claim scope、source scope 和 worker mode。
- provider worker 只能做候选发现、指定页面检索或独立复核，不能产出最终研究结论。
- 外部研究意图继续由 Codex + KnowledgeRadar 路由，不进入 provider auto 候选池。
- 结果必须有 URL、检索时间、摘录、主张、不确定性和证据等级。

#### 6.7 MCP 和文档产品化

- MCP 工具返回统一的阶段、状态、执行者、generation、下一步和失败分类。
- `runtime_info`、`health`、`job_timeline`、`result`、`cancel` 和 plan 工具读取同一状态源。
- 更新 README、安装、排错、卸载、provider 文档、路线图、发布说明和接受清单。
- 将“当前 task 继续开发、合同变化每代一次 fresh context、旧代际自动回收”写成用户能执行的流程。

#### 6.8 版本和发布准备

- 全项目统一到新版本，例如 `0.3.0-alpha`。
- 更新 `CHANGELOG.md`、Release notes、README 下载链接、路线图、脚本默认值、MCP/Plugin/Skill manifest。
- 新增大 PR 专属 release gate 和远端包复验。
- PR 合并后再创建新 tag/Release；不在 PR 中覆盖旧 `v0.2.0-alpha`。

### 2. 明确不纳入

- 不实现当前 Codex 对话的强制自我替换；官方现有能力和公开 issue 仍不能保证 Skill、Plugin cache、MCP manager 和当前 tool snapshot 同步切换。
- 不强杀 Codex Desktop，不为了清理旧目录停止 provider。
- 不自动替用户安装 provider、登录、扫码、选择账号域或读取认证数据库。
- 不把 Codex Praetor 扩展成通用多 agent 平台。
- 不在本 PR 新建常驻分布式数据库、远程队列或云端 worker 服务；本项目仍是 Windows 本地控制面。
- 不把外部联网研究交给 Qoder、CodeBuddy 或 MiMo 作为最终来源。
- 不新增 UI 设计或不可验证的 Codex Desktop 原生界面承诺。

## 七、详细实施步骤

### 阶段 0：冻结基线

1. 从最新 `origin/main` 创建 `codex/productization-v0-3-0` 分支。
2. 记录 base SHA、当前稳定 active receipt、当前远端 Release、当前 provider readiness 和本地全局规则版本。
3. 确认工作树干净；所有安装验证使用临时 `UserProfileRoot`。
4. 写一份迁移矩阵，逐项标注“已有实现、需要重构、需要新增、明确不做”。

阶段门：没有基线 receipt 和迁移矩阵，不开始 provider 或 job 重构。

### 阶段 1：先建立合同和状态模型

1. 定义 generation v2、provider readiness v2、job v2、promotion receipt v2 JSON schema。
2. 定义状态迁移表、允许的终态、幂等键和失败分类。
3. 所有 schema 增加 schema version 和 backward-compatible reader。
4. 为旧 receipt、旧 readiness、旧 job 写只读迁移/降级逻辑；旧数据不能被静默当成新 generation ready。
5. 先写 schema parser、状态迁移和负向测试，再改运行脚本。

阶段门：模拟 generation mismatch、stale readiness、missing completion、old receipt 时，系统必须 fail closed，且不改变旧 active receipt。

### 阶段 2：把 release 入口收束到一个 controller

1. 抽取 artifact verify、surface copy、tree hash、receipt 写入和 rollback helper。
2. 让现有 `install-user.ps1`、Skill publisher、cache publisher 和 `complete-codex-praetor-release.ps1` 调用 canonical controller。
3. 删除发布脚本中的直接旧目录删除；所有旧路径写入 retirement manifest。
4. 将 `stable`、`dev`、`test` 三个 channel 的根目录和 receipt 分离。
5. 维护任务只安装在 stable 的真实用户 profile；隔离测试不能污染任务计划程序。
6. 让 health 同时显示 active generation、staged generation、retirement 状态和维护任务状态。

阶段门：人为制造中途失败、旧目录锁定、active receipt 缺失、reparse point 越界、surface hash 漂移，均不得产生半激活或误删 active。

### 阶段 3：统一 provider adapter 和 readiness

1. 定义 provider adapter 接口：`discover`、`render_command`、`probe_version`、`run_canary`、`parse_result`、`classify_failure`。
2. 将 Qoder、CodeBuddy、MiMo 的命令拼接从主 wrapper 中收敛到 adapter。
3. readiness 记录完整 tuple 和 generation。
4. 路由按安全边界、任务类型、generation、profile、readiness、冷却、成本排序。
5. 任何 `unknown/stale/blocked/manual_required` provider 都不能进入 auto。
6. 真实 canary 不再只返回固定 marker；必须同时验证工具合同、输出协议、主仓库 clean、worktree 约束和语义结果。

阶段门：改变 CLI hash、版本、model、agent、profile、task kind、runtime contract 或 generation 任一字段，旧 canary 都必须失效。

### 阶段 4：统一 durable job 和进程所有权

1. 所有 dispatch 先创建 job.json 和 event `planned`，blocking/background 不再两套逻辑。
2. 创建 GUID worktree 和 job scratch，写入 base SHA、generation、provider tuple 和 deadline。
3. worker 进程启动后记录 PID、启动时间、命令 hash、stdout/stderr 文件和进程拥有方式。
4. watcher 负责等待、timeout、进程树退出、completion 和锁释放。
5. 使用 Job Object 或明确的进程组实现 worker 进程树管理；不能用模糊命令行搜索杀进程。
6. 完成后先写 `worker_completed`/异常终态，再由 Codex 验收写 `accepted/rejected/retry/human_required/skipped`。
7. plan controller 只依赖 Codex verdict 解锁后续任务。

阶段门：exit 0 + stderr、exit 0 + marker 缺失、exit 0 + tool-not-found、timeout、cancel、worker 自行退出、watcher 崩溃、同秒并发 worktree 全部有确定分类和可恢复结果。

### 阶段 5：统一 MCP 和外部研究合同

1. MCP 工具全部从同一状态读取器读取 generation、readiness、job 和 receipt。
2. route intent 明确区分 `external_research`、`local_audit`、`code_change`。
3. research contract 缺字段时直接拒绝 provider worker。
4. result 工具只返回压缩证据和下一步，不倾倒完整日志、JSONL、SQLite 或账号信息。
5. fresh-context proof 记录工具集合、generation、关键调用和上下文时间。

阶段门：研究任务不能产生 provider dispatch；未完成 Codex 验收的 worker 不能解锁依赖任务；旧 generation 不能调用真实 dispatch。

### 阶段 6：产品文档与版本收束

1. 更新 README 和英文切换页的当前版本与下载入口。
2. 更新安装、排错、卸载、provider、用户验收清单和路线图。
3. 删除或改写已经完成但仍被列为“近期目标”的项目条目。
4. 增加“普通修改/合同变化/发布影响变更”的用户动作表。
5. 全局规则的开发环境测试只做语义检查，不绑定语言。
6. 完成新版本 manifest、Release notes、CHANGELOG 和脚本默认值同步。

阶段门：public-entry consistency、public scan、版本一致性、中文规则语义检查全部通过。

### 阶段 7：大 PR 前置审计

按顺序执行：

1. `git status --short --branch`、冲突标记和 `git diff --check`。
2. PowerShell 解析、MCP `npm test`、产品验证、开发环境验证。
3. native regression、provider contract matrix、job lifecycle matrix。
4. release package determinism、public marker/secret scan、public entry consistency。
5. isolated closeout：stage、activate、drift rejection、retirement lifecycle。
6. fresh-context native MCP proof 和 provider readiness proof。
7. 最新目标分支、远端 CI、用户普通路径和文档检查。

只有这些完成，才给出 PR 标题和完整描述；标题、正文分别放独立代码块。

## 八、验证矩阵

### 1. 发布和代际

| 场景 | 预期 |
|---|---|
| 旧 tag 已存在 | 拒绝复用 |
| 远端 zip 与当前 main 不同 | 拒绝交付 |
| stage 中途失败 | 不写 active receipt |
| active surface 被锁定 | 新 generation 不回滚，旧路径进入 blocked/pending |
| active generation 被回收 | 永远拒绝 |
| reparse point 越界 | 拒绝删除 |
| 解锁后重试 | 删除成功并记 `deleted` |
| 无 active receipt | 不删除旧缓存，记 `deferred_no_active_receipt` |

### 2. Provider readiness

| 变更 | 预期 |
|---|---|
| CLI path/hash/version 变化 | 旧 canary stale |
| model/agent/profile 变化 | 旧 canary 不匹配 |
| task kind 变化 | 旧 canary 不复用 |
| generation/contract hash 变化 | 必须重新 canary |
| CodeBuddy tool-not-found | `tool_contract_mismatch`，不进 auto |
| CodeBuddy permission denied | `permission_denied`，保留诊断 |
| Qoder Plan | 只标 work state，不当权限证明 |
| MiMo Build | 不能标 readonly，必须 worktree |
| provider 未登录 | `manual_required`，不伪装 ready |

### 3. Job 生命周期

| 场景 | 预期 |
|---|---|
| exit 0 + 普通 stderr | 成功候选，保留 stderr |
| exit 0 + marker 缺失 | semantic failure |
| exit 0 + tool-not-found | tool contract mismatch |
| 非零 exit | worker failed |
| timeout | timed_out，进程树终止结果可见 |
| cancel | 只有进程树和锁确认释放后才终态 |
| watcher 崩溃 | watcher_failed/orphaned，不伪装 completed |
| 两个同秒任务 | GUID worktree 无碰撞 |
| worker completed | plan 进入 awaiting_verification |
| Codex rejected | 依赖不解锁 |

### 4. 研究和用户边界

| 场景 | 预期 |
|---|---|
| external research | Codex + KR 主路由，不派 provider |
| worker research support | 必须有 research contract 和 supervisor_verified |
| provider 缺失 | 本体、dry-run、status 仍可用，真实派工 disabled/blocked |
| 新工具合同代际 | 只要求一次 fresh context |
| 普通代码编辑 | 不要求新 task |
| 用户未确认发布 | 不创建 tag/Release |

## 九、用户动作边界

### Codex 自动完成

- 建分支、实现、测试、构建、隔离安装、生成 PR 材料。
- 合并后切回主线、下载远端包、stage/hash-verify、生成 proof/readiness、执行 activate 和回收器。
- 读取真实状态并报告旧代际回收状态。

### 用户只需要做

- 在 GitHub 页面创建和合并 PR。
- 在不可逆发布前确认版本/tag/Release。
- 工具合同变化的每个 generation 打开一次 fresh Codex context。
- 按 provider 官方流程完成登录、扫码、授权或 API 配置。
- 需要真实 provider canary 时明确允许它消耗额度。

### 用户不需要做

- 不需要每次改文件都新建 task。
- 不需要手动强删 Codex 旧 cache。
- 不需要强杀 Codex 进程帮助清理。
- 不需要把 provider token、账号数据库或截图交给项目。

## 十、发布后收口顺序

大 PR 合并后必须按以下顺序，而不是直接说“收口完成”：

1. 切到最新 `main`，确认合并提交和 CI。
2. 构建新的不可变版本，例如 `v0.3.0-alpha`。
3. 发布 zip、`.sha256`，最好同时启用 GitHub immutable release/attestation 能力。
4. 从远端下载资产，校验 tag、commit、zip hash、sidecar hash 和内容清单。
5. 在真实 stable profile 执行 stage。
6. 由于 Plugin/Skill/MCP 来源和 generation 变化，打开一次 fresh context，记录原生工具集合和关键调用。
7. 对已声明支持的 provider tuple 运行 canary，生成新 generation 的 readiness。
8. activate 并运行 retirement reconcile。
9. 检查 `health=ready`、active receipt、维护任务和 retirement 状态。
10. 按普通用户路径下载、解压、双击 `setup.cmd`，完成一次 dry-run 和状态查询。
11. 只有远端包、稳定安装、health、fresh proof、provider readiness 和回收状态全部有证据，才称“产品已交付”。

## 十一、最终建议

下一轮不要先做 `0.2.1-alpha` 的单独发布修补，再做 provider/job PR；那会再次产生“旧版本已经发布、主线又继续漂移、稳定安装无法证明来自哪次代码”的双轨状态。

正确路线是：

```text
一个大 PR：统一规则测试、release controller、generation receipt、readiness v2、provider adapters、job controller、MCP 状态、文档和版本元数据
-> 用户创建并合并 PR
-> 一个不可变新 Release
-> 一次 fresh-context 验收
-> 一次 stable activation
-> 自动退休旧 generation
-> 产品交付
```

这不是把所有未来想法无边界塞进一个 PR。它把已经存在的分散实现和同一条可靠性链上的缺口收束到一个控制面；通用多 agent 平台、云端队列、Codex 当前会话自替换和 provider 自动登录明确排除在外。

## 十二、来源与限制

核心来源见同目录的《证据登记》文件。重要外部来源包括：

- OpenAI Codex 官方 customization、AGENTS、Plugins、MCP、App Server 和 changelog 页面：确认规则链、插件/Skill/MCP 的宿主边界及新会话/工具刷新语义；部分正文在本机抓取时返回 403，因此对精确运行时细节保持边界说明。
- OpenAI Codex GitHub issues #16653、#20605、#21138、#24390、#25285、#15508、#19155：确认已有会话工具快照、旧 cache 路径和 MCP schema stale 的公开问题证据；issue 不是修复承诺。
- Microsoft Restart Manager、`RmGetList`、`MoveFileExW`、PowerShell `Start-Process`、`about_Redirection`、Job Objects：确认占用诊断、延迟文件操作、stdout/stderr 分离、退出码和进程组管理边界。
- GitHub Immutable Releases、artifact attestations：确认 tag、资产和发布证明应不可变，并可建立构建 provenance。
- CodeBuddy permissions/worktree/headless、Qoder permissions/tools、MiMo modes/agents：确认三家 provider 的权限、工具、工作状态和 worktree 语义不能混用。
- Squirrel.Windows update process、WinSW deferred file operations、Temporal Workflow：作为成熟更新器、延迟文件操作和 durable workflow 的工程模式证据，不直接当作 Codex Praetor 的依赖或产品合同。
