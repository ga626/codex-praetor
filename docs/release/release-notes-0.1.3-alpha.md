# Codex 执政官 0.1.3-alpha 发布说明

`0.1.3-alpha` 重点解决两个发布后的实际问题：产品检查不再被开发者个人电脑上的 Codex 配置误伤，同一份源码每次构建出的安装包也能保持一致。

## 用户能感受到什么

- 从 GitHub Release 下载的仍是一份双击 `setup.cmd` 即可进入中文向导的 Windows 安装包。
- 真实派工、结果读取、计划推进和 Codex 验收闭环保持不变。
- 发布包现在可复现。同一份源码重复构建会得到相同的文件摘要，维护者能更可靠地判断 GitHub 上的包是不是当前版本。
- 旧任务和 worker 工作目录积累后，可以先查看安全清理预览。只有已经合并、没有未保存改动且超过保留期的内容才会进入候选，默认不会删除。

## 发布和检查流程的变化

- 产品本身的检查与开发者本机环境检查已经分开。PR、CI 和 Release 不再依赖某台电脑上的全局规则、已安装 skill 副本或 provider 登录状态。
- 首页、安装指南、路线图、发布说明和 GitHub Release 现在由同一项一致性检查约束，避免代码更新了但用户下载入口仍指向旧版本。
- 发布脚本会为新版本创建新标签和新 Release。已有 Release 默认不可覆盖，只有同一提交的坏包修复并明确启用替换开关时才允许更新资产。

## 安全边界

- 清理工具默认只预览，不会触碰有未保存改动的 worker 工作目录，也不会清理仍在保留期内的任务。
- 发布包继续排除本机配置、账号数据、token、provider 缓存、任务日志、内部报告和开发运行态。
- Codex Praetor 仍不替用户登录 provider，不读取账号数据库，也不默认创建 Codex 原生 subagent。

## 发布验收

发布后必须从 GitHub Release 重新下载 `codex-praetor-setup-0.1.3-alpha.zip` 和对应 `.sha256`，确认：

- 远端安装包与本地构建摘要一致。
- `setup.cmd` 可以被 Windows 正确识别，安装向导版本显示为 `0.1.3-alpha`。
- 首页、安装指南、路线图和发布说明都指向 `v0.1.3-alpha`。
- GitHub Release 资产、说明和源码标签对应同一个提交。

只有这些检查全部通过，`0.1.3-alpha` 才算真正交付给用户。
